# Claude 명령어: Update Roadmap

완료된 작업을 `ROADMAP.md`에 체크하고 진행 상황을 갱신합니다.

## 사용법

```
/update-roadmap
```

## 이 저장소의 ROADMAP 구조

**다른 프로젝트의 로드맵과 형식이 다릅니다.** 아래 구조를 전제로 동작하세요.

- 파일 위치: **루트 `ROADMAP.md`** (`docs/` 아래가 아닙니다)
- 총 **Task 45개** (`Task 001` ~ `Task 045`), **Phase 4개**
- Task 헤더: `### Task NNN: 제목` (뒤에 ` - 우선순위`가 붙기도 함)
- Phase 헤더: `## Phase N: 제목`
- 진행 표기는 **GFM 체크박스**입니다 — `- [ ]` / `- [x]`. **✅ 이모지를 쓰지 않습니다.**

각 Task는 다음 줄들로 구성됩니다.

```markdown
### Task 006: 시드 계층 PRNG와 결정론 유틸리티를 구축한다 - 우선순위

- **담당**: 2팀 시뮬레이션엔진팀 / 리뷰: 1팀 코어·품질팀
- **일정**: 1일차 ~ 6일차 (2026-07-21 ~ 2026-07-28) / 추정 4.5인일 / 담당 2팀
- **근거**: FR-MT-015, FR-AD-003, NFR-DT-001~005·008
- **구현 사항**
  - [ ] `src/lib/sim/rng/` 에 xoshiro128\*\* 또는 mulberry32 구현
  - [ ] 시드 계층 파생 함수
- **수락 기준**: 동일 시드 100만 회 추출 결과가 재실행 시 바이트 단위 동일.
- **테스트**: Vitest — 시드 재현성, 분포 균등성.
```

## 프로세스

1. `ROADMAP.md`를 읽습니다.
2. 미완료 Task 목록을 보여줍니다 — 체크박스가 하나라도 `[ ]`인 Task.
3. **"어떤 Task를 완료했나요? (예: 006 또는 006,007)"** 로 묻습니다.
4. 입력받은 Task의 `**구현 사항**` 체크박스를 `- [ ]` → `- [x]`로 바꿉니다.
5. **수락 기준을 확인합니다** — 아래 §수락 기준 확인 참조.
6. 진행 상황 줄과 최종 갱신 날짜를 갱신합니다.
7. 변경 요약을 출력합니다.

일부 항목만 끝났다면 Task 번호 대신 **해당 체크박스만** 지정해 받으세요. 안 끝난 것을 끝났다고 표시하는 것이 이 명령어의 최악의 실패입니다.

## 수락 기준 확인 — 체크만 하고 넘어가지 마십시오

각 Task에는 `**수락 기준**`이 있습니다. 체크박스를 전부 `[x]`로 바꾸기 전에 **그 기준이 실제로 충족됐는지 사용자에게 확인**하십시오.

```
Task 006의 수락 기준입니다:
  "동일 시드 100만 회 추출 결과가 재실행 시 바이트 단위 동일"

이 기준이 충족됐습니까? (검증 방법도 함께 알려주세요)
```

충족 여부가 불확실하면 **완료로 표시하지 말고** 미완료로 두거나, 끝난 체크박스만 개별로 표시하십시오.

## 진행 상황 갱신

문서 상단(제목 바로 아래 메타 블록)의 진행 줄을 갱신합니다. 없으면 추가하세요.

```markdown
> **진행**: 12/45 Task 완료 (27%) · Phase 1 완료 · Phase 2 진행 중 · 최종 갱신 2026-07-20
```

- 완료 Task = **모든 체크박스가 `[x]`인 Task**. 일부만 체크된 것은 완료로 세지 않습니다.
- Phase 상태 — 해당 Phase의 Task가 전부 완료면 `완료`, 하나라도 진행 중이면 `진행 중`, 전부 미착수면 `대기`.
- 날짜는 반드시 `date +%F`로 **실제 오늘 날짜**를 확인해 쓰십시오. 기억으로 쓰지 마십시오.

## 절대 수정 금지

이 명령어가 바꿔도 되는 것은 **체크박스 상태와 진행 줄뿐**입니다.

- ❌ Task 번호·제목, Phase 구성
- ❌ 체크박스 **문구** (상태 `[ ]`→`[x]`만 변경)
- ❌ `**수락 기준**`, `**테스트**`, `**근거**`, `**담당**`
- ❌ `**일정**` 줄 — 일정은 `docs/team-schedule/`가 단일 소스입니다. 여기서 손대지 마십시오.

## 일정 문서와의 관계

`ROADMAP.md`와 `docs/team-schedule/`는 같은 계획을 말해야 합니다 (README의 동기화 규칙 참조).

| 상황 | 조치 |
|---|---|
| 예정대로 완료 | 이 명령어로 체크만 하면 됨. 일정 재산출 불필요 |
| **일정보다 지연/선행** | 일정이 어긋난 것 → **`schedule-planner` 에이전트로 재산출** 후 ROADMAP 동기화 |
| **Task 추가·삭제, 구현 사항 증감** (스코프 변경) | 이 명령어의 범위 밖. `development-planner`로 로드맵을 고친 뒤 **`schedule-planner`로 일정 재산출** |

체크를 마친 뒤, 완료일이 `**일정**` 줄의 종료 일차와 어긋나면 **사용자에게 알리고 재산출을 권하십시오.**

## 검증

편집 후 다음이 편집 전과 같은지 `grep -c`로 확인하십시오. 다르면 잘못 건드린 것입니다.

| 항목 | 명령 |
|---|---|
| Task 수 (45) | `grep -c '^### Task ' ROADMAP.md` |
| 체크박스 총수 | `grep -c '\- \[[ x]\]' ROADMAP.md` |
| 수락 기준 줄 수 (45) | `grep -c '^- \*\*수락 기준\*\*' ROADMAP.md` |
| 일정 줄 수 (45) | `grep -c '^- \*\*일정\*\*' ROADMAP.md` |

`git diff ROADMAP.md`에 체크박스 상태(`[ ]`↔`[x]`)와 진행 줄 외의 변경이 보이면 되돌리십시오.

## 출력 예시

```
현재 미완료 Task (44개 중 상위 5개):
  Task 001  확정 결정(D-15~D-26)을 설계 전제로 정착시킨다   [1/6]
  Task 002  도메인 TypeScript 타입 47종을 정의한다          [0/7]
  Task 003  공통코드 36개 그룹 체계를 설계한다              [0/5]
  ...

어떤 Task를 완료했나요? (예: 006 또는 006,007): 006

Task 006 수락 기준: "동일 시드 100만 회 추출 결과가 재실행 시 바이트 단위 동일"
→ 충족 확인됨 (Vitest 시드 재현성 테스트 통과)

✅ Task 006 완료 — 체크박스 5개 [x] 처리
📊 진행: 1/45 Task 완료 (2%) · Phase 1 진행 중
🔍 검증: Task 45 / 체크박스 273 / 수락 기준 45 / 일정 45 — 편집 전과 동일

⚠️  일정 확인: Task 006의 예정 종료는 6일차(2026-07-28)입니다.
    실제 완료일이 이보다 늦다면 `schedule-planner`로 일정 재산출을 권합니다.
```
