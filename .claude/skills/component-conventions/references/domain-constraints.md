# 도메인 제약 상세 — 정보 비노출과 표기 규약

`SKILL.md` 6절의 확장. 화면을 만들다 **머지 거부**로 되돌아오는 사유는 거의 전부 여기에 있습니다.
공통 원칙 하나로 요약하면: **보이면 안 되는 값은 UI에서 가리는 게 아니라 애초에 클라이언트로
오지 않아야 합니다.** 가리는 구현은 DevTools·네트워크 탭·`data-*`로 전부 새기 때문입니다.

## 목차

1. PA(잠재능력) 비노출 — 선수 관련 전 화면
2. 미래 정보 비노출 — 진행 중 경기가 걸린 전 화면
3. 표기 규약 — 단위·승부차기·이름
4. 화면별 빠른 참조

## 1. PA(잠재능력) 비노출 — R-5 / FR-PL-004

선수는 현재능력(CA/OVR)과 잠재능력(PA)을 가지지만, **PA는 사용자에게 어떤 경로로도 노출되지
않습니다.** 허용된 미래 정보는 **스카우트 등급 ★1~5(범위 표기)** 뿐입니다.

| # | 규칙 |
|---|---|
| P-1 | PA 원값을 표시하는 칸이 와이어프레임·마크업·DOM 어디에도 없다. **`data-*` 속성·주석·툴팁 포함** |
| P-2 | 스카우트 등급은 **서버가 산출한 등급값**을 받는다. 클라이언트가 PA를 받아 등급으로 환산하는 구조는 금지 — 원값이 네트워크에 실리는 순간 위반 |
| P-3 | 성장 곡선에 **PA 상한선·잠재 도달선·"성장 여력" 수치**를 그리지 않는다 |
| P-4 | 능력치 아코디언은 **현재 속성값(1~30)** 만 보여준다. 속성별 잠재값·성장 여력 표기 금지 |
| P-5 | **Mock 팩토리에서도** 화면에 전달되는 선수 객체에 PA 필드를 넣지 않는다 |

P-5가 중요합니다. Mock 단계에서 PA가 딸려 오면 실 데이터 전환 시 그대로 남습니다.

### 이미 구현돼 있습니다 — 다시 만들지 마세요

P-1·P-2는 **타입으로 강제**됩니다. 직접 방어 코드를 짜지 말고 아래를 쓰세요.

```ts
// src/lib/data/DataSource.ts:188
export type PublicPlayerProfile = Omit<Player, 'pa'> & {
  readonly scoutRating: 1 | 2 | 3 | 4 | 5;
};
```

- **선수 컴포넌트의 props 타입은 `Player`가 아니라 `PublicPlayerProfile`입니다.** `Player`를
  받아 놓고 화면에서 PA를 안 그리는 방식은 원값이 네트워크에 실리므로 P-2 위반입니다.
  타입을 좁히는 것만으로 P-1이 구조적으로 보장됩니다.
- 조회는 `getPlayerProfile(playerId)` / `getTeamSquad(teamId)`(둘 다 `DataSource` 계약,
  `PublicPlayerProfile` 반환)를 씁니다.
- **등급 환산을 컴포넌트에서 다시 하지 마세요.** `toPublicProfile()`
  (`@/lib/data/player-profile`)이 `Math.ceil(pa / 6)` 6점 균등 분할로 이미 산출합니다.
  이 함수는 21일차에 Mock 밖으로 옮겨져 Mock·Supabase 어댑터가 동등하게 참조합니다(결함 A 조치).

미결로 남은 건 **매핑 구간 값의 소유**(W-27)뿐입니다 — 밸런싱 튜닝(031b)이 상수를 대체할 수
있습니다. 구간을 바꿔야 한다면 컴포넌트가 아니라 그 함수에서, 소유 팀과 조율해 바꿉니다.

`ScoutRating` 표시 컴포넌트(★1~5)는 아직 없고 **5팀 화면 로컬 신규**입니다. 시그니처는
`scoutRating: 1|2|3|4|5`를 받고 **PA는 받지 않습니다.** 툴팁 문구는 정해져 있습니다 —
"잠재력 추정 등급(정확한 값은 공개되지 않습니다)".

**자가 점검**

```bash
# 화면·컴포넌트 계층에 PA가 새는지
grep -rni "\.pa\b\|potentialAbility" src/components/ src/app/ | grep -v node_modules
# Player를 직접 받는 선수 컴포넌트가 있는지 (PublicPlayerProfile이어야 함)
grep -rn "Player\b" src/components/domain/ src/components/composite/ | grep -i "props\|state<"
```

## 2. 미래 정보 비노출 — C-23 / NFR-SEC-004 / FR-MT-016

경기는 실시간으로 진행되고, **아직 방영되지 않은 시점의 정보는 결과를 역산시킵니다.**

> 예: 68분 시점에 시즌 득점이 이미 +2 되어 있으면, 남은 22분의 결과가 노출된 것과 같습니다.

| # | 규칙 |
|---|---|
| C-23 | 경기 결과·이벤트는 **서버가 `min(now − kickoff_at, 경기 길이)` 범위만 응답**한다. 전량 응답 후 클라이언트에서 숨기는 구현(CSS `display:none`, 조건부 렌더, 클라이언트 필터)은 **머지 거부**. 종료 전 결과 필드는 **`null`로 응답** |
| S-5 | 시즌/통산 스탯은 **종료된 경기까지만** 집계. 진행 중 경기분 미합산 |
| S-6 | 컨디션·피로는 **경기 종료 시 갱신값**을 표시. 실시간 감소분 미반영 |
| S-7 | 부상 타임라인은 **엔진이 공개한 경과분 이하**의 이벤트만 노출 |
| S-8 | 위는 **UI 필터가 아니라 데이터 계약**이다. 어댑터 양쪽(Mock·실데이터)에서 컷오프된 값이 내려와야 한다 |

그래서 컴포넌트 타입에서 **종료 전 스코어가 `number | null`** 인 것은 실수가 아니라 계약입니다
(`MatchCard.tsx`의 `homeScore`/`awayScore` 주석 참조). null을 "0으로 기본값 주기"로 처리하지 마세요.

경과분도 UI가 `(now - kickoff)`로 직접 계산하지 않습니다. 엔진(H-24 `src/lib/sim/schedule/worldclock.ts`)이
제공한 값을 **표시만** 합니다.

**반대로, 위반이 아닌 것** — UI 계층에서 현재 시각을 읽는 것 자체는 정상입니다. 카운트다운,
폴링 tick, 킥오프 UTC → 로케일 로컬 변환은 지적 대상이 아닙니다(C-2 단서). `Math.random()`·
`Date.now()` 금지는 `src/lib/sim/**` 한정입니다.

## 3. 표기 규약

### 금액 — 단위는 pt (L-03)

몸값·급여·이적료·재정·배당 전부 **포인트(pt)** 입니다. `1,240,000 pt` 형태.
**원화 기호(₩)·"원" 표기 금지.** 실제 화폐로 오인되면 안 되기 때문입니다.
공통 서식을 포맷터에 넣을지는 **W-35**로 열려 있습니다.

### 승부차기 — D-19 / C-15 / R-13

- 승부차기 골은 **`goals`·득점왕 집계에 합산하지 않습니다.**
- **연장 득점은 정상 득점으로 포함합니다.**
- 스코어보드는 분리 표기: `2-2 (승부차기 4-3)`
- 순위표 득/실 열에서도 PSO 골은 제외합니다.

### 이름 — D-17 / R-3

- **선수명·클럽명·감독명은 번역 대상이 아닙니다.** 변수로 주입합니다.
- **국적·포지션·부상 등급·전술 성향·페이즈 같은 enum은 번역키를 경유합니다.**
- 구장명 번역 여부는 **W-12**로 미결입니다.
- 실명 선수/실존 클럽/리그 명칭/엠블럼/초상 자산은 **PR에 포함 시 예외 없이 머지 거부**(C-1).
  전량 시드 기반 절차적 생성입니다(D-16).

### 능력치 스케일

1~30입니다(C-4). 0~100이나 0~20으로 바꾸지 마세요. 포지션 숙련도는 5단계 + 미보유(6표기)이고,
**내부 배율 수치(1.00/0.95/…)는 노출하지 않습니다** — 라벨(Natural/Accomplished/…)만 씁니다.

### 엔티티 ID 번호 주의

와이어프레임 01~04와 05~06의 엔티티 번호 체계가 다릅니다(**W-26**). 정본은
`docs/require/05-data-requirements.md` 쪽(`E-02 League` / `E-15 Fixture` / `E-16 MatchEvent` /
`E-17 MatchLineup` / `E-19 PlayerMatchStat`)이고 05·06 문서가 이를 따릅니다.

또 하나 헷갈리기 쉬운 것: 와이어프레임의 **영역 ID(`E1`~`E9`, 하이픈 없음)** 와
**엔티티 ID(`E-07`, 하이픈 있음)** 는 서로 다른 네임스페이스입니다.

### 필드·타입이 문서와 다를 때

**TS가 옳습니다.** `src/types/**`가 단일 소스이고, `docs/require/05-data-requirements.md`는
동기화하지 않는 초기 초안이라 이미 뒤처져 있습니다(9일차 결정, I-58). 05 문서를 갱신하지도 마세요.
