  "새 세션에서 팀원들 대기 상태 확인해줘" 같은 요청에도 이 스킬을 사용한다.
  일차 완료 후 팀원 정리와 `/compact`/`/clear` 타이밍 관리도 이 스킬의 범위다.
  팀원 패널에서는 사용하지 않는다.

# Agent Team 일차별 스프린트 운영
 
팀장(lead) 세션에서만 실행하는 워크플로우입니다. 사용자가 "N일차 진행해줘"라고
요청하면 아래 절차를 순서대로 따릅니다.
 
## 고정 팀 로스터
 
| 번호 | 팀명 | 참조 파일 |
|---|---|---|
| 1 | 코어품질팀 | `docs/team-schedule/01-코어품질팀.md` |
| 2 | 시뮬레이션엔진팀 | `docs/team-schedule/02-시뮬레이션엔진팀.md` |
| 3 | 데이터밸런싱배당팀 | `docs/team-schedule/03-데이터밸런싱배당팀.md` |
| 4 | UI기반i18n팀 | `docs/team-schedule/04-UI기반i18n팀.md` |
| 5 | 화면배팅UX팀 | `docs/team-schedule/05-화면배팅UX팀.md` |
| 6 | DB인프라팀 | `docs/team-schedule/06-DB인프라팀.md` |
 
보조 참조 파일: `ROADMAP.md` (팀원이 필요시에만 참조 가능), `README.md` (완료 후 갱신 대상)
 
## 범위 지정 실행 (예: "5~10일차 진행해줘")
 
여러 일차를 한 번에 요청받으면, **일차를 하나씩 완전히 순차적으로** 처리한다.
절대 여러 일차를 동시에 진행하거나, 팀 소환만 먼저 몰아서 하지 않는다.
 
1. 요청 범위(예: 5~10일차)를 확인하고, 처리 순서(5→6→7→8→9→10)를 사용자에게
   짧게 보고한다.
2. **N일차 하나에 대해 0~8단계를 전부 완료**한다 (대상 팀 선별 → 소환 →
   작업 → 1차/2차 교차 점검 → 이슈 로그 작성 → shrimp/ROADMAP/README 갱신 →
   commit & push → **팀원 정리 및 세션 compact**까지 끝낸다).
3. N일차의 세션 정리(8단계)까지 끝난 뒤에만 N+1일차로 넘어간다. 이전 일차가
   끝나지 않은 상태에서 다음 일차 팀을 소환하지 않는다.
4. 각 일차는 커밋을 개별적으로 남긴다 (범위 전체를 모아서 한 번에 커밋하지
   않는다). 커밋 메시지에는 해당 일차 번호만 명시한다.
5. 범위 내 특정 일차에 업무가 있는 팀이 하나도 없으면(모든 팀 md에 해당
   일차 섹션 없음), 그 일차는 "업무 없음"으로 짧게 보고만 하고 바로 다음
   일차로 넘어간다 (0~8단계를 억지로 수행하지 않는다).
6. 범위 전체가 끝나면 처리한 일차 목록과 각 일차별 이슈 유무를 최종 요약해
   사용자에게 보고한다.
## 0단계 — 새 세션 시작 시 (기존 팀원이 이미 대기 중인 경우에만)
 
새 세션을 시작했는데 이전에 spawn된 팀원들이 이미 대기(idle) 상태로 남아있는
경우, 새 지시를 내리기 전에 먼저 각 팀원에게 지금까지 작업한 내용을 확인한다.
- 각 팀원에게 개별적으로 "지금까지 작업한 내용을 요약해줘"라고 확인
- 확인된 내용을 팀장이 취합해서 사용자에게 짧게 보고
- 대기 중인 팀원이 없으면(=첫 세션) 이 단계는 건너뛰고 바로 1단계로 진행
## 1단계 — 대상 팀 선별 (반드시 먼저 수행)
 
1. 6개 팀 md 파일을 팀장이 각각 열어 "N일차" 섹션이 존재하는지 확인한다.
2. **N일차 섹션이 없는 팀은 이번 라운드에서 절대 호출하지 않는다.** 업무가 없다는
   이유만으로 팀을 소환하는 것은 금지된다.
3. 판별 결과를 사용자에게 짧게 보고한다.
   예: "N일차: 코어품질팀, DB인프라팀만 해당 업무 있음 → 이 2팀만 소환합니다.
   나머지 4팀은 N일차에 업무 없어 이번 라운드 제외합니다."
4. ROADMAP.md는 전체 맥락 파악용 참고 자료로만 사용하고, 소환 여부의 최종
   판단 기준은 각 팀 md 파일의 N일차 섹션 존재 여부다.
## 2단계 — 팀원 소환
 
대상으로 선별된 팀만 Agent Team으로 spawn한다. spawn 시 다음을 반드시 지정한다.
 
- **model**: `sonnet` (전 팀원 고정, 예외 없음)
- **읽을 파일**: 본인 담당 md 파일 1개만. 다른 팀의 md 파일은 열람 금지.
- **ROADMAP.md**: 필요한 경우에만 참조 허용
- **지시문 템플릿**:
  > 당신은 {팀명}입니다. `docs/team-schedule/{파일명}`의 N일차 섹션만 확인해서
  > 해당 업무를 수행하세요. 다른 팀의 md 파일은 열람하지 마세요. 작업을 시작하기
  > 전 shrimp task manager로 계획을 먼저 수립하고, 계획이 끝난 뒤에 실제 작업을
  > 진행하세요. 계획 수립과 개발 과정 모두 chain-of-thought(CoT) 방식으로,
  > 즉 결론을 바로 내지 말고 단계별로 근거와 중간 판단을 명시적으로 드러내며
  > 사고하세요.
## 3단계 — 팀원 작업 프로토콜 (각 팀원이 지켜야 할 규칙)
 
1. shrimp task manager로 **계획을 먼저 수립**한다. 계획 없이 바로 작업을
   시작하지 않는다.
2. 계획 수립 단계와 실제 개발 단계 모두 **chain-of-thought(CoT) 방식**으로
   사고한다. 결론이나 최종 코드를 곧바로 제시하지 않고, 문제 분석 → 대안 검토
   → 선택 근거 → 구현 순서로 중간 사고 과정을 명시적으로 드러내며 진행한다.
3. 계획에 따라 N일차 업무를 수행한다.
4. 업무가 완료되면 `ROADMAP.md`에서 **자신의 담당 항목만** 업데이트한다.
5. shrimp task manager의 **전체 진행 상태 업데이트는 하지 않는다** — 이는
   팀장 전용 작업이다.
6. 완료 시 팀장에게 완료 보고를 전달한다.
## 4단계 — 라운드 중 추가 팀원 필요 시
 
진행 중 예상치 못한 필요 인력(예: 긴급 버그 수정, 교차 검증)이 발견되면 팀장이
그 시점에 추가로 팀원을 spawn할 수 있다. 추가된 팀원에게도 2~3단계와 동일한
규칙(sonnet 고정, 담당 파일만 열람, shrimp 계획 선행)을 그대로 적용한다.
 
## 5단계 — 교차 점검 (해당 일차 참여 팀 전원 완료 후)
 
이번 라운드에 소환된 **모든** 팀의 작업이 완료되면, 2단계에 걸쳐 교차 점검을
진행한다. **1차는 팀원끼리, 2차는 팀장이 추가로** 진행한다 (팀장이 1차 점검을
대신하거나 건너뛰지 않는다).
 
**1차 — 팀원 간 교차 점검**
1. 팀장이 이번 라운드 참여 팀 전원에게 "서로 작업 내용을 공유하고 교차
   점검하라"고 지시한다.
2. 참여 팀들은 서로의 변경 사항(코드/문서 등)을 직접 확인하고, 연동 지점에
   이슈가 없는지 점검한다.
3. 각 팀은 1차 점검 결과(이슈 유무, 발견된 이슈 내용)를 팀장에게 보고한다.
**2차 — 팀장의 추가 검증**
4. 팀원 간 1차 점검이 모두 끝난 뒤, 팀장이 전체 결과물을 대상으로 추가
   검증을 진행한다 (팀원들이 놓쳤을 수 있는 전체 정합성, 팀 간 누락된
   연동 지점 등 확인).
5. 팀장의 2차 검증에서 발견된 이슈가 있으면 해당 팀에 재작업을 요청하고,
   재작업 완료를 확인한 뒤 6단계로 진행한다.
 
## 6단계 — 일차 작업 로그 작성
 
교차 점검이 끝나면 이슈 유무와 **상관없이 항상** `docs/dailyWorkLog/{N}Day.md`
파일을 작성한다 (예: 1일차 → `docs/dailyWorkLog/1Day.md`).
 
- 이슈가 있으면: 발견된 이슈와 담당 팀, 후속 조치 필요 여부를 정리
- 이슈가 없으면: "이슈 없음"으로 명시적으로 기록 (파일 생략 금지)
- 참여 팀 목록, 각 팀 작업 요약도 함께 기록
## 7단계 — 전체 완료 후 처리
 
일차 로그 작성까지 끝난 뒤 진행한다.
 
1. 팀장이 shrimp task manager의 전체 진행 상태를 업데이트한다.
2. `ROADMAP.md`, `README.md` 등 갱신이 필요한 문서를 확인하고 업데이트한다.
3. 변경 사항을 `git add`한다 (`docs/dailyWorkLog/{N}Day.md` 포함).
4. 일차 번호와 참여 팀을 포함한 커밋 메시지로 commit한다.
   예: `chore: N일차 스프린트 완료 - 코어품질팀, DB인프라팀`
5. remote git repository로 push한다.
## 8단계 — 세션/컨텍스트 정리 (compact/clear)
 
commit & push까지 끝난 뒤에만 진행한다. **팀이 아직 활성 상태일 때 `/compact`를
실행하지 않는다** — 진행 중인 팀 상태에서 compact하면 팀 구성 정보가 리더
세션에서 소실되는 알려진 문제가 있다 (CLAUDE.md는 compact 후 재주입되지만
팀 구성 정보는 재주입되지 않음).
 
1. 이번 라운드에 참여한 팀원 전원에게 작업 종료를 명확히 지시하고, 더 이상
   필요 없는 팀원은 완전히 종료한다. 단순히 idle 상태로 방치하지 않는다
   (idle 팀원은 계속 살아있는 상태로 context를 점유하며, 리더가 이를 인지하지
   못해 다음 라운드에 중복으로 재소환하는 문제가 보고되어 있다).
2. 팀원이 모두 정리된 것을 확인한다.
3. 그 상태에서 리더 세션의 `/compact`(또는 여러 일차 연속 진행을 마치는
   시점이라면 `/clear`)를 실행한다.
4. 다음 일차 팀원은 이전 컨텍스트를 이어받지 않는 **새 spawn**으로 생성한다
   (팀원 컨텍스트를 줄이는 방법은 compact가 아니라 종료 후 재생성이다 — 팀원은
   각자 독립된 context window를 가지므로 리더의 compact가 팀원 컨텍스트에
   영향을 주지 않는다).
## Hard Rules (예외 없이 지킬 것)
 
- 해당 일차에 업무 섹션이 없는 팀은 **절대 호출하지 않는다**.
- 팀원은 본인 담당 md 파일 외 다른 팀의 md 파일을 읽지 않는다 (ROADMAP.md는 예외).
- shrimp task manager로 계획을 먼저 수립하지 않고 바로 작업을 시작하지 않는다.
- 계획 수립과 개발 과정은 **chain-of-thought(CoT) 방식**으로 진행한다.
  결론/코드를 바로 내지 않고 분석 → 대안 검토 → 근거 → 실행 순서로 사고 과정을
  드러낸다.
- shrimp task manager의 **전체** 진행 상태 업데이트는 팀장만 수행한다.
- 모든 팀원의 모델은 `sonnet`으로 고정한다.
- 일차 진행은 사용자(팀장)의 명시적 요청이 있을 때만 시작한다. 자동/추측 트리거 금지.
- 팀원 패널(서브패널)에서는 이 스킬을 사용하지 않는다. 오케스트레이션은 항상
  리더 세션에서만 이루어진다.
- 교차 점검은 반드시 팀원 간 1차 점검을 먼저 거친 뒤, 팀장의 2차 검증을 추가로
  진행한다. 팀장이 1차를 대신하거나 건너뛰지 않는다.
- `docs/dailyWorkLog/{N}Day.md`는 이슈 유무와 상관없이 매 일차마다 반드시 작성한다
  (이슈 없어도 "없음"으로 명시 기록, 파일 생략 금지).
- 새 세션 시작 시 대기 중인 팀원이 있으면, 새 지시 전에 반드시 각자 작업 내용을
  먼저 확인한다.
- 여러 일차를 한 번에 요청받아도 **한 일차씩 순차적으로**(commit&push까지 완료 후
  다음 일차) 처리한다. 여러 일차를 동시에 진행하거나 팀 소환을 몰아서 하지 않는다.
- 팀이 활성 상태(팀원이 살아있는 상태)에서 리더 세션 `/compact`를 실행하지 않는다.
  반드시 그날 팀원을 정리한 뒤에만 compact/clear를 진행한다.
- 다음 일차 팀원은 이전 팀원의 컨텍스트를 이어받지 않는 새 spawn으로 생성한다.
  팀원 컨텍스트를 줄이는 수단은 compact가 아니라 종료 후 재생성이다.

1. 팀장이 01~06 md 파일을 확인 → 5일차 섹션이 있는 팀: 시뮬레이션엔진팀,
   화면배팅UX팀, UI기반i18n팀
2. 사용자에게 보고: "5일차: 시뮬레이션엔진팀, 화면배팅UX팀, UI기반i18n팀 3팀만
   해당 업무 있음. 나머지(코어품질팀, 데이터밸런싱배당팀, DB인프라팀)는
   이번 라운드 제외합니다."
3. 3팀만 sonnet으로 spawn, 각자 본인 md 파일의 5일차 섹션만 확인
4. 각 팀 shrimp task manager 계획 → 작업 → ROADMAP.md 자기 항목 업데이트 →
   팀장에게 완료 보고
5. 3팀 모두 완료 확인 후, 3팀이 직접 서로 작업 내용 공유 및 1차 교차 점검 →
   팀장이 이어서 전체 정합성 2차 검증
6. 점검 결과를 이슈 유무와 상관없이 `docs/dailyWorkLog/5Day.md`에 기록
7. 팀장이 shrimp task manager 전체 상태 갱신, ROADMAP.md/README.md 정리,
   git commit & push
8. 5일차 참여 팀원(시뮬레이션엔진팀, 화면배팅UX팀, UI기반i18n팀) 완전 종료 →
   팀원 전원 정리 확인 → 리더 세션 `/compact` 실행 (범위 진행 중이면 이어서
   6일차 대상 팀 md 확인으로 넘어감)
