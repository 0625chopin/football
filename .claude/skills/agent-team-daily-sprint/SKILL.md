# Agent Team 일차별 스프린트 운영

팀장(lead) 세션 전용. "N일차 진행해줘" 요청 시 아래 절차를 순서대로 따른다.

## 고정 팀 로스터

| 번호 | 팀명 | 참조 파일 |
|---|---|---|
| 1 | 코어품질팀 | `docs/team-schedule/01-코어품질팀.md` |
| 2 | 시뮬레이션엔진팀 | `docs/team-schedule/02-시뮬레이션엔진팀.md` |
| 3 | 데이터밸런싱배당팀 | `docs/team-schedule/03-데이터밸런싱배당팀.md` |
| 4 | UI기반i18n팀 | `docs/team-schedule/04-UI기반i18n팀.md` |
| 5 | 화면배팅UX팀 | `docs/team-schedule/05-화면배팅UX팀.md` |
| 6 | DB인프라팀 | `docs/team-schedule/06-DB인프라팀.md` |

팀별 산출물 기록은 `docs/team-schedule/outputs/NN-<팀명>.md`에 누적한다(팀원 열람 대상 아님).
`ROADMAP.md`·`README.md`는 팀장만 갱신한다.

## 컨텍스트 예산 규칙 (모든 단계에 우선 적용)

문서가 크다: `ROADMAP.md` 108KB, `docs/ISSUES.md` 152KB, 팀 md 18~42KB,
`{N}Day.md` 10~28KB. **어느 것도 전문(全文)으로 읽지 않는다.**

1. `grep -n`으로 필요한 절/행의 줄 번호를 먼저 찾는다.
2. Read는 반드시 `offset`+`limit`로 그 블록만. (`ROADMAP.md`의 Task 1건 ≈ 18줄,
   `### Task NNN`으로 검색)
3. ISSUES는 중복 확인용 grep만. 등재는 append이므로 통독이 필요 없다.

## 범위 지정 실행 (예: "5~10일차 진행해줘")

일차를 하나씩 완전히 순차 처리한다. 동시 진행/소환 몰아치기 금지.

1. 처리 순서(5→6→...→10)를 짧게 보고.
2. 일차 1개당 0~8단계 전부 완료 후에만 다음 일차로 이동.
3. 일차별 개별 커밋(번호만 명시, 범위 통합 커밋 금지).
4. 업무 있는 팀이 하나도 없는 일차는 "업무 없음"만 보고하고 스킵(0~8단계 생략).
5. 범위 종료 시 일차별 이슈 유무를 최종 요약 보고.

## 0단계 — 새 세션 시작 시 (컨텍스트 복구)

팀원에게 직접 묻지 않고 파일로 먼저 복구한다(대화형 재요약은 최후 수단).

1. 최신 `docs/dailyWorkLog/{N}Day.md`에서 `grep -n "^## "`로 절 위치를 잡고,
   **마지막 두 절(「다음 일차 인계」·「미해결·판정 대기」)만** offset+limit로 읽는다.
2. **`ROADMAP.md`는 읽지 않는다** — 소환 판별 기준이 아니다(1단계 참조).
3. 이걸로 충분하면 바로 1단계로 (대부분 여기서 종료).
4. idle 팀원이 실제 발견되면(=이전 세션이 8단계 미완 종료) → "마지막 수정 파일 +
   완료 여부"만 초경량 질의 → 확인 즉시 종료 → 로그와 차이 나면 짧게 보고.

## 1단계 — 대상 팀 선별 (grep 1회)

```bash
grep -n "^| \*\*{N}\*\* " docs/team-schedule/0*.md
```

- 행이 반환된 팀 = 참여, **행이 없는 팀 = 미참여(절대 호출 금지)**.
- `| **1~8** |` 같은 범위 행은 대기/종료 구간이라 걸리지 않는다 — 의도된 동작.
- 이 한 번의 grep이 참여 팀 판별과 **그날 할 일 원문**을 동시에 준다. 팀 md를
  열어 표를 훑지 말 것.
- 판별 결과를 짧게 보고(예: "N일차: A, B팀만 해당, 나머지 제외").
  `ROADMAP.md`는 참고용일 뿐 판단 기준이 아니다.

## 2단계 — 팀원 소환

- model: `sonnet` 고정 (예외 없음)
- 지시문에 **1단계 grep으로 얻은 해당 일차 행 원문을 그대로 붙여 넣는다** —
  팀원이 표를 다시 찾을 필요를 없앤다.
- 지시문 골자:

  > {팀명}, 아래가 당신의 N일차 작업 행입니다.
  > `{grep 행 원문}`
  > `{파일명}`에서 **§1 소유 경로**만 확인하고, 필요하면 §3 인계물 / §4 대기
  > 구간을 부분 읽기하세요. **파일 전문 읽기 금지, 타 팀 md 열람 금지,
  > `ROADMAP.md`·`docs/ISSUES.md` 열람 금지.**
  > 산출물이 여러 파일이나 여러 결정에 걸치면 shrimp로 계획을 먼저 세우고,
  > 단일 파일 작업이면 3~5줄 인라인 계획으로 갈음하세요.
  > 판단이 갈리는 지점에서만 분석→대안→근거를 밝히고, 자명한 단계는 바로
  > 구현하세요. 완료 보고는 3단계 템플릿 15줄 이내.

## 3단계 — 팀원 작업 프로토콜

1. 계획 선행(계획 없이 착수 금지) — 규모에 따라 shrimp 또는 인라인 계획.
2. N일차 업무 수행 → **본인 개발 범위만** 테스트 (타 팀/연동 지점은 팀장 몫).
3. **`ROADMAP.md`를 읽지도 편집하지도 않는다.** 갱신은 팀장이 7단계에서 일괄
   반영한다. shrimp 전체 상태 갱신도 팀장 전용.
4. 완료 보고 템플릿 (**15줄 이내**):
   - 작업 요약
   - 변경 파일 목록
   - 본인 테스트 결과(통과/실패) — 필수
   - ROADMAP 반영 요청: `Task NNN / 체크할 항목 / 한 줄 결과`
   - 이슈 후보(있으면)

## 4단계 — 라운드 중 추가 팀원

예상 밖 필요 인력(버그 수정, 팀장 검증 보조) 발견 시 팀장이 즉시 추가 spawn
가능. 2~3단계 규칙(sonnet 고정, 부분 읽기, 계획 선행) 동일 적용.

## 5단계 — 검증 (팀원 간 교차 점검 없음, 팀장 단독)

**전제조건**: 참여 팀 **전원의 개발이 끝나야만** 시작한다. 일부만 먼저
끝나도 부분 검증 금지 — 미완료 팀이 있으면 계속 대기.

1. 전원의 완료 보고(테스트 결과 포함) 확인.
2. 팀장이 전체 결과물(변경 사항, 팀 간 연동, 전체 정합성) 직접 검증.
3. 이슈 발견 시 **해당 팀원에게 구체적으로 피드백**(파일/기능/문제 명시).
4. 팀원이 본인 범위 수정 + 재테스트 후 재보고.
5. 팀장 재검증, 미해소 시 3~4 반복. 전원 해소되면 6단계로.

## 6단계 — 일차 작업 로그

검증 완료 후 이슈 유무와 **무관하게 항상** `docs/dailyWorkLog/{N}Day.md` 작성
(이슈 없으면 "이슈 없음" 명시, 생략 금지). 참여 팀 목록·작업 요약 포함.

- **본문 12KB 이내**. 상세 근거·긴 로그 전문 붙여넣기 금지 — 요약 + 커밋 해시·
  파일 경로 참조로 대체.
- 절 순서 고정: 팀별 작업 / 팀장 검증 / 이슈 / **다음 일차 인계** /
  **미해결·판정 대기**. 뒤의 두 절은 **항상 마지막 두 절**이어야 한다
  (다음 세션 0단계가 이 위치를 전제로 부분 읽기한다).

## 7단계 — 전체 완료 후 처리

1. shrimp 전체 상태 갱신(팀장).
2. **`ROADMAP.md` 일괄 갱신** — 전 팀 보고의 "ROADMAP 반영 요청"을 모아 한 번에.
   해당 `### Task NNN` 블록만 grep→offset 읽기 후 편집(전문 읽기 금지).
3. `README.md` 갱신, 팀별 산출물 기록은 `docs/team-schedule/outputs/`에 append.
4. `git add`(dailyWorkLog 포함) → 5. 일차·참여팀 명시 커밋
   (예: `chore: N일차 스프린트 완료 - A팀, B팀`) → 6. push.

## Hard Rules (예외 없음)

- **전문 읽기 금지**: `ROADMAP.md`·`docs/ISSUES.md`·팀 md·`{N}Day.md`는 grep으로
  줄 번호를 찾아 offset+limit 부분 읽기만
- 1단계 grep에 행 없는 팀 절대 호출 금지 / 팀원은 담당 md 외 열람 금지
  (`ROADMAP.md`·`ISSUES.md` 포함 — 팀원은 아예 열지 않는다)
- **`ROADMAP.md` 편집은 팀장 전용**(7단계 일괄) / shrimp 전체 상태 갱신도 팀장만
- 계획 선행 필수(규모에 따라 shrimp 또는 인라인) / CoT는 판단이 갈리는 지점만
- 완료 보고는 템플릿 15줄 이내 / dailyWorkLog 본문 12KB 이내
- 전 팀원 모델 `sonnet` 고정 / 일차 진행은 명시적 요청 시에만(자동 트리거 금지)
- 팀원 패널에서 미사용, 오케스트레이션은 항상 리더 세션
- 팀원 간 교차 점검 없음: 본인 범위만 테스트, 전체 검증은 팀장 단독 + 이슈 시
  피드백→재수정→재검증. **검증은 참여 팀 전원 개발 완료 후에만 시작**
- dailyWorkLog는 이슈 유무 무관 매 일차 필수 작성(생략 금지)
- 새 세션 시작 시 팀원에게 먼저 묻지 않고 최신 `{N}Day.md`의 마지막 두 절만 확인
  (idle 팀원 있을 때만 초경량 질의)
- 여러 일차 요청도 한 일차씩 순차(commit&push 완료 후 다음) / 팀 활성 중
  `/compact` 금지 / 다음 일차 팀원은 새 spawn(컨텍스트 미승계)

## 실행 예시

"5일차 진행해줘" →
0) 최신 `{N}Day.md`의 인계·미해결 절만 부분 읽기(ROADMAP 안 읽음) →
1) `grep -n "^| \*\*5\*\* " docs/team-schedule/0*.md`로 참여 팀 + 작업 행 확보·보고 →
2) 해당 팀만 sonnet spawn (작업 행 원문 주입) → 3) 각 팀 계획→작업→본인 테스트→
15줄 보고 → 4) 전원 완료 확인 후 팀장 단독 검증, 이슈 시 피드백→재수정→재검증 →
5) `dailyWorkLog/5Day.md` 기록(12KB 이내, 인계·미해결이 마지막 두 절) →
6) shrimp/**ROADMAP 일괄**/README 갱신, commit&push →
7) 팀원 전원 종료 확인 후 `/compact`(범위 진행 중이면 다음 일차로).
