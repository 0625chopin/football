## 3. 비주얼 디렉션 — "Floodlit(야간 조명)"

### 3-1. 왜 밋밋했나 (원인 규명)

24~26일차 Task 012가 확정한 것은 **시맨틱 5색·브레이크포인트·타이포 스케일**이었고, **중립색은 create-next-app/shadcn 기본값이 그대로 남아 있었다**:

- `--background`~`--sidebar-*` 전량이 **채도 0의 순수 회색**(`oklch(L 0 0)`)
- **`--chart-1~5`가 명도만 다른 회색 5단계** — 계열 구분이 사실상 불가능
- 다크 모드 `--primary`가 `oklch(0.922 0 0)` = 거의 흰색 → 브랜드가 아예 없음

34~35일차에 화면이 처음 렌더되면서 이 상태가 드러난 것이다. **디자인이 나빴다기보다 아직 정해지지 않은 축이 있었다.**

### 3-2. 확립한 명제

이 제품의 정체성을 **"지금 계산되고 있는 가상 리그 세계를 *중계로* 지켜본다"**로 잡았다. 세계 시계·페이즈·배속·일시정지가 있는 시뮬레이션이라는 점이 이 제품에만 있는 성질이고, 그것이 화면에 보여야 한다고 판단했다. 따라서 화면의 재질은 **대시보드가 아니라 중계 화면 + 매치데이 프로그램**이다.

### 3-3. 핵심 판단 3가지

| 판단 | 근거 |
|---|---|
| **브랜드 브라이트 = 조명 호박색(hue 68~70)** — 초록을 쓰지 않는다 | `--promotion`(승격)이 이미 초록이다. 브랜드 초록과 겹치면 시맨틱 색의 구분력이 희석된다(NFR-A11Y-002가 요구하는 것은 "색을 보조 신호로 쓰되 그 구분력은 보장하라"이다) |
| **중립 = 야간 잔디 잉크(hue 172~180, C 0.006~0.03)** | 무채색 대신 청록 쪽으로 아주 약간만 기울인다. 회색이 "죽어 보이는" 문제의 직접 원인이었다 |
| **`--board-*` = 라이트/다크 공통으로 항상 어두운 중계 표면** | 경기장은 사용자의 OS 테마와 무관하게 늘 야간이다. 라이트 모드에서 헤더·푸터·라이브 보드가 어두운 것은 결함이 아니라 의도다 |

### 3-4. 시그니처는 하나

**홈의 라이브 보드** — 세계 시계(시즌·페이즈·카운트다운)와 진행 중 경기를 같은 어두운 표면에 올려, 둘이 하나의 상태라는 것을 재질로 말한다. 나머지 화면은 전부 조용하게 뒀다.

### 3-5. 색상 안전성 — 코드 작성 **전에** 수치로 검증

`contrast.test.ts`와 동일한 수식(OKLCH→linear sRGB→CVD 행렬→ΔE / WCAG 대비)으로 후보 팔레트를 스크립트로 전수 실측한 뒤 CSS를 썼다. 최소 여유 구간:

- **다크 `--relegation` 대 배경 3.599:1**(하한 3) — 다크 `--background`의 L을 0.163보다 올리면 깨진다
- 라이트 `--muted-foreground` 대 `--muted` 5.108:1(하한 4.5)

**시맨틱 5색(ΔE 여유 0.56, I-144)은 일절 건드리지 않았다.**

### 4-3. 4상태 파일 60개 → 공용 껍데기 3개

`loading`/`error`/`not-found` × 20라우트 = 60개 파일이 **주석 한 줄(라우트 경로)만 다른 완전 동일 파일**이었다. `RouteLoading`/`RouteNotFound`/`RouteError`(신규, `src/components/state/`)로 모으고 60개는 얇은 래퍼로 만들었다.

이 통합이 실제로 결함 하나를 드러냈다 — **35일차 I-170의 쿠키 로케일 근사가 루트 `[lang]/loading.tsx` 한 곳에만 적용되고 나머지 19개는 `DEFAULT_LOCALE` 고정으로 남아 있었다.** `/en/*`의 로딩 화면이 항상 한국어였다는 뜻이다. 껍데기로 모으면서 20개 전부가 같은 근사를 쓰게 됐고, **이 종류의 표류가 구조적으로 불가능해졌다.**

- `<main>` 중첩 제거(레이아웃이 이미 `<main>`을 연다 — 홈 `page.tsx`도 같은 문제였다)
- not-found에 **홈 복귀 버튼** 추가(종전엔 막다른 길)
- error에 16일차부터 카탈로그에 있었으나 **아무도 쓰지 않던 `error.generic.description`** 연결

## 5. 팀장 검증 (Playwright 실측)

### 5-1. 반응형 매트릭스 자동 판정 — **I-153 최초 실측**

27일차에 "소비처 0곳 + Playwright 미설치"로 **미검증 종료**됐던 Task 012 수락 기준("두 테마 × 두 로케일 레이아웃 깨짐 0")을 이번에 처음 실제로 쟀다.

**뷰포트 6종(320/375/768/1024/1440/1920) × 테마 2종 × 경로 7종 = 84조합**을 `documentElement.scrollWidth > clientWidth`로 자동 판정 → **전건 오버플로 0**.

**눈으로 훑지 않고 자동 판정한 것이 결정적이었다** — 처음 돌린 매트릭스가 눈으로는 못 본 결함을 잡았다:

| 발견 | 수치 | 조치 |
|---|---|---|
| 320px 사이드바 오버플로 | 536 → 320 | `SideNav` 접기 + `main` `min-w-0` (**I-183 해소**) |
| **자리표시자 18개의 `<pre>` 미줄바꿈** | 355 → 320 | `whitespace-pre-wrap` + `overflow-x-auto` 부여 |

두 번째 건은 **ROADMAP에 "깨짐 0"이라고 먼저 적어 두려던 참에 매트릭스를 돌려서 잡혔다.** `<pre>`는 기본이 `white-space: pre`라 한 줄짜리 JSON이 문서를 밀어낸다. 자리표시자라 Task 015~021에서 곧 사라질 마크업이지만, 남아 있는 동안 NFR-RS-001 하한을 깨뜨리고 있었다. 같은 편집에서 `<main>` 중첩(루트 레이아웃이 이미 연다)도 제거했다.

> **교훈**: 반응형 "깨짐 0"은 스크린샷으로 판정하지 말 것. 사이드바 오버플로는 눈에 띄었지만 자리표시자 35px 초과는 스크린샷에서 전혀 보이지 않았다.
