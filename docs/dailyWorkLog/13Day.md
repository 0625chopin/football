## 4. 팀장 확정 — 공통코드 범위 검증 (확정 1~4 + 판정 대기 1건)

### 확정 1 — 16일차: **CHECK가 아니라 트리거**로, 조건부로, 형태만

3팀·6팀이 **독립적으로** 같은 기술적 문제에 도달했다: `schema-design.md:848`의 `min_value ≤ value_num ≤ max_value`는 **plain CHECK로 구현 불가능**하다. `min_value`/`max_value`는 **E-41**, `value_num`은 **E-42** — 다른 테이블이고, Postgres `CHECK`는 같은 행의 컬럼만 참조할 수 있다(서브쿼리도 불가). §6.2.2의 "CHECK" 표기 자체가 부정확했다.

조건은 `min_value IS NULL OR (value_num BETWEEN min_value AND max_value)` — **NULL 통과**. 지금 넣어 두면 3팀이 언제 어떤 그룹부터 값을 채우든 **채워지는 즉시 자동으로 보호가 걸린다**. 37일차로 미루면 16~36일차 공백이 생기고, 그때 추가하는 비용이 더 크다.

### 확정 2 — 값 채우기는 **37일차 이후, 그룹별 점진**. 16일차까지 채우는 안은 기각

3팀 근거 3건 전부 타당: ⑴ 05문서 5.12.1절 표에 그룹별 허용 범위 컬럼이 애초에 없어 근거가 새로 생기지 않음 ⑵ 확정 3의 구조 문제 ⑶ `ODDS_PARAM`·`RATING_WEIGHT`·`OVR_WEIGHT`·`MANAGER_MATCHUP` 같은 엔진 튜닝 그룹은 **올바른 범위가 031b 밸런싱 루프(66~68일차, R-01)에서야 실측으로 드러난다** — 52일 앞서 확정하면 억측이거나 나중에 되돌려야 한다.

### 확정 3 — **범위의 단일 소스는 DB**(`common_code_group`)

FR-AD-011(배포 없이 밸런싱, 매직 넘버 하드코딩 금지)·FR-AD-012(범위 검증)가 공통코드 체계의 존재 이유다. 이미 `value` 자체에 쓰는 우선순위(월드 오버라이드 → DB 전역 기본값 → TS 하드코딩 폴백, `loader.ts`)와 **동형**이다. `catalog.ts`의 `null`은 "하드코딩 폴백에 범위 없음"일 뿐 소스가 아니다.

> **3팀의 축 구분이 정확했다**: CLAUDE.md의 "TS가 옳다"(I-58) 원칙은 **엔티티 구조/타입** 충돌에 적용되는 것이지, 이건 **튜닝 가능한 런타임 데이터**라 다른 축이다.

### 확정 4 — `minValue`/`maxValue`/`jsonSchema`를 E-41 → **E-42로 이동 (채택, 당일 반영)**

> **결론**: 1팀이 배치 반영을 **승인(A)**하고 **13일차 당일 반영**했다. 3필드 전부 이동. → **I-93**
> **부수 효과**: 확정 1의 "트리거로"가 **무효가 되고 plain CHECK로 되돌아간다** — 같은 행에 모였기 때문. 6팀 예측대로 16일차 작업이 줄었다.
> **6팀 14일차 착수 범위**: 보류 없이 예정대로 E-41~E-47 전 7테이블.



**6팀 논거 3건**:
1. **`unit` 선례** — `unit`("분"·"%"·"pt")은 이미 **E-42 코드 레벨**에 있다. `value_type`은 "이 그룹은 전부 이 물리 타입으로 저장한다"는 **스토리지 계약**이라 그룹 공유가 자연스럽지만, `min/max`는 "이 파라미터가 의미상 어디까지 허용되는가"라는 **도메인 지식**이라 `unit`과 같은 축이다.
2. **16일차 작업이 오히려 줄어든다** — E-42로 내리면 `value_num`·`min_value`·`max_value`가 **같은 행**에 모여, 확정 1의 트리거가 **다시 plain CHECK로 돌아간다**. 트리거보다 CHECK가 싸고 실패 지점이 적다.
3. **지금이 가장 싼 시점** — E-41/E-42는 **아직 테이블이 없다**(14일차 생성 예정). `ALTER TABLE` + 데이터 마이그레이션이 아니라 **최초 `CREATE TABLE` 컬럼 정의만** 바꾸면 된다.

**제약**: `src/types/config.ts`의 `CommonCodeGroup`은 **8일차 동결 타입**이라 변경은 **이슈 배치 반영**만 가능하며 그 주체는 1팀이다. 팀장이 임의로 결정할 수 없어 1팀 판단을 요청했다.

**함께 판단할 것**: `jsonSchema`도 현재 E-41에 있고 **동형 문제일 수 있다**(한 그룹 안의 JSON 코드들이 서로 다른 스키마를 가질 수 있음 — 오늘 `CUP_PARAM`이 정확히 그 사례). 나눠서 두 번 고치는 것이 최악이므로 함께 판단하도록 지시했다.

**영향 범위 예고**(6팀): 채택 시 §8.2 H-08 대응표의 E-41(13→11컬럼)·E-42(16→18컬럼) 카운트가 바뀐다.

### ⚠️ 절차 위반 1건 — 소유 경로 밖을 깨뜨린 채 완료 보고

- 1팀은 `catalog.ts`가 3팀 소유라 **자신이 고칠 수 없다는 것을 알면서** 착수했다. 자기 경로만 고치고 남의 경로를 깨진 채 두는 것은 완료가 아니라 **다른 팀에게 떠넘긴 미완**이다.
- 이번엔 3팀이 대응 가능한 시간대라 손실이 작았으나, 3팀이 이미 종료했다면 **당일 커밋 자체가 불가능**했다.
- **확정 원칙**: 소유 경로 밖을 깨뜨리는 변경은 ⑴ 해당 팀의 대응 가능 여부를 먼저 확인하거나 ⑵ 양 팀이 같은 시점에 함께 반영하도록 팀장이 조율한 뒤 착수한다. **판단을 먼저 회신하라고 한 이유가 그 조율을 위해서였다.**
- 팀장은 1팀의 "커밋 진행하셔도 될 것 같다"는 의견에도 **커밋을 보류**했다 — 게이트 빨간불 상태의 커밋은 1팀 자신이 15일차에 세울 H-03 3단 머지 게이트가 금지할 일이므로, 오늘 그 원칙을 그대로 적용했다.

**후속 조치**: 3팀 — `catalog.ts` 56~68행 `Pick<>`에서 3키 제거 + 36항목 × 3줄 = 108줄 제거 + 25~30행 stale 주석 갱신(값은 **채우지 않는다** — 확정 2 유효). 6팀 — `schema-design.md` §3.11 컬럼표 이동, 848행 CHECK 문구를 "트리거 불필요, plain CHECK로 충분"으로 정정, H-08 대응표 컬럼 수 재계산.

### 5.2 `snapshot_id` 순서 의존 (6팀) — 금지선 기록

- `fixture.snapshot_id`: NOT NULL, **FK 없음**(E-44가 14일차라 연결 불가). `season.snapshot_id`: nullable, FK 없음.

- 실제 노출은 낮다 — `@supabase/*` 미설치라 앱 코드에서 쓸 경로가 없고 MCP 직접 호출이 유일한 수단.

### 5.8 3팀 PRNG 사용 — 3중 검증

`generate.ts`의 블랙리스트 재추첨 루프에서 state 재사용 버그가 없는지 **팀장·2팀·3팀 자체 테스트 3중으로 확인**. `let cursor = state`가 루프 밖, `cursor = familyStep.state`가 루프 안 — 매 시도 전진하므로 무한 동일값 루프 없음.

**2팀이 모듈 소유자로서 보강한 계약 확인**:
- `bound=0` → `RangeError`(조용한 이상값 생성 없음). `namePools.ts`의 모든 풀이 10개 이상이라 미발동.
- 모듈로 편향 → `limit = 2^32 − (2^32 % bound)` 거절 샘플링. 거절 횟수까지 재현되므로 결정론 유지.
- **⚠️ `bound === 1`일 때만 입력 `state`를 그대로 반환**(entropy 미소비, 선택지가 하나뿐이라 의도된 설계). "매 호출마다 state가 전진한다"는 가정이 깨지는 **유일한** 경우 → **I-91**.
