### 2.2 시드 계층 파생 — 구조적 독립성 3중 보장

배당 프리시뮬 독립성을 "우연히 안 겹침"이 아니라 구조로 요구했고, 2팀이 세 겹으로 설계했습니다.

1. **상위 2비트 네임스페이스 예약**(payload 하위 30비트) → 두 네임스페이스의 시드 값 집합이 **수학적으로 서로소**
2. **자식 시드가 부모 태그 상속** → 어떤 깊이에서도 프리시뮬이 본경기로 넘어올 수 없음
3. **초기 PRNG 상태까지 서로소** — `createState`의 첫 워드가 `splitmix32(seed+φ)`이고 splitmix32는 XOR-shift·홀수곱만으로 구성된 **32비트 전단사**이므로 `seed₁≠seed₂ ⇒ state₁≠state₂`

3번이 핵심입니다. 시드 값이 달라도 PRNG 초기 상태가 충돌하면 스트림이 겹치는데, **1일차 자기 산출물(`prng.ts`)의 성질을 근거로** 그 가능성까지 배제했습니다.

**드러난 구조적 문제**: 2팀이 `grep -n "xg" ROADMAP.md` = **0건**을 확인해 준 덕에, 2팀의 누락이 아니라 **ROADMAP 자체의 누락**임이 밝혀졌습니다. 공수 추정이 전부 ROADMAP 체크박스 개수에서 역산되는 구조라, **ROADMAP이 놓친 것은 공수에도 잡히지 않습니다.** 같은 성격의 누락이 다른 Task에 있을 수 있습니다.

**공수 보정**: 023 6.5 → **7.0** / 024 6.5 → **7.0** / 팀 합계 40.5 → **41.5인일**, 부하율 115% → **118%**.
완화 요인 — 틱 엔진은 슛 판정 시 어차피 득점 확률 `p`를 계산하며 **그 `p`가 곧 xG**입니다. 신규 모델이 아니라 "계산해 쓰고 버리던 값을 기록·합산"하는 성격이라 +0.5×2에 그쳤습니다. 일정 영향 없음(구간 내 흡수, 인계일 불변).

**산출식 (가) 최소안 확정** — 슛 시점 득점확률 `p` = xG, 슛 위치는 추상 존(zone) 코드. 좌표 기반 곡선(+1.0~1.5인일)은 3차 확장 여지.

**세운 원칙**: *"요구사항에 근거가 없다"는 판단은 반드시 `docs/require/` 원문을 직접 확인한 뒤에만 내린다. ROADMAP은 파생물이지 원본이 아니다. ROADMAP↔요구사항 불일치를 발견하면 판정이 아니라 불일치 사실 자체를 보고한다.*

### 3.6 E-2 `ASSIST`↔`GOAL` — 원자성 보장 2겹

**참조 방향 `ASSIST` → `GOAL`(자식→부모) 확정.** 근거: UI가 `GOAL` 행을 그릴 때 "나를 가리키는 ASSIST"를 찾으므로 소비 측 분기가 적습니다.

5팀이 **깜빡임 실패 모드**를 지적했습니다 — `GOAL`만 노출되고 `ASSIST`가 다음 틱으로 밀리면 "도움 없는 골"이 잠깐 표시됩니다. 2팀 답변:
- 엔진 구조상 둘은 **동일 틱 단일 판정에서 함께 생성**되어 `minute`이 항상 같음
- 그럼에도 **컷오프가 인덱스 단위로 자르면** 한쪽만 잘릴 수 있으므로 두 겹으로 차단
  1. **컷오프 단위 = 분(틱)** — 같은 분 이벤트는 전부 포함되거나 전부 제외
  2. **원자 그룹 불변식** — 참조로 묶인 이벤트는 같은 틱에만 존재 + 15일차 Vitest에 "참조 대상이 컷오프 밖인 이벤트 0건" 검증

`OWN_GOAL`·`PENALTY_SCORED`·`PENALTY_SHOOTOUT`에는 도움 없음. **`ASSIST`는 정규·연장 `GOAL`에만 선택적**이므로 UI는 **"골인데 도움 없음"을 정상 상태로** 처리해야 합니다.

`npx tsc --noEmit`이 **종료코드 1**로 실패합니다. 원인은 팀 코드가 아니라 Next.js 생성물입니다.

```
.next/dev/types/routes.d.ts(57,3): error TS1128
.next/dev/types/routes.d.ts(58,1): error TS1128
```

파일 끝에 `React.ReactNode`의 앞부분이 잘린 `ct.ReactNode`가 덧쓰여 있습니다. `tsconfig.json`의 `include`에 `.next/dev/types/**/*.ts`가 있어 검사 대상입니다.

### 7.2 개발 서버가 이미 깨져 있음
```
HTTP 500 (Internal Server Error)
```
`routes.d.ts` 생성 시각(03:25)이 개발 서버 기동 시각과 같고, 복구 전후 모두 500입니다. **서버가 기동 직후 타입 생성 중 실패한 뒤 계속 500**을 반환하고 있었습니다.

### 7.3 팀장 조치 시도와 실패
1. 깨진 파일만 삭제(백업 후) → `validator.ts`가 `routes.js`를 못 찾아 **다른 에러 3건**
2. `page.tsx` touch + `localhost:3000` 요청으로 재생성 유도 → **500, 재생성 안 됨**
3. **백업 복구 → 원상 복귀**

**판단 착오였습니다.** "지우면 개발 서버가 재생성한다"고 보고 실행했으나 서버가 이미 재생성 불능 상태였습니다. 되돌렸지만 **서버 상태를 먼저 확인했어야** 했습니다.

### 7.4 필요한 조치
개발 서버 재시작이 필요하며 **팀장 터미널에서** 수행해야 합니다(타 세션 프로세스 종료는 하지 않음).
```
pkill -f "next dev" && rm -rf .next && npm run dev
```

**현재 영향은 제한적입니다** — 1·2일차 산출물은 타입·문서·순수 함수뿐이고 화면 코드가 없어 진행에 지장 없습니다. 다만 **1팀 15일차 `scripts/gate.sh` 작성 전까지 반드시 정상화**돼야 하며, 게이트에서 생성물을 어떻게 다룰지는 **1팀 15일차 정식 과제**로 넘깁니다.

### 7.5 팀장 검증 절차 결함 (정정)
1·2일차에 사용한 검증 명령이 잘못돼 있었습니다.
```bash
npx tsc --noEmit 2>&1 | tail -5; echo "exit=$?"   # $?는 tail의 종료코드
```
**tsc가 실패해도 항상 `exit=0`이 찍혔습니다.** 팀 소스에 에러가 없었던 것은 사실이나 검증 자체는 무효였습니다. **1일차 "tsc 통과" 보고는 "팀 소스 에러 0건"으로 정정**합니다. 이후 종료코드를 변수로 받아 확인합니다.
