## 3. 이슈 — **있음** (신규 5건 I-83~I-87, 기존 갱신 3건)

### 3.1 최우선 — I-46 거짓 해소 (발견: **팀장 2차 검증** / 해소: 1팀 당일 재작업) → **I-84**

1팀이 `vitest.config.ts`의 런타임 `test.include`에 `**/*.type-test.ts`를 추가하고 "I-46 반영"으로 보고·ROADMAP 기록했다. 팀장이 **반례를 직접 주입해 실증**한 결과 거짓 해소였다:

```ts
expectTypeOf<{ a: string }>().toEqualTypeOf<{ b: number }>();  // 명백히 거짓
```
- `npx vitest run` → **통과(초록불)** — esbuild 트랜스폼이 `expectTypeOf` 단언을 런타임에 **완전히 소거**
- `npx tsc --noEmit` → 오류 1건 검출

해로운 점 3가지: ① I-46이 닫힌 것으로 기록돼 아무도 재점검하지 않게 됨 ② 타입 단언이 깨져도 `npm run test`가 초록불이라 CI가 test만 돌리면 회귀를 놓침 ③ 14일차 커버리지 임계 작업에서 vacuous 테스트 10파일만큼 기준이 부풀려짐.

**조치(1팀, (A)안 채택)**: 런타임 `include`에서 제거하고 `test.typecheck: { enabled: true, checker: 'tsc', include: ['**/*.type-test.ts'] }` 도입. **재현 검증을 1팀·팀장이 각각 독립 수행** — 반례 주입 시 `Type Errors 1 failed`로 실제 실패, 제거 후 `24 files / 291 tests / Type Errors: no errors` 복귀. 1팀이 ROADMAP·ISSUES에 거짓 해소였던 경위를 **숨기지 않고 그대로 기록**했다.

**남은 주의 2건**: ⓐ vitest 4.x typecheck는 **experimental** — 메이저 업그레이드 시 breaking 가능 ⓑ 실행 시간 +5초(8.6초 → 13초대), 14일차 커버리지·15일차 게이트 설계에 반영 필요.
