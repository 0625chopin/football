## 2. 이날의 핵심 발견 — **AS-10 무효화 확정**

가정 **AS-10**("스탯은 이벤트 로그로부터 파생 가능")이 **성립하지 않음**이 오늘 확정됐습니다. 세 팀이 각기 다른 각도에서 붙어 도달한 결론입니다.

### 2.1 어떻게 나왔나

| 순서 | 팀 | 무엇을 하다가 | 발견 |
|---|---|---|---|
| ① | **1팀** | Task 004 계약 설계 중 I-34(라이브 컷오프) 1차 판정 | "집계 테이블 직접 서빙 금지, **노출된 이벤트에서 매 요청 재계산**한 값만 응답"을 명문화 |
| ② | **6팀** | 그 계약을 자기 물리 스키마로 구현 가능한지 교차 점검 | **`PlayerStatCoreValues` 56필드 중 약 30개가 `MatchEventType` 23종에 대응 이벤트가 없음** — `enums.ts`에 `PASS`/`TACKLE`/`INTERCEPT`/`CLEARANCE`/`DUEL`/`CROSS`/`DRIBBLE`/`TOUCH` 리터럴 **0건**(직접 grep 확인) |

**②가 핵심입니다.** 1팀이 오늘 세운 계약이 **절반의 필드에는 원리적으로 적용 불가능**했습니다. 성능 문제가 아니라 **소스 데이터 부재** 문제입니다 — 6팀은 인덱스 관점에선 "문제 없음"(경기당 이벤트 수십 건, `(match_id, sequence)` 인덱스로 충분)을 별도로 확인했습니다.

- 재계산 **불가**(~30): `passes*`, `keyPasses`, `longBalls*`, `crosses*`, `throughBalls`, `dribbles*`, `dispossessed`, `touches`, `tackles*`, `interceptions`, `clearances`, `blocks`, `aerialDuels*`, `groundDuels*`, `errorsLeadingTo*`, `punches`, `catches`, `sweeperActions`
- 재계산 **가능**(~25): `goals`/`assists`/`shots*`/`saves`/카드류/`offsides`/`xg`

### 2.2 2팀 판정 — **(b) 직접 카운터**, `MatchEventType` 23종 유지

가장 결정적인 근거는 **AS-10 자신이 이 케이스를 이미 예견해 뒀다**는 것입니다:

> `docs/require/06-prioritization-and-risks.md` §6.6 — AS-10의 **"무효화 시 영향"** 칸: **"스탯 독립 저장 및 별도 정합성 보정 전략 필요"**

즉 오늘은 새 문제가 터진 게 아니라 **이미 문서화돼 있던 분기점에 실제로 도달한 날**입니다. 나머지 근거:

1. **요구사항이 원래 두 개념을 분리**해 뒀음 — FR-MT-002는 "다음 이벤트 타입을 지원한다: [23종]"으로 **닫힌 집합**, FR-ST-001은 "선수 지표를 경기 단위로 **기록**한다"이지 "이벤트에서 파생한다"가 아님
2. **볼륨** — 패스만 경기당 700~1000건. 이벤트화하면 경기당 수천 건 → 5팀 와이어프레임의 "경기당 수백 건" 전제, 16일차 성능 벤치(p95 ≤ 50ms), D-18("이벤트는 타입 코드만 저장")과 전부 충돌
3. **동결 비용** — `MatchEventType` 확장은 `enums.ts`(H-01) 변경 → C-7 배치 필요 + 5팀이 이미 23종 고정 전제로 #1~18 템플릿 작성 완료
4. **결정론은 유지됨** — 이 엔진의 재현성 SSOT는 이벤트 로그가 아니라 **`matchSeed` 재시뮬레이션**(NFR-QA-003 "100경기 전건 일치"). 30개 필드도 같은 RNG 스트림으로 진행 중 증분하는 것이지, **AS-10이 실제로 금지하는 "사후 임의 배분"이 아님**

### 2.3 1팀 최종 판정 — **(a) 채택, (b) 기각**

2팀 근거를 받고 1팀이 I-34에 **9일차 2차 판정**으로 확정했습니다. 2팀이 제안한 문구(`matchSeed` 재시뮬레이션으로 분리 명문화)보다 **더 보수적인 쪽**을 택했습니다:

- `LIVE` 중: **event-backed 필드만** 컷오프 재계산. 비-event-backed 필드는 **0**(자리표시자 — 실제 0이 아님을 명시. `null`이 아닌 이유는 `PlayerMatchStat`이 동결 타입이라 전 필드 non-null)
- `matchRating`은 가장 민감(S-4)해 `FINISHED` 전엔 **중립 고정값(6.0)**만
- **(b) 기각 사유**: "이벤트로 안 남긴다는 사실만 명문화"하면 결국 사전 계산된 최종값을 그대로 노출하는 것과 다르지 않아, **사후 배팅 방지가 핵심인 NFR-SEC-004 취지에 반함**
- `MatchTeamStatComparison.possessionAvg`도 같은 갭(터치 기반) — 동일 처리. 나머지(`shots`/`corners`/`fouls`/카드류/`xg`)는 이벤트 근거 있음
- 안전권 필드 목록은 계약에 고정하지 않고 Mock(007)·Supabase(034) 구현 책임으로 위임, **041 침투 테스트에 "LIVE 중 비-event-backed 필드 비0 노출" 케이스 편입**

### 2.4 11일차로 넘긴 것

2팀이 `stats.ts`(11일차)에서 **56개 필드 전부에 대한 매핑표**를 만들기로 했습니다 — (a) **Tier A**(이벤트 폴드) vs **Tier B**(직접 카운터/재시뮬레이션) 구분 ⊕ (b) Tier A 안에서 여러 이벤트 타입이 같은 필드에 기여하는 폴드 규칙. 5팀·1팀 동의.

**이유**: 오늘만 같은 패턴의 질문이 두 번 나왔습니다(I-60의 FOUL/PENALTY_AWARDED, #17 SAVE의 `saves`/`shotsFaced`가 `PENALTY_MISSED` 서브케이스를 합산하는지). 개별 확인요청으로 반복 처리하지 않고 한 표로 일괄 정리합니다.

**미결(2팀이 팀장 판단 요청)**: Tier B 30개 필드를 라이브 컷오프 시점에 **정확한 부분합**으로 보여주려면 최종 합계만으로는 부족합니다. 후보 — ① 경기 종료 후에만 노출 ② 틱마다 부분 합계 재시뮬레이션 ③ `tick.ts` 틱 배열에 보조 정보 적재. **1팀의 (a) 판정이 사실상 ①에 가깝게 확정**됐으므로, 11일차 착수 전 2팀이 이 전제로 재확인합니다.

- `getMultiAwardRanking({subjectType, limit?})` — "통산 다관왕"은 전 시즌 누적이라 **별도 집계 DTO 채택**. 무한정 자라는 `Award` 원장을 클라이언트가 재집계하는 건 `getPlayerStatRanking`이 확립한 "서버 파생, 클라이언트 재집계 금지"(R-10)와 어긋남
- `getSponsors()`
- `getSponsorContracts({sponsorId?, status?})` — **`SponsorSummary` DTO는 만들지 않음**. "계약 팀 수"는 필터 배열 길이로, "부도 위험"은 기존 `Sponsor.balance`/`bankruptAtSeason`으로 조합 가능. **결측 엔티티가 없는 경우까지 DTO를 만들면 "결측 엔티티에만 비영속 DTO" 원칙이 흐려짐**(`MatchTeamStatComparison`과 대조)

### 5.2 [5팀] 와이어프레임 필드명 오표기 `stoppage` → `addedTime`

2팀 `tick.ts`와 대조하다 **2일차 원문 오류**를 발견했습니다. 실제 도메인 필드명은 `addedTime`인데 04번 문서가 `stoppage`로 적고 있었고, **44일차 구현자가 `event.stoppage`를 참조하면 즉시 깨지는 지점**이었습니다.

**팀장 2차 검증에서 정정 미완을 재적발**: 5팀이 §E-3만 고쳤으나 같은 파일 2곳 + **이를 인용하는 01·03번 문서 2곳**이 남아 있었습니다. 01·03이 "세 화면(01·03·04)이 동일 규칙을 쓴다"고 04를 인용하고 있어, 부분 정정이 오히려 **문서 간 불일치를 새로 만든 상태**였습니다. → 4곳 전량 정정 완료(`grep` 0건 확인). 141행 정정 이력 각주는 **의도적으로 존치**.

### 5.3 [1팀] `person.ts` I-47 주석 오기 — 자체 발견

6팀 스키마를 검토하다 자기 파일에서 발견. "능력치 범위 1~30 검증은 **Task 034**(어댑터) 담당"으로 적혀 있었으나 실제 범위 CHECK는 **Task 032**(마이그레이션) 소관이고 034는 어댑터 코드라 무관. → 주석 정정(구조 변경 0, 팀장 diff 검증 완료).

### 5.4 [6팀] `player.pa` 유출 경로 — 12일차로 이월

`PublicPlayerProfile = Omit<Player,'pa'>`는 **애플리케이션 계층 규약일 뿐**입니다. DC-05가 요구하는 "공개 SELECT" RLS 하에서 **Supabase PostgREST가 `player` 테이블에 자동 생성하는 REST 엔드포인트로 `pa`가 직접 조회**될 수 있습니다 — 어댑터를 우회하는 경로이며, `match_event`가 "경과 시간 필터를 뷰/보안 함수로 강제"하는 것과 **동일 유형의 위험**입니다.

→ 6팀이 12일차 RLS 설계에서 전용 뷰(`player_public`) 또는 `REVOKE SELECT (pa) ON player FROM anon, authenticated`로 반영. **041 침투 테스트 항목**이기도 합니다.

## 8. 운영 이슈 — shrimp task manager 전역 상태 공유

**5개 팀(2·3·4·5·6)이 동일 현상을 독립 보고**했습니다. `split_tasks`를 `clearAllTasks` 모드로 호출하면 **다른 팀의 진행 중 태스크까지 삭제**됩니다. 팀별 격리 없이 단일 저장소를 공유하고 있습니다.

- **산출물 영향 0** — 전 팀이 실제 작업을 파일 시스템(Read/Edit/Write)으로 수행했고, shrimp는 계획 보조로만 썼습니다. `git diff`로 전량 실재 확인.
- **조치**: 발견 즉시 전 팀에 `append` 모드 전용 / `clear_all_tasks`·`delete_task` 금지 공지. 이후 재발 없음.
- **1팀은 애초에 shrimp를 쓰지 못했습니다** — 세션 시작 시점에 MCP가 미연결이라 조사 기반 CoT 계획으로 대체. 결과물 품질에는 영향 없었습니다.
- **후속 필요**: 이 도구를 팀 병렬 운영에 계속 쓰려면 팀별 격리가 필요합니다. 격리가 불가능하면 계획 도구를 팀별 파일 기반으로 바꾸는 것이 안전합니다.

## 10. 총평

**동결이 버텼습니다.** 5개 팀이 각자의 계층으로 동결본을 번역하는 동안 **타입 구조 변경 요구는 0건**이었습니다. 오늘 나온 변경은 전부 **주석 정정 2건**(`stat.ts` 폴드 규칙, `person.ts` Task 번호)뿐이고 둘 다 팀장이 diff로 구조 무변경을 확인했습니다.

**가장 값진 것은 AS-10 무효화를 오늘 발견한 것입니다.** 1팀이 계약을 세우고 → 6팀이 그 계약을 구현 가능한지 물리 계층에서 검증하고 → 2팀이 엔진 관점에서 판정하는 3단 검증이 아니었다면, 이 간극은 **12일차(H-08 필드 대응표)나 44일차(라이브 중계 구현)에서 터졌을 것**입니다. 그때는 이미 Mock·Supabase 양쪽 구현이 잘못된 전제 위에 올라가 있었을 시점입니다.

**주의할 패턴 하나**: 오늘 정정 3건 중 2건이 **"고쳐야 할 곳을 다 안 고친"** 사례였습니다(5팀 `stoppage` 4곳 중 1곳만, 4팀 소유 경로 서술). 문서 간 상호 참조가 촘촘해질수록 **전파 범위 확인까지가 정정**입니다. 부분 정정은 아무것도 안 고친 것보다 나쁠 수 있습니다 — 정정된 문서와 안 된 문서가 서로 다른 사실을 말하게 되기 때문입니다.
