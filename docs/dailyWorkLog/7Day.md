## 3. 교차 점검 — 1차(팀원 간)

세 팀 모두 **소유 경로 침범 0건 / 타입 재정의(C-6) 0건 / 서브경로 import(C-5) 0건**을 상호 확인했습니다.

### 3.1 이날의 최대 성과 — 같은 결함을 세 팀이 서로 다른 경로로 발견

**정규시간 PK 득점 시 `GOAL`과 `PENALTY_SCORED`가 둘 다 발생하는가**라는 질문에, 세 팀이 각자 다른 근거로 도달했습니다.

| 팀 | 발견 경로 | 판단 |
|---|---|---|
| 2팀 | 엔진 이벤트 생성 로직 설계 중 "미정"으로 식별. 우선순위 **낮음**(023-11 착수 전 확인) | 미결로 보류 |
| 5팀 | D1 스코어보드가 `GOAL`/`OWN_GOAL`/`PENALTY_SCORED` **3계열을 fold**해 스코어를 산출하므로, 이중 발생 시 **1골이 2골로 집계**됨을 지적. 우선순위 **격상 요청** | 확정 요구 |
| 2팀(재점검) | 5팀 문서의 fold 로직을 읽고 **"이 로직이 성립하려면 PK 득점은 `PENALTY_SCORED` 단독이어야 한다"** 는 역산 도출 | 단독 발생 제안 |

**팀장 확정: 정규시간·연장 PK 득점은 `PENALTY_SCORED` 단독 발생, `GOAL`을 추가로 발생시키지 않는다** (I-43).

이 건이 중요한 이유는 **엔진 팀만 보면 "스탯 집계 시점에 정하면 되는 낮은 우선순위"였다는 점**입니다. 화면 팀의 fold 전제와 맞붙였을 때 비로소 정합성 버그로 드러났습니다. 교차 점검이 없었다면 023-11(15일차경) 또는 그보다 늦게 발견됐을 것입니다.

### 3.2 부수 확정

- **승부차기는 킥마다 별도 `MatchEvent`** (같은 `type=PENALTY_SHOOTOUT`, 다른 `sequence`/`detail`)로 반복 발생 — `MatchEventType` 리터럴 1종 ≠ 인스턴스 1건 (I-44). 5팀 10일차 문구를 "단일 요약문"이 아닌 "킥 단위 반복 문구"로 설계하도록 사전 전달.
- **W-25 회신** (2팀→5팀): 엔진은 GOAL(seq N)과 ASSIST(seq N+1, `relatedEventSequence=N`)를 **같은 틱의 한 판정 배치**에서 생성. 단 **클라이언트 동시 도달은 엔진 보장 밖**(1팀 H-02 폴링 계약, 11일차)이라는 단서를 5팀 문서에 명시.
- **`CardReason` enum(5팀 제기)** → **신설 보류**(I-41 OPEN). FR-MT-011이 카드 임계값만 규정하고 사유 taxonomy는 요구사항 어디에도 없어, 지금 값을 만들면 근거 없는 가정입니다. 5팀은 8~9일차 카드 문구에서 **"사유"를 TBD 슬롯으로 비워두고 진행**(자유문자열 임시 가정 금지 — 나중에 enum이 들어오면 문구가 틀어짐).
- **`detail` jsonb 인코딩 규약 4건(2팀 제기)** → **타입 추가 대상 아님**(I-42). `detail`은 원래 미확정 스키마 여유 공간이고, 나머지는 계산 로직 소유처·집계 규약 문제입니다.

**#2가 이날 가장 위험했던 항목입니다.** I-16은 **"해소" 상태**라 본문이 최종 기록으로 굳습니다. 8일차 동결을 이 상태로 넘겼다면, 이후 누군가 "시드는 32비트로 확정됐다"는 근거로 I-16을 인용할 수 있었습니다. **해소된 이슈의 본문은 아무도 다시 읽지 않는다** — 이것이 이번에 확인된 사각지대입니다.

### 4.2 팀장 판단 오류 — 5팀을 6일차에 호출하지 않았음

5팀의 "이벤트 23종 중계 문구 초안"은 팀 일정표 §4에 **6~10일차** 배정입니다. 그런데 6일차 로그는 5팀을 *"유휴 구간"* 으로 적고 미호출했습니다. **1일치를 통째로 흘린 팀장 오류입니다.**

- 원인: 5팀 대기 구간표가 `1~5` / `6~10` / `11~14` … 같은 **구간 단위 배정**이라, 팀별 md의 "N일차" 행만 grep하는 방식으로는 걸리지 않습니다. 3·4·6팀의 대기 구간(`1~8일차` 통짜)과 형태가 같아 보여 같은 취급을 해버렸습니다.
- 실제로는 **5팀만 구간 안에 일차 단위 진척이 성립하는 배정**(23종을 나눠 쓰는 작업)이고, 3·4·6팀은 H-01 수신 대기라 성격이 다릅니다.
- 조치: 7일차에 착수하되 4일치(7·8·9·10일차)로 6/6/6/5 분할해 **10일차에 23종이 끝나도록** 재계획했습니다. 잃은 1일은 5팀 자체 여유(27일 유휴 구간)로 흡수되어 일정 영향은 없습니다.
- **재발 방지: 대기 구간표가 "구간"으로 적혀 있어도, 그 안에 일차 단위로 쪼갤 수 있는 실산출물이 있으면 호출 대상이다.**

### 4.3 운영 이슈 — shrimp task manager 공용 인스턴스 충돌

2팀이 `plan_task→analyze_task→reflect_task→split_tasks`로 세운 3개 태스크가, 동시 작업 중이던 다른 팀(1팀 추정)의 계획 수립에 **덮어써져 소실**됐습니다. shrimp가 프로젝트 단위 단일 인스턴스라 병렬 팀이 같은 태스크 공간을 공유하기 때문입니다.

- 이날은 실제 파일 작업·검증이 shrimp와 무관하게 전부 완료돼 피해가 없었습니다.
- 다만 **팀이 3개 이상 동시에 도는 일차(8일차 = 전 6팀)에서는 계획이 서로를 지울 수 있습니다.** 8일차 소환 시 각 팀에 "shrimp 계획이 사라지면 자기 md의 해당 일차 섹션을 단일 소스로 삼아 진행하라"고 지시할 것.

### 4.4 CLAUDE.md 오기술 교정 (팀장 직접)

`CLAUDE.md`가 **"테스트 러너 없습니다"** 라고 기술하고 있었으나, **Vitest는 5일차에 이미 선도입**됐고 `npm test`(`vitest run`)가 동작하며 오늘 기준 **6파일 120테스트**가 돌고 있습니다. 실제와 다른 기술이 남아 있으면 새로 합류하는 팀이 테스트를 새로 도입하려 들 수 있어 교정했습니다.
