### 2.1 34속성 — 카테고리 축

기술 10 / 정신 10 / 신체 8 / GK 6 = **34** (`03-functional-requirements.md` FR-PL-002 :351~358). `PlayerAttribute`(E-08)와 `PlayerAttributeHistory`(E-09)가 `PlayerAttributeValues` 단일 선언을 공유해 C-6(재선언 금지)을 지킵니다. 3일차에 `__unfilled34Attributes?: never` 자리표시자로 **어떤 구현체도 의존할 수 없게** 막아둔 것이 오늘 파괴적 변경 없이 교체되는 것으로 회수됐습니다.

### 2.2 `hash.ts` — 외부 의존을 늘리지 않고 SHA-256을 얻는 법

2팀은 세 선택지를 놓고 판단했습니다.

| 선택지 | 기각/채택 사유 |
|---|---|
| `node:crypto` | **기각** — Node 전용. 6팀 H-09(Edge Function이 엔진 코어를 그대로 호출) 규약과 충돌 |
| `crypto.subtle` (WebCrypto) | **기각** — `digest()`가 항상 Promise. 엔진의 나머지 전부(prng/derive/precision)가 동기 순수 함수인데 여기만 비동기가 되면 시즌 파이프라인 전체에 `await`가 침투 |
| **순수 TS 직접 구현** | **채택** — 완전 동기, 32비트 정수 연산만 사용해 `prng.ts`(xoshiro128\*\*)와 동일한 이식성 논리 확보. 외부 의존 0이라 1팀에 `package.json` 변경을 요청할 필요도 없음 |

성능 트레이드오프는 **상태 해시가 시즌·경기 종료 등 저빈도 지점에서만 호출**되어 핫패스에 들어오지 않으므로 무시 가능하다고 판단했습니다. 검증은 `node:crypto`와 대조하되 **블록 경계(55/56/57/63/64/65바이트)** 를 전건 포함해 패딩 버그를 배제했습니다.

### 2.3 `sort.ts` — tiebreak 키를 타입으로 강제

`TiebreakKeys<T> = readonly [SortKey<T>, ...SortKey<T>[]]` 튜플로 **최소 1개**를 강제했습니다. 임시 파일에 `stableSortBy(items, [])`를 넣어 `TS2345: Source has 0 element(s) but target requires 1`을 실제로 재현해 증명한 뒤 삭제했습니다. NFR-DT-008의 "tiebreak 키 없이 호출 시 타입 오류"를 문자 그대로 충족합니다.

**동률 최종 폴백은 이 헬퍼의 책임이 아닙니다.** 정렬 헬퍼가 `PrngState`를 받으면 정렬과 난수 소비가 한 함수에 섞이고, 호출부마다 규칙이 다릅니다(순위표는 동률 구간 추첨, 컵 시딩은 D-24의 다른 폴백). `findTiedRuns()`로 동률 구간만 뽑아 호출자에게 넘기는 합성 구조로 분리했습니다.

### 3.2 W-38 — 경기 단위 팀 스탯 엔티티가 없다

5팀이 04 문서 D5(팀 스탯 비교)를 정정하려다 참조 대상 엔티티가 정본에 없다는 것을 발견했습니다. 스탯 계열은 E-19 PlayerMatchStat / E-20 PlayerSeasonStat / E-21 PlayerCareerStat / E-22 TeamSeasonStat 4종뿐이고 **팀×경기 축이 비어 있습니다.**

팀장이 정본을 직접 확인한 결과 **실제 공백이 맞았습니다** — FR-ST-002 본문이 *"클럽 지표를 **경기·시즌 단위로 기록**한다"* 로 경기 단위를 명시 요구하고, 수용 기준 ②가 *"시즌 집계가 **경기 스탯 합계**와 정합"* 인데, 경기 단위 팀 스탯이 없으면 그 기준을 검증할 대상 자체가 없습니다. E-22가 필드를 "FR-ST-002 지표 전량"으로만 적어 공백을 덮고 있었습니다.

**1팀 판정 = 옵션2(파생으로 충분, 신규 엔티티 불필요).** 근거:

1. 선수 귀속 지표 → `SUM(PlayerMatchStat) WHERE matchId AND teamId` (`teamId` 필드 존재)
2. 팀 전용 카운트(코너킥) → `MatchEvent.type='CORNER'` + `teamId` COUNT (`MatchEvent`에도 `teamId` 존재)
3. 비율형(`pass_accuracy`·`possession_avg`) → 위 합산값에서 계산. **엔진 알고리즘 선택이지 스키마 공백이 아님**
4. 성적 계열(`biggest_win`·`current_form`) → `Fixture` 스캔
5. 시간대 분포 → GOAL/OWN_GOAL의 `minute`+`teamId` 구간화

**결정적 근거는 부수 논거였습니다** — 별도 `TeamMatchStat`을 저장하면 이벤트 로그와 독립적으로 갱신되는 **두 번째 캐시**가 생겨, I-25/C-23의 "이벤트–스탯 단일 소스·단일 응답 컷오프" 제약이 깨질 경로가 하나 늘어납니다(NFR-SEC-004). **결손을 메우려다 보안 표면을 넓히는 것이 더 나쁩니다.**

**성립 조건**: SUM/COUNT 집계 로직이 **한 곳에만** 구현되어 026(2팀)과 017(5팀)이 공유해야 합니다. 따로 짜면 미묘한 불일치가 나고, 그러면 신규 엔티티를 만든 것보다 나빠집니다. 이 조건을 `docs/devStep/03.*` §4 후속조치 6번으로 등재해 **9일차 Task 004 착수 항목으로 고정**했습니다.

## 7. 팀장 판단 오류와 교정

성과만 적으면 같은 실수가 반복되므로 오늘 팀장이 틀린 것을 남깁니다.

### 7.1 이슈 번호를 선점 지칭해 충돌을 만들었다

팀장이 W-38 판정을 지시하며 *"I-36 등재 여부를 보고하라"* 고 **번호를 먼저 지칭**했는데, 같은 시각 1팀은 ESLint 건을 I-36으로 제안하고 있었습니다. 번호가 겹쳐 1팀이 스왑을 제안했고, 팀장이 다시 기각·재배정하는 왕복이 발생했습니다.

**교정**: 미확정 이슈에 번호를 **미리 붙여 지칭하지 않습니다.** 번호는 등재 시점에 소유자(1팀)가 부여하고 팀장이 승인합니다. 오늘은 "I-37 예약"이라는 편법으로 봉합했으나 애초에 만들지 않았어야 할 혼선입니다.

### 7.2 이미 문서화된 규약을 확인하지 않고 예시를 던졌다

5팀에 와이어프레임 07/08 분할을 지시하며 *"예: 07=운영 콘솔+scheduler, 08=config"* 라는 예시를 붙였습니다. 그러나 `00-공통규약.md`가 **1일차에 이미 07=`/admin`, 08=config+scheduler로 매핑을 확정**해 두었습니다. 5팀이 기존 규약을 지키고 팀장 예시를 따르지 않은 것이 옳았습니다.

**교정**: 팀장이 예시를 줄 때는 **해당 영역에 이미 확정된 규약이 있는지 먼저 확인**합니다. 팀장의 즉흥 예시가 문서화된 합의를 흔들면, 팀원은 규약과 지시 중 무엇을 따를지 매번 판단해야 합니다.

### 7.3 3일차 계획 자체가 모순이었다

3일차 §7이 F·G·H(E-16·E-17·E-20 관련 변경요청)를 **5일차**로 배정했는데, 같은 표가 E-15~E-20 **정의를 4일차**로 배정했습니다. 같은 필드를 4일차에 만들고 5일차에 다시 고치는 계획이었습니다. 1팀이 착수 전 질의해 (b)로 정리됐습니다.

**교정**: 변경요청을 일차에 배분할 때 **그 필드의 정의일과 겹치는지 대조**합니다. 정의일이 더 늦으면 정의일로 흡수하고, 더 이르면 별도 반영일을 잡습니다.

### 7.4 지시–보고 타이밍 관리에 실패했다

메시지 크로스가 **최소 5회** 발생했습니다. 팀원이 유휴로 전환되는 시점과 팀장 지시가 엇갈려, 1팀은 같은 보고를 3회, 2팀은 판정을 2회, 5팀은 교차 점검 결과를 2회 반복 전송했습니다. 팀장도 "미수행"으로 오판해 재지시를 보냈습니다.

**교정**: 팀원 보고가 도착하면 **즉시 다음 지시를 보내지 않고**, 그 팀이 진행 중인 작업이 있는지 확인한 뒤 보냅니다. 여러 건을 지시할 때는 **완료 보고 지점을 명시**해(오늘 "1번 완료 시 1보, 나머지 완료 시 2보"처럼) 왕복을 줄입니다. 이 방식을 쓴 마지막 지시에서는 크로스가 발생하지 않았습니다.
