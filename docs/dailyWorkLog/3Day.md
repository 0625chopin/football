### 2.1 타입 골격 — 4일차 이후를 위한 "자리 남기기" 설계

1팀은 오늘 범위(E-01~E-08) 밖 파일을 **빈 파일이 아니라 자리표시자**로 만들었습니다. 이 선택이 이후 일차의 재작업을 줄입니다.

- **`brand.ts`** — 지금은 `type TeamId = string` 별칭뿐이고 **7일차에 브랜드 타입으로 승격**합니다. 3~6일차 파일들이 `string`을 직접 쓰면 7일차에 전 파일을 고쳐야 하므로, ID 참조점을 한 곳으로 모았습니다.
- **`enums.ts`** — E-01~E-08이 요구하는 **최소분만** 선언하고 6일차에 전량 확정합니다. 각 파일이 enum성 유니온을 개별 선언하면 C-6(재선언 금지) 위반이므로 단일 선언 지점을 먼저 확보했습니다.
- **`PlayerAttributeValues`** — `__unfilled34Attributes?: never`로 두어 **어떤 구현체도 의존할 수 없게** 했습니다. 4일차에 34속성으로 교체할 때 파괴적 변경이 나지 않습니다.

일정 앞당김이 아니냐는 점을 확인했고, **아닙니다** — 6·7일차 소관(enums 전량 확정, 브랜드 승격)은 손대지 않았고 각 파일 주석에 해당 일차를 명시해 두었습니다.

### 2.2 `precision.ts` — epsilon을 쓰지 않는 이유

부동소수 비교에서 흔한 `Math.abs(a-b) < 1e-6` 방식은 **이행성이 깨집니다**(a≈b, b≈c인데 a≉c). 2팀은 확률을 **6자리 정수(마이크로 단위, scale 1,000,000)로 정규화한 뒤 정수 비교**하는 구조를 택해 오차가 존재할 수 없게 만들었습니다.

반올림은 이원화했습니다. ① 0.5 경계에서 1e-9 이상 떨어지면 `Math.round`가 증명적으로 정확(구간 [0,1]의 스케일 오차 상한 ≈1.11e-10) ② 0.5 근방만 ECMA-262가 규정한 `toFixed`로 처리.

검증: 대표 오차 케이스(0.1+0.2==0.3 등) 전건, 무작위 30만 건 참조 구현 일치, units 왕복 1,000,001건, p=0/p=1 판정 각 20만 회 → **전건 PASS**.

## 3. 교차 점검 결과

### 3.1 `World` 타입에 시간 환산 앵커가 없다 — **3일차 최대 발견**

2팀이 H-24(월드시간↔실시간 환산 계약, 30일차 5팀 인계) 소비자 관점에서 `World`를 검토한 결과입니다.

환산식은 이렇습니다.
```
worldMin = anchorWorldMin + (realNow − anchorRealTime) × speed − 정지구간
```

현행 `World`에는 `speedMultiplier`·`isPaused`·`pausedTotalMinutes`만 있습니다.

| 결손 | 내용 |
|---|---|
| **①(치명)** | **앵커 부재.** 배속이 *언제부터* 현재 값이었는지 모르면 실시각→경기분 변환 자체가 성립하지 않습니다. `createdAt`은 배속이 한 번도 안 바뀐 경우에만 유효합니다 |
| **②(치명)** | **현재 정지 구간의 시작 시각 부재.** `pausedTotalMinutes`는 누적치뿐이라 `isPaused=true`인 동안의 경과분을 뺄 수 없고, 재개 시 누적치 갱신도 불가능합니다 |
| **③(권고)** | 배속·정지 변경을 5팀이 감지할 **단조 증가 리비전**이 없어, 없으면 필드 diff 폴링을 해야 합니다 |

이것은 필드 몇 개가 빠진 문제가 아니라 **계약이 성립하지 않는** 문제입니다. 8일차에 이대로 동결되면 H-24(30일차)를 만들 수 없고, 5팀 015(홈·라이브, 34~38일차)가 연쇄로 막힙니다. **동결까지 5일 남은 지금 잡은 것이 이 교차 점검의 최대 성과입니다.**

→ **I-31 등재. 1팀 5일차 `world.ts` 반영(동결 전 완료), 6팀 12일차 H-08 컬럼 반영.**

### 3.2 시드 payload가 30비트뿐 — 시즌당 충돌 수백 건

2팀 `derive.ts`는 상위 2비트를 네임스페이스 태그(MAIN / ODDS_PRESIM 서로소 보장, NFR-DT-006)로 예약합니다. 따라서 32비트 규약에서 **실효 payload는 30비트**입니다.

시즌당 이벤트 시드 ~6×10⁵개 기준 생일 충돌 기대값 ≈ n²/2^31 ≈ **수백 건/시즌**. 재현성에는 영향이 없고(동일 시드→동일 출력은 그대로) 통계적 상관만 미미하게 발생합니다.

`bigint` 도입은 2팀이 근거 3건으로 기각했습니다 — ①PRNG(xoshiro128\*\*)가 32비트 워드 연산으로만 구성되어 bigint가 경로에 들어올 수 없음 ②JSON 직렬화 불가(스냅샷·Edge Function 왕복에 변환 필요) ③틱 핫패스에서 수 배 느림(023 p95≤50ms 위협).

**해법은 `number` 유지 + 상한만 완화**입니다. 53비트 안전 정수로 하면 payload 51비트가 되어 충돌이 소멸하고, JSON·성능 이점은 그대로입니다.

부수 효과가 하나 있습니다 — **완화하면 물리 컬럼은 `bigint`가 정답이 되어 I-29(시드 컬럼 타입 불일치)도 함께 해소**됩니다. 두 이슈가 반대 방향으로 당기는 게 아니라 같은 방향이었습니다.

→ **I-32 등재. 5일차 T2-a 개정 → 7일차 `brand.ts`. R-03 영향 없음(기준 스냅샷 생성 34일차 이전).**
