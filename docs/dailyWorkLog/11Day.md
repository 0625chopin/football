## 3. 이슈 — **있음**

### 3.1 최우선 결함 — `polling.ts` 서버/클라이언트 경계 (발견: 4팀 / 해소: 1팀)

`polling.ts` 88행에 `'use client'`가 있어 **파일 전체가 클라이언트 모듈**인데, 같은 파일의 `fetchResult`/`fetchListResult`를 JSDoc이 "서버 컴포넌트가 직접 await해도 안전"하다고 예시 코드까지 붙여 안내하고 있었다. RSC 번들러는 `'use client'` 파일의 **모든 export**를 client reference로 치환하므로 "React 미의존 순수 async 함수"라는 사실과 무관하게 서버에서 호출할 수 없다. 13일차 이후 4·5팀이 소비하려 할 때 막혔을 결함이다.

→ `src/lib/data/fetch-result.ts`로 분리, `'use client'`를 파일 첫 줄로 이동, JSDoc 예시 경로 갱신으로 해소(**I-74**).

### 3.2 등록 함수 미호출 반복 (발견: 4팀 / 판정: 팀장 → **D-32**)

`installHardcodedFallback()` 호출처가 코드베이스에 0건이라 `loadConstants('UI_PARAM')`이 항상 실패, "공통코드에서 폴링 주기를 읽는다"는 설계가 죽어 있었다. **I-67과 같은 버그의 두 번째 재발**이었다.

1팀은 "레지스트리가 달라 결합도를 올리지 말자"며 분리를 주장했으나, 호출처를 소비 측에 늘리면 세 번째 재발을 부른다고 판단해 **절충**: `bootstrapDataSource()`는 어댑터 등록 전용 의미를 유지하고, 별도 단일 진입점 **`bootstrapApp()`**을 신설해 내부에서 둘 다 호출. 4팀은 13일차에 `await bootstrapApp()` **한 줄만** 호출한다.

### 3.3 폴링 안전망의 비용 역전 (발견: 6팀 / 판정: 사용자 → **D-29**)

5000/3000ms가 정상값이 아니라 **공통코드 조회 실패 시의 안전망**이었다. 즉 장애 상황에서 시스템이 가장 비싼 설정으로 폴백하는 역전 구조였다. 예산 문서 §2.5 실측 기준 5초 = 월 $133.7 vs 30초 = $1.6.

→ 정상값 5초/3초는 유지, **안전망만 30초/15초**로 하향(1팀 `polling.ts`, 3팀 `fallback.ts` 양쪽). 두 레이어 값이 어긋나면 취지가 우회되므로 항상 동일하게 유지한다.

### 3.4 Tier B 40필드 vs NOT NULL 스키마 (발견: 6팀 / 판정: 사용자 → **D-30**)

6팀 스키마는 공유 56컬럼을 전부 NOT NULL로 정의했는데 2팀 `accumulatePlayerMatchStats()`는 Tier A 16필드만 산출한다. **나머지를 누가 계산해 저장할지 어느 팀도 소유를 선언하지 않은 상태**였다.

→ **NOT NULL 유지**(nullable 완화 기각), 재시뮬레이션 컴포넌트 **소유 2팀 배정**, Task 032 착수 전 계약 확정 + 6팀 재검수 조건.

2팀 후속 설계에서 중요한 규명 3건이 나왔다:
- 재시뮬레이션 대상은 40개가 아니라 **26개**(`NO_EVENT_TYPE`). 9개는 로스터 인계 후 폴드, 5개는 `detail` 스키마 확정 후 폴드 — **56필드 산출 경로가 3갈래**다.
- "matchSeed 재시뮬레이션"은 **tick 엔진 재실행이 아니다** — 다시 돌려도 같은 23종 로그만 나온다. 별도 시드 통계 모델이며 Tier A→B 단방향 의존.
- **I-65 근본 원인은 `stats.ts`가 아니라 10일차 `events.ts`** — 구조 마커(`FULL_TIME` 등)가 사건 이벤트와 같은 확률 가중치 풀에서 추첨돼 생성되지 않을 수 있다. 신 31일차에 `ensureStructuralMarkers`로 해소.
