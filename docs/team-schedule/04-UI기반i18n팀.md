**005를 이 팀에 이관한 근거 — 이번 재편에서 가장 중요한 결정입니다.**

ROADMAP Task 005의 구현 사항에 이렇게 적혀 있습니다.

> *"Task 011에서 확정한 로케일 라우팅 전략과 정합하도록 세그먼트 구조를 결정 (경로 세그먼트 방식 채택 시 전 라우트가 로케일 하위에 배치)"*

즉 005(라우트)와 011(i18n 라우팅)은 **`src/app/**`의 동일한 세그먼트 구조를 결정하는 한 쌍**입니다. 이를 두 팀이 나눠 가지면 "각 일차에서 팀끼리 최소한으로 영향" 원칙을 정면으로 위반하는 **유일한 동시 편집 지점**이 됩니다. 005를 통째로 이 팀에 넘겨 9~22일차 동안 `src/app/**`의 단독 소유를 보장했습니다.

**대가**: 1팀(코어·품질)의 공수가 4.0인일 줄고 이 팀이 그만큼 늘어, 이 팀이 크리티컬 패스에 올랐습니다. 그래도 동시 편집 충돌보다는 낫습니다.

### 담당 요구사항 ID 범위

FR-UI-000·001·008·009·010·011·012·013·014·020·021·025·026, FR-ST-004, FR-AW-005, FR-YT-006, FR-LG-011~013·015, FR-EC-011, NFR-RS-001~005, NFR-A11Y-001~006, NFR-PF-009, NFR-MT-006
결정 전제: **D-18**(다국어 ko/en), **D-17**(고유명사 번역 대상 아님), D-19(승부차기 별도 표기)

### 소유 경로 — **이 팀만 커밋합니다**

```
src/app/layout.tsx, loading.tsx, error.tsx, not-found.tsx   전역 골격 (루트 레벨 — 세그먼트 layout.tsx는 각 라우트 소유 팀)
src/app/globals.css                                          Tailwind v4 @theme inline 토큰
src/app/sample/**                                            /sample 쇼케이스
src/app/stats/**, awards/**, archive/**, transfers/**        019 담당 화면
src/app/playoffs/**, cup/**, sponsors/**                     020 담당 화면
src/i18n/**                                                  단, messages/*/enums.* 는 3팀 기여
                                                             messages/*/match.ts·bet.ts 는 5팀 기여 (골격·키 구조는 이 팀)
src/components/ui/**                                         shadcn 프리미티브
src/components/domain/**                                     도메인 표현 컴포넌트 8종
src/components/state/**                                      상태·유틸 컴포넌트 6종
components.json
```

**5팀(화면·배팅UX)과의 경계** — 겹치지 않습니다.

| 구분 | 4팀(이 팀) | 5팀 |
|---|---|---|
| `src/app/` | `layout/loading/error/not-found` 전역 골격 + **stats·awards·archive·transfers·playoffs·cup·sponsors** 라우트 | **`/`·leagues·matches·players·teams·admin·bet·my** 라우트의 `page.tsx` 본문 |
| `src/components/` | `ui/`(프리미티브) `domain/`(8종) `state/`(6종) | `composite/`(복합 7종) |

> **9~13일차에 이 팀이 005로 전 라우트의 `loading/error/not-found`를 만들고, 5팀은 28일차 이후에야 `page.tsx` 본문을 만듭니다.** 같은 파일을 같은 날 만지는 상황이 발생하지 않습니다.

### 7.2 학습데이터와 다른 주요 변경점

1. **`params`/`searchParams`는 Promise** — `await params` / `await searchParams` 필수(동기 접근은 v15까지만 하위호환, 향후 제거 예정). layout·page·route·`generateMetadata` 전부 동일.
2. **`PageProps<'/route'>` / `LayoutProps<'/route'>` 전역 타입 헬퍼** — `next dev`/`next build`/`next typegen` 실행 시 라우트 구조 기반으로 자동 생성되는 전역 타입. 별도 import 없이 사용하며, `params`/`searchParams`/named slot(`@slot`)을 강타입으로 추론. 수동으로 `{ params: { slug: string } }` 형태를 다시 선언할 필요가 없다.
3. **`middleware.ts` → `proxy.ts` 개명(middleware는 deprecated)** — i18n 라우팅 가이드의 로케일 감지·리다이렉트 예제도 `proxy.js`에서 수행한다. Task 011(로케일 라우팅 전략)·Task 005(레이아웃 골격) 구현 시 `middleware.ts`가 아니라 `proxy.ts`를 프로젝트 루트(또는 `src/`)에 생성해야 한다.
4. **App Router에는 내장 `next.config`의 `i18n` 옵션이 없다**(Pages Router 전용이던 legacy 옵션과 다름). 다국어(D-18)는 `app/[lang]/` 동적 세그먼트 + `proxy.ts`(로케일 감지·리다이렉트) + 로케일별 딕셔너리(`getDictionary`) 조합을 직접 구현해야 하며, 서드파티 라이브러리(`next-intl` 등) 도입 여부는 아직 미정(`src/i18n/**`는 이 팀 단독 소유이므로 4팀 자체 판단 사항. 단 `messages/*/enums.*`는 3팀이 23일차 이후 값을 채우므로, 라이브러리 선택이 그 파일 형식을 바꾸면 H-09 인계 시 3팀에 통지 필요).

### 7.3 Task 005/011과의 연결 메모

- Task 011(로케일 라우팅 전략, ROADMAP 기준 **14~22일차·L 규모**, 9일 소요)에서 "경로 세그먼트 방식"을 채택할 경우, Task 005의 전 라우트가 `app/[lang]/...` 하위로 재배치되어야 하며 루트 `proxy.ts` 신설이 필요하다는 점을 확정 전에 반드시 반영해야 한다.
- 이번 일차(9일차)에는 위 사실 확인·기록까지만 수행했고, `src/app/**`·`src/i18n/**`·`proxy.ts` 등 실제 코드/디렉터리는 생성하지 않았다(10일차 이후 Task 005 실착수 시 처리).

### 7.4 결정 기록 — `app/[lang]/` 조기 배치 (9일차, 팀장 승인)

> **결정**: Task 005(10일차 실착수)의 전 라우트를 **처음부터 `app/[lang]/` 하위**에 스캐폴딩한다. 평면으로 만든 뒤 Task 011 단계(14~22일차)에서 재배치하지 않는다.
> **승인**: 팀장, 9일차(2026-07-31), 1차 교차 점검(1팀 `DataSource.ts` 검토와 병행 진행) 결과에 근거.

**근거**
1. **일정 구조상 재작업이 확정적**: ROADMAP 기준 Task 005는 9~13일차에 종료되지만, Task 005 체크리스트 자체가 "Task 011에서 확정한 로케일 라우팅 전략과 정합"을 요구한다. Task 011(로케일 전략 확정)은 그보다 늦은 14~22일차에 진행된다 — 즉 **결정 없이 그대로 두면 Task 005가 참조해야 할 전략이 Task 005 종료 시점에 아직 존재하지 않는 구조**이며, 평면으로 만들 경우 15일차 전후로 17개 라우트 × (loading/error/not-found 3파일)를 되돌려야 하는 재작업이 뒤따른다.
2. **선택지가 사실상 하나뿐**: 9일차에 확인한 Next.js 16 공식 문서(§7.1) 기준 App Router에는 내장 `next.config` i18n 옵션이 없고, 공식 가이드가 제시하는 유일한 실질 패턴이 `app/[lang]/` 경로 세그먼트다(도메인 분리 방식은 단일 도메인 서비스라 부적합). D-18(ko 기본/en 2차)도 이미 확정 사항이라 Task 011의 "세그먼트 방식 채택 여부" 자체가 결론이 정해진 질문에 가깝다.
3. **조기 결정 비용이 가장 저렴한 시점**: 5팀의 `page.tsx` 본문 착수는 28일차이고, 현재 `src/app/**`는 이 팀 외에 아무도 건드리지 않는다. 지금 결정하면 충돌·이중작업 없이 반영된다.
4. **되돌릴 리스크는 낮음으로 평가**: Task 011이 비-세그먼트 전략(쿠키 전용 등)을 채택할 가능성은 위 2번 근거로 낮다고 판단.
