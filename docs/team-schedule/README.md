### 이 표를 오해하지 마십시오

**"6명이 10명보다 11일 빠르다"가 아닙니다.** 단축의 원인은 인원이 아니라 **배정 재설계**입니다.

- ROADMAP의 10명 배정은 **팀원 8 한 사람이 Task 011~021을 52.0인일 직렬로 수행**하는 구조라, 나머지 6명이 무엇을 하든 90일이 하한이었습니다. 인원을 10명에서 20명으로 늘려도 90일입니다(브룩스의 법칙).
- 6팀 재편에서 UI 트랙을 **UI기반팀 + 화면팀 2트랙**으로 쪼개면서 이 병목이 74일로 내려갔습니다. 같은 재설계를 10명에 적용하면 74일 부근이 되고, 그때는 여유가 훨씬 많습니다.

**6명으로 줄여서 실제로 잃는 것은 기간이 아니라 여유입니다.**

1. 공수 하한이 37.8일 → 44.1일로 **6.3일 상승**. 이 값이 74일에 근접하면 더 이상 재설계로 단축할 수 없습니다.
2. 비크리티컬 트랙 여유가 **32% 감소**. 한 트랙이 지연되면 다른 트랙에서 사람을 빼 메울 여지가 그만큼 없습니다.
3. **원 팀원 7(배팅)이 전담 팀을 잃었습니다.** 배팅 도메인은 이제 3팀의 부업입니다. Phase 4에서 이 결정이 비용으로 돌아올 가능성이 가장 큽니다.

## 11. 추정 규칙 (전 팀 공통)

**기본 공수 = 미완료 체크박스 수 × 0.5인일**, 이후 보정 계수를 곱합니다.

| 조건 | 계수 |
|---|---|
| `**테스트**` 항목 존재 | ×1.3 |
| 외부 시스템 의존 (Supabase, Edge Function, 인증, shadcn, Vitest 도입) | ×1.3 |
| 알고리즘/도메인 난도 (시뮬 엔진, 확률 산출, 정산, 절차적 생성) | ×1.4 |
| 순수 문서·규약 정리 | ×0.6 |
| **볼륨 보정** (체크박스 1개가 실제로는 다수 산출물 — 013 컴포넌트 21종, 018·019 다화면) | 개별 명시 |

라벨: **S**(≤2인일) / **M**(3~5) / **L**(6~10) / **XL**(>10)
**XL은 분할했습니다** — Task 013(8.5인일, 컴포넌트 21종) → 013A(14종)/013B(7종), Task 031 → 031a(시드)/031b(튜닝 루프), Task 034 → 034a(구현)/034b(전환 검증).

**파싱 규칙**: `### Task NNN: <제목>` 헤더로 Task를 식별하고, 해당 헤더 이후 다음 `###`/`##` 전까지의 `- [ ]` / `- [x]` 를 체크박스로 집계했습니다. `- **담당**:` 줄에서 담당·리뷰어·의존을, `- **테스트**:` 줄 유무로 검증 부하를 판정했습니다. **완료 체크박스 1건(Task 001의 첫 항목)은 잔여 공수에서 제외**했습니다.

**일수 환산**: 공수(인일) ÷ 0.8인일/일 → 올림. 예: 4.5인일 ÷ 0.8 = 5.6일 → 6일.
