### 2.2 Edge Function 실행 시간 — **요금은 $0, 그러나 CPU 2초 한도가 설계 제약**

**Supabase Edge Function은 실행 시간을 과금하지 않는다** (호출 수만 과금). 따라서 실행 시간은 **비용**이 아니라 **한도 리스크**로 취급해야 한다.

**월 총 실행 시간 추정**

```
시즌당 라운드 처리 틱 수
  = 리그1 46 + 리그2 38 + 리그3 30 = 114 라운드
  + 컵 6슬롯 + 플레이오프/정산/프리시즌 약 10틱
  ≈ 130 틱/시즌
월 10시즌 → 라운드 처리 틱 = 1,300회/월
no-op 틱 = 43,834 − 1,300 = 42,534회/월

월 CPU 시간
  = 42,534 × 0.2초(NFR-PF-013 no-op ≤ 200ms)
  + 1,300 × 1.5초(NFR-PF-003 30경기 ≤ 1.5초)
  = 8,507초 + 1,950초
  = 10,457초 ≈ 2.9시간/월
```

| 항목 | 값 | Supabase 요금 | (참고) Vercel Fluid로 이전 시 |
|---|---|---|---|
| 월 CPU 시간 | 2.9시간 | **$0** | Active CPU 2.9h × $0.128 = **$0.37** |

**한도 대비 여유 — 여기가 진짜 문제**

| 시나리오 | CPU 소모 | Supabase CPU 한도 2초 대비 |
|---|---|---|
| no-op 틱 | 0.2초 | 10% — 안전 |
| 라운드 30경기 (NFR-PF-003) | 1.5초 | **75% — 마진 25%** ⚠️ |
| catch-up 1회 상한 50경기 (NFR-CR-004) | **2.5초 (추정)** | **125% — 한도 초과** 🔴 |
| 배당 1경기 N=3,000 (NFR-PF-004) | 10초 | **500% — 단일 호출 불가** 🔴 |

> **결론 ②: 예산 한도가 아니라 CPU 2초 한도가 아키텍처를 제약한다.**
> - `catch-up` 1회 처리 상한 **50경기는 CPU 2초를 초과할 위험**이 있다. → **30경기로 하향** 권고 (4절).
> - 배당 산출은 **반드시 다중 호출로 분할**해야 한다. → 2.4절.
> - 비용을 지불해 이 한도를 푸는 방법은 Supabase 플랜 승급으로는 **불가능**하다(Team $599도 CPU 2초 동일). 유일한 우회는 ⓐ 배치 크기 축소, ⓑ Vercel Function(Fluid, 장시간 실행 가능)이나 외부 워커로 이전 — ⓑ는 월 $1 미만의 추가 비용으로 가능하나 아키텍처 복잡도가 늘어난다.

### 2.4 배당 산출 비용 — 몬테카를로 N값 민감도

**요구 사항 확인**

- `ODDS_PARAM.MC_N_MATCH` = 3,000 / 경기 (NFR-PF-004: **≤ 10초/경기**, 라운드 30경기 ≤ 60초 병렬)
- `ODDS_PARAM.MC_N_SEASON` = 300 (NFR-PF-005: 라운드당 ≤ 90초, 비동기)

**⚠️ 즉시 드러나는 모순**

```
NFR-PF-001: 경기 1건 풀 시뮬 = p95 50ms
3,000회 × 50ms = 150초 / 경기
NFR-PF-004의 상한은 10초 → 15배 초과
```

→ **배당용 프리시뮬은 풀 엔진(90틱 이벤트 생성)을 그대로 3,000번 돌릴 수 없다.**
반드시 **경량 결과 모델**(라인업·계수까지만 반영하고 이벤트 로그를 생성하지 않는 축약 시뮬)이 필요하며, 요구 예산은 **시뮬 1회당 ≤ 3.3ms** = 풀 엔진의 **1/15 비용**이다.
→ 이 사실을 Task 035 착수 전에 팀원 7에게 전달해야 한다. (4절·8절)

**Edge Function 호출 수 환산** (CPU 2초/호출 한도 준수 전제)

```
경기 마켓:
  경기당 CPU 10초 ÷ 2초 한도 = 최소 5호출, 안전 마진 포함 8호출
  월 경기 수 = 1,244경기/시즌 × 10시즌 = 12,440경기
  월 호출 = 12,440 × 8 = 99,520회

시즌 마켓:
  라운드당 CPU 90초 ÷ 2초 = 45호출
  월 라운드 = 130 × 10 = 1,300라운드
  월 호출 = 1,300 × 45 = 58,500회

배당 관련 월 총 호출 ≈ 158,020회 → 요금 $0 (Free·Pro 한도 내)
월 CPU 시간 = 12,440×10초 + 1,300×90초 = 124,400 + 117,000 = 241,400초 ≈ 67시간/월
```

| 항목 | Supabase 요금 | (참고) Vercel Fluid Active CPU 환산 |
|---|---|---|
| 배당 산출 월 67시간 CPU | **$0** (호출 수만 과금, 한도 내) | 67h × $0.128 = **$8.58/월** |

> **결론 ④: 배당 산출의 Supabase 요금은 $0이다.** N값을 낮출 **금전적 동기가 없다.**
> N값을 제약하는 것은 ⓐ **킥오프 T−30분 시간창**, ⓑ **CPU 2초/호출 한도로 인한 호출 분할 복잡도**, ⓒ **워커 이전 시 발생할 비용($8.58/월)** 뿐이다.

**N값 민감도 — 정확도 손실 정량화**

확률 추정의 표준오차 `SE = √(p(1−p)/N)`, 배당 `d = (1/p)×(1/1.06)`, 배당 상대오차 ≈ `SE/p`

| N | SE (p=0.45) | 배당 상대오차 | 오버라운드 6% 대비 | 경기당 CPU | 라운드 30경기 | 판정 |
|---|---|---|---|---|---|---|
| 300 | 2.87 %p | ±6.4% | **107% — 마진 완전 잠식** | 1.0초 | 30초 | 🔴 불가 |
| 500 | 2.23 %p | ±4.9% | 83% | 1.7초 | 50초 | 🔴 불가 |
| 1,000 | 1.57 %p | ±3.5% | 58% | 3.3초 | 100초 | ⚠️ 위험 |
| **2,000** | **1.11 %p** | **±2.5%** | **41%** | 6.7초 | 200초 | ✅ **하한선** |
| **3,000 (기본)** | **0.91 %p** | **±2.0%** | **34%** | 10.0초 | 300초 | ✅ **권고 유지** |
| 5,000 | 0.70 %p | ±1.6% | 26% | 16.7초 | 500초 | ⚠️ NFR-PF-004 위반 |
| 10,000 | 0.50 %p | ±1.1% | 18% | 33.3초 | 1,000초 | 🔴 시간창 초과 |

> **판정 기준(가정)**: 배당의 몬테카를로 오차가 **오버라운드 마진(6%)의 50%를 넘으면** 북메이커 마진이 통계 노이즈에 잠식되어 장기 기대수익이 음(−)으로 뒤집힐 수 있다. → **N ≥ 2,000 필수**.
>
> **비용 절감액**: N을 3,000 → 1,000으로 낮춰도 **월 절감 $0** (Supabase), 워커 분리 시에도 $8.58 → $2.86, **월 $5.7 절감**. 그 대가로 배당 정확도가 34% → 58%(마진 잠식률)로 악화. **명백히 손해다.**

**시즌 마켓(MC_N_SEASON = 300)에 대한 별도 판정 — 개선 권고**

```
시즌 우승 p ≈ 0.20 → SE = √(0.2×0.8/300) = 2.31 %p
배당 상대오차 = 2.31/20 = ±11.5%  🔴 마진 6%의 192% — 사용 불가 수준
```

> **N=300은 통계적으로 부족하다.** 그러나 라운드당 90초 예산을 늘릴 수는 없다.
> **해법: 빈도를 낮추고 N을 올린다.**
>
> | 방식 | 재산출 주기 | N | 월 CPU | SE (p=0.2) | 배당 오차 |
> |---|---|---|---|---|---|
> | 현행 | 매 라운드 | 300 | 117,000초 | 2.31 %p | ±11.5% 🔴 |
> | **권고** | **5라운드마다** | **1,500** | **117,000초 (동일)** | **1.03 %p** | **±5.2%** ✅ |
>
> **월 CPU 총량이 동일하므로 비용 증가는 $0이면서 정확도가 2.2배 개선된다.** 시즌 마켓은 라운드 1회로 확률이 크게 변하지 않으므로 5라운드 간격의 신선도 손실은 무시할 수준이다. → 4절 권고 반영.

### 5.3 수익 모델별 BEP 역산

#### 시나리오 A — 광고 (디스플레이 배너)

```
가정: 웹 디스플레이 광고 RPM = $2.0 (스포츠·게임 카테고리 일반적 범위 $1~3의 중앙값)
필요 월 PV = 월 비용 ÷ $2.0 × 1,000
사용자당 월 PV = 10 PV/세션 × 15 세션/월 = 150 PV (가정: 관전형 서비스는 세션 빈도가 높다)
```

| 상쇄 대상 | 필요 월 PV | 필요 MAU | 필요 DAU (MAU의 20% 가정) |
|---|---|---|---|
| 1차 $51 | 25,500 | **170명** | **34명** |

## 7. 예산 리스크

### R-B1 — 폴링 대역폭의 비선형 비용 증가 (영향: 높음 / 가능성: 높음)

```
월 비용 ∝ 동시 접속 수 ÷ 폴링 주기
```

- 동시 접속이 125명 → 500명(4배)이 되고 폴링이 15초로 유지되면 대역폭 비용은 $23.2 → **$192.5 (8.3배)**. Egress·Edge Request가 **동시에 한도를 넘으면서 이중으로 튀기 때문**이다.
- 사용자 증가가 곧 비용 증가인 **유일한 항목**이며, 사용자 증가는 (수익 모델이 없으므로) **비용만 늘리고 매출을 늘리지 않는다.**
- **대응**: ① 폴링 주기를 공통코드로 두어 **운영 중 즉시 상향 가능**하게 할 것(P-7, NFR-CFG-001). ② Vercel/Supabase 사용량 알림을 **한도의 70% 지점**에 설정. ③ 동시 접속 500명 돌파 시 Realtime 전환을 착수 트리거로 등록(L3).

### R-B2 — 크론 상시 실행의 고정비 성격 (영향: 중간 / 가능성: 확실)

- 이 제품은 **사용자가 0명이어도 세계가 계속 돌아간다.** 월 43,834회 크론 + 월 10시즌 = 월 12,440경기가 아무도 안 봐도 시뮬레이션되고 저장된다.
- 결과: **DAU와 무관하게 월 $26~$51의 하한선이 존재**하고, DB는 **아무도 안 써도 월 0.6 GB씩 자란다.**
- 서비스를 "잠시 중단"할 수 없다는 점이 특히 위험하다. 크론을 멈추면 세계가 정지하고, 재개 시 catch-up(NFR-CR-004)이 밀린 라운드를 전부 처리해야 한다. **30일 중단 = 10시즌 = 12,440경기 catch-up** → 1회 처리 상한 30경기 기준 415틱 = 7시간. 처리 자체는 가능하지만 그 사이 세계 시각과 실시각이 어긋난다.
- **대응**: ① 프로젝트 중단 시나리오를 **"크론 중지 + DB 스냅샷 보존 + Free 강등"** 으로 사전 정의(월 $0로 동결 가능하나 DB 500MB 제약으로 데이터 export 필요). ② 고정비 $26~$51을 **연간 예산으로 사전 확보**하고 월 단위 의사결정에서 제외.

### R-B3 — 컴퓨트 승급의 계단식 비용 점프 (영향: 중간 / 가능성: 높음)

- 디스크는 $0.125/GB로 **연속적**이지만 컴퓨트는 **Micro $10 → Small $15 → Medium $60 → Large $110**으로 **계단식**이다. Small → Medium 한 칸이 **월 +$45 (연 $540)** 로, 다른 전 항목 합계를 넘는다.
- 승급 트리거는 용량이 아니라 **RAM**이다. `match_event` 인덱스가 shared_buffers에 안 들어가는 순간 p95가 급락하고, 이를 튜닝으로 해결하지 못하면 승급 외 선택지가 없다.
- **대응**: ① **아카이빙(L2)을 "언젠가 할 최적화"가 아니라 Task 042의 필수 항목으로 승격.** ② `season_id` RANGE 파티셔닝을 **Task 032(최초 마이그레이션)에서 선반영**(P-10) — 나중에 하면 재마이그레이션 비용 발생. ③ NFR-PF-008/011 지표를 상시 모니터링해 승급 시점을 **성능 저하 이전에** 포착.
