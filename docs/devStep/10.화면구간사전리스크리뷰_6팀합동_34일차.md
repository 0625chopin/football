### 2-1. [상] 킥오프 월드분 앵커가 타입·DB 양쪽에 없다 → **I-174 등재**

H-24 계약대로 경과분을 계산하려면 **킥오프 순간의 월드분**을 영속해야 합니다.

| 확인처 | 결과 |
|---|---|
| `src/types/match.ts:46-84` `Fixture` | 킥오프 관련 필드는 `kickoffAt` 하나. 월드분 앵커 **없음** |
| 원격 `fixture` 테이블(23컬럼 전수 조회) | `kickoff_at timestamptz` 하나. 월드분 대응 컬럼 **없음** |

**→ 월드는 월드분을 알고 킥오프는 벽시계만 아는 비대칭.** 두 좌표계를 같은 축에서 비교할 수 없습니다.

`world_minutes_at_speed_change + EXTRACT(EPOCH FROM (now()-speed_changed_at))/60*speed_multiplier`,
`is_paused`면 고정. 서버측 설계는 이미 "월드분 좌표계" 방향으로 한 발 나가 있습니다.
2팀은 이 함수와 `worldclock.ts`의 중복/충돌 여부를 검토해야 합니다.

**소유**: 1팀(계약) + 2팀(앵커 공급 주체 확정) + 3팀(Mock) + 6팀(마이그레이션). **기한 36일차.**

## 3. 34일차 산출물의 실제 버그 — 실행으로 확정

`src/components/composite/MatchCard.tsx:57-58`이 H-24 앵커 규약을 위반합니다.

```ts
const kickoffWorldMinutes = worldMinutesAt(clock, kickoffAt)   // ← clock은 "현재" 스냅샷
return Math.round(matchElapsedMinutesAt(kickoffWorldMinutes, clock, now))
```

계약(`worldclock.ts` 헤더): "`kickoffWorldMinutes`는 **킥오프 순간의 `WorldClockSnapshot`**으로
`worldMinutesAt`을 한 번 호출해 **미리 캡처해 둔 값**" — 이유까지 명시("경기 진행 중에는 배속이 바뀔 수
있어 킥오프 시점의 앵커와 현재 시점의 앵커가 달라질 수 있으므로").

**1팀이 실제 `worldclock.ts`를 import해 실행한 결과**(저장소 파일 무수정, 임시 리졸버는 `/tmp`):

```
--- A: 현행 테스트 3케이스 ---
A1: 45 (기대 45)   A2: 40 (기대 40)   A3: 3 (기대 3)     ← 3/3 통과
--- B: 킥오프 후 배속 변경 (킥오프 T0+10, 배속 1→4 at T0+20, now T0+30) ---
B1 현행식: THROW  (종료 시각이 시작 시각보다 앞설 수 없습니다)
B2 계약대로: 50   (= 10실분×1 + 10실분×4)
--- C: 정지 중 (킥오프 T0+10, T0+30 정지, now T0+40) ---
C1 현행식: 0      C2 계약대로: 20
```

| 갈래 | 증상 | 원인 |
|---|---|---|
| ⓐ | **throw → 서버 컴포넌트 렌더 사망** | `clock.speedChangedAt > kickoffAt` → `worldclock.ts:88 realMinutesBetween`이 `toMs < fromMs`에서 던짐 |
| ⓑ | **정지 중 항상 `0'`** | `worldclock.ts:101-103` — `isPaused`면 `now`와 무관하게 동결값 반환, 두 호출이 같은 값 |
| ⓒ | 배속 변경 시 오값 | 현재 배속을 킥오프까지 소급 적용 |

**테스트가 왜 통과했는가**: `MatchCard.test.ts:18-28`의 `baseClock()`이 `speedChangedAt: T0` 고정이고
3케이스 모두 킥오프 이후 전이가 없습니다. 배속 케이스도 `speedMultiplier: 4`로 **처음부터 4배속**이지
중간 변경이 아닙니다. **이 조건에서는 잘못된 식과 올바른 식이 수학적으로 동일합니다**(두 앵커가 같아
소거됨). 정지 케이스는 아예 없습니다. → 34일차 게이트(tsc 0 / lint 0 / test 1288 pass)가 초록이었던 이유.

**현재 차단성 없음** — `src/app/[lang]/page.tsx:113-115`가 `elapsedMinutes: null`로 고정(34일차 5팀 판단).
**단 I-169가 풀리는 순간 즉시 노출됩니다.**

**1팀 판정**: 이것은 "렌더 테스트가 없어서"가 아니라 **어댑터 함수조차 계약의 예외 경로를 검증하지
않아서**입니다. 후자는 브라우저와 무관하며 **일반 `.test.ts`로 지금 당장 잡을 수 있었던 것**입니다.

**근본 해소**: 4-1의 `getMatchClockContext` 신설(서버가 앵커를 캡처해 주면 UI가 재계산할 이유가 소멸).

### 4-1. `getMatchClockContext` 신설 — 1팀 판정 완료, 배정만 필요

```ts
getMatchClockContext(fixtureIds: readonly FixtureId[]): Promise<{
  readonly now: Timestamp;
  readonly clock: WorldClockSnapshot;              // getWorldStatus()와 동일 출처, 원자적 동시 반환
```

- **`now`만 신설하는 5팀 원안은 기각**(2팀 반박 채택). `now` 단독은 "안 보이던 버그를 매 폴링마다
  실행되게 만드는" 조치입니다 — 3절 재현 B가 그대로 터집니다.

  재컴파일(R-H)이 필요합니다. 반면 **`src/lib/data/*.ts`는 동결 대상이 아닙니다** → 타입 배치를 기다리지

- `now`와 `clock`을 **한 번의 호출로 원자적으로** 반환하는 것이 핵심 — 따로 조회하면 두 호출 사이에
