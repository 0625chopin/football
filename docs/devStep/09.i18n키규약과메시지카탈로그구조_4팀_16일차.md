## 2. 키 네이밍 규약

- **경로 형태**: `<namespace>.<section>.<element>` — TypeScript 중첩 객체 접근으로 구현한다(`t.common.nav.matches`처럼). 별도 i18n 라이브러리가 없으므로(현재 런타임 의존성은 `next`/`react`/`react-dom` 3개뿐, CLAUDE.md "아직 도입되지 않은 것") `t('a.b.c')` 같은 점(dot) 문자열 파싱 방식 대신 **객체 프로퍼티 체인**을 쓴다 — 오타가 나면 TS 컴파일이 바로 잡아준다.
- **케이싱**: 모든 세그먼트는 camelCase. 네임스페이스 파일명만 소문자 단수형(`common`, `league`, `match` …)으로 고정한다.
- **`namespace`**: 아래 §4의 8종 중 하나. 화면이 어느 라우트에 속하는지가 아니라 **메시지가 어느 도메인 개념에 속하는지**로 정한다(예: `playoffs`/`cup`/`transfers`/`awards`/`archive`/`sponsors` 라우트는 4팀 소유 화면이지만, 내비게이션 라벨처럼 화면 전역에서 쓰는 chrome 문구는 route별 네임스페이스가 아니라 `common`에 둔다).
- **`section`**: 화면 내 구획(`list`/`detail`/`header`/`nav`/`footer`/`empty`/`error`/`filters` 등). 여러 네임스페이스에서 반복 재사용되는 구획명은 의도적으로 통일했다(`list.title`/`empty.message`/`error.loadFailed` 패턴 — §4 표 참고). 카탈로그 감사 스크립트(§4 "48~59일차 대기 구간" 예정 작업)가 패턴 매칭하기 쉽도록 하기 위함이다.
- **leaf 키**: 명사형 라벨은 `title`/`label`/`message`, 동작 실패·에러는 `loadFailed`/`accessDenied`처럼 `동사원형+실패명사` 조합, 버튼류는 동사(`save`/`cancel`/`confirm`).
- **인터폴레이션(예약, 미구현)**: 문자열 내 변수는 `{paramName}` 중괄호 표기로 남겨둔다(예: `common.header.seasonPhaseLabel = "시즌 페이즈: {phase} (준비 중)"`). 실제 치환 로직(포맷터)은 오늘 범위가 아니다 — Task 011 후속 증분에서 구현하며, 그때도 키 문자열 자체는 바꾸지 않고 포맷 함수만 추가하면 되도록 지금부터 이 표기를 통일한다.
- **복수형(예약, 미구현)**: 필요해지면 ICU MessageFormat 관례를 따라 `_one`/`_other` 접미사 키 쌍을 추가한다(예: `match.count_one` / `match.count_other`). 현재 8개 네임스페이스에는 복수형이 필요한 키가 없어 실제 사용 예는 없다 — 규약만 예약해 둔다.

## 3. 카탈로그 구조 — ko-canonical / en-satisfies

```
src/i18n/
  locales.ts              (기존, 15일차) — SUPPORTED_LOCALES/DEFAULT_LOCALE
  messages/
    ko/
      common.ts  league.ts  match.ts  player.ts  team.ts  stat.ts  admin.ts  error.ts
    en/
      common.ts  league.ts  match.ts  player.ts  team.ts  stat.ts  admin.ts  error.ts
    index.ts               — locale → 8네임스페이스 집계 카탈로그
```

각 네임스페이스는 **`ko` 파일이 값과 타입의 기준(canonical)**이고, `en` 파일은 그 타입을 만족(satisfy)해야 한다.

```ts
// src/i18n/messages/ko/league.ts
export const league = {
  list: { title: "리그 목록" },
  // ...
};
export type LeagueMessages = typeof league;

// src/i18n/messages/en/league.ts
import type { LeagueMessages } from "../ko/league";
export const league: LeagueMessages = {
  list: { title: "Leagues" },
  // ...
};
```

`ko` 객체 리터럴에는 `as const`를 쓰지 않는다 — `as const`를 쓰면 leaf 값이 리터럴 타입(`"리그 목록"`)으로 굳어 `en` 쪽 다른 문자열 값이 타입 에러가 난다(오늘 첫 구현에서 직접 겪은 문제). `as const` 없이 선언하면 TS가 leaf를 `string`으로 widen하므로, `en`이 **같은 키 구조를 가지되 다른 문자열 값**을 쓰는 정상적인 경우를 그대로 허용하면서도, 키가 빠지거나(부족 프로퍼티) 오타로 늘어나면(초과 프로퍼티, excess-property-check) 컴파일 타임에 막힌다. 이 방식이 "48~59일차 대기 구간"에 예정된 "번역 키 누락·고아 키 감사 스크립트"보다 먼저, 개발 시점에 즉시 드리프트를 잡아준다.

`index.ts`는 8개 네임스페이스를 `{ ko: {...}, en: {...} }` 형태로 모아 barrel export한다. 소비자(22일차 `LocaleSwitcher.tsx`, 이후 각 화면)는 `messages[lang].common.nav.matches`처럼 접근한다.
