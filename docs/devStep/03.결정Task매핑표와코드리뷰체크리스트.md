## 2. 코드 리뷰 체크리스트

> 적용 범위: 전 팀 전 PR. 1팀 상시 리뷰 게이트(24~68일차)의 판정 근거이며, `docs/TEAM.md` 6.3절 **9개 조건**을 대체하지 않고 **보완**한다.
> C-1~C-4는 **D-16 절차적 생성 원칙**으로 이번 일차에 신설한 항목이다.
>
> **적용 대상 구분** (2일차 교차 점검 반영): **코드 PR = 전 항목**, **문서 PR(와이어프레임·설계 문서 등) = C-1 · C-9 · C-11 · C-15~C-19 만 판정**한다. 나머지는 코드 전제 항목이므로 문서 PR에서 "해당 없음"으로 처리하고 미판정을 결함으로 세지 않는다.

### 2.1 절차적 생성 (D-16 · D-17) — **신설**

- [ ] **C-1. 외부 축구 데이터 도입 0건** — FM 데이터 파일 임포트, 스크래핑, 외부 축구 데이터 API 연동, 실명 선수/실존 클럽/리그 명칭/엠블럼/초상 자산이 **PR에 포함되지 않았다.** 위반 시 **머지 거부**(예외 없음).
- [ ] **C-2. 결정론 (적용 범위: `src/lib/sim/**` 및 생성 경로 한정)** — 해당 경로의 모든 무작위성이 **시드를 명시 인자로** 받는다. `Math.random()`·`Date.now()`·`new Date()` 미사용(16일차 ESLint 룰도 `src/lib/sim/` 한정으로 구성). **단서: UI 계층의 현재 시각 취득은 정상 동작이다** — 카운트다운, 폴링 tick, 로컬 시각 변환(5팀 규약 R-9)은 이 항목의 위반이 아니며 리뷰에서 지적하지 않는다.
- [ ] **C-3. 생성기 단일 소유** — 이름·엠블럼 생성은 **공유 생성기(Task 007)** 를 호출한다. Mock 전용/엔진 전용으로 **중복 구현하지 않는다**. **실명 블랙리스트 필터**를 우회하는 생성 경로가 없다.
- [ ] **C-4. 능력치 체계** — 1~30 스케일을 따르고, 외부 데이터 스키마를 모사한 임포트 DTO 타입이 없다.

### 2.3 미래 정보 비노출 (NFR-SEC-004 · FR-MT-016) — **신설**

- [ ] **C-23. 미래 정보 비노출** — 경기 결과·이벤트는 **서버가 `min(now − kickoff_at, 경기 길이)` 로 산정한 범위만 응답**한다. 전량 응답 후 클라이언트에서 숨기는 구현(CSS `display:none`, 조건부 렌더, 클라이언트 필터)은 **머지 거부**. 종료 전 결과 필드는 **null로 응답**한다.
  - **검증 방법**: UI 스냅샷이 아니라 **네트워크 응답 payload를 직접 검사**한다 — 응답 JSON에 경과분 이후 이벤트가 **0건**인지 확인. (육안·Playwright 스냅샷으로는 잡히지 않는 유일한 종류의 결함이다. "전량 수신 후 CSS로 가리는" 구현은 화면상 정상으로 보이고 UI 레벨 수락 기준도 통과한다.)
  - **소관 3팀 공용**: 서버 응답 산정 = 1팀(004 어댑터 계약)·2팀(023 엔진) / 클라이언트 미필터링 = 5팀(017). 개별 팀 규약이 아니라 **공용 항목**이다.
  - **스탯도 같은 축(W-21 4제약, `docs/ISSUES.md` I-25)**: ① 서버 컷오프 강제 ② **이벤트–스탯 단일 소스·단일 응답 컷오프**(분리 시 스탯이 1틱 선행해 다음 이벤트가 선노출된다) ③ 파생 지표(**선수 평점·xG·점유율**)에도 동일 컷오프 ④ **노출된 이벤트에서 재계산한 값만 응답**(사전 집계 최종 스탯 직송 금지).
  - **왜 게이트에서 막는가**: 1팀 041 침투 테스트(75~79일차)에서 발견되면 017·023을 함께 고쳐야 한다. 2차에서는 결과를 미리 아는 사용자의 베팅으로 **정산 신뢰가 붕괴**하는 상품 무결성 문제다.
