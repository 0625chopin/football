## 0. 단일 월드 전제 — 확정 (D-15)

**이 프로젝트의 도메인 타입과 조회 계층은 월드가 1개뿐이라는 전제 위에 설계한다.** 1차 릴리스에서 이 전제는 확정 사항이며, 재검토 대상이 아니다 (가정 `AS-09` 해제됨).

| 계층 | 원칙 |
|---|---|
| **도메인 타입 (`src/types/**`)** | 하위 엔티티 타입에 **`worldId` 스코핑 필드를 도입하지 않는다.** `E-01 World`만 `WorldId`를 가진다 |
| **조회 계층 (`src/lib/data/**`)** | 현재 월드를 **1회 해석**해 사용한다. 메서드 시그니처에 월드 인자를 두지 않는다 (Task 004 계약의 전제) |
| **UI** | 월드 선택/스위처 화면·상태를 만들지 않는다 (요구사항 범위 밖) |
| **물리 스키마 (6팀 소유)** | 하위 테이블의 `world_id` **FK 컬럼은 그대로 유지**한다. 단일 레코드 보장은 DB 제약(부분 유니크 인덱스 등)으로 강제한다 |

> **확장 여지 (1줄):** 다중 월드로 확장할 때는 `world_id` FK가 이미 스키마에 있으므로 마이그레이션 없이 조회 계층에 스코핑 필터만 추가하면 된다.

**용어 정리** — 작업표의 "`world_id` 미도입"은 **도메인 타입 레벨**의 원칙이고, PRD 6.3의 "`world_id` FK 유지"는 **물리 스키마 레벨**의 원칙이다. 둘은 충돌하지 않으며, 위 표대로 계층을 나눠 적용한다.

## 1. 결정 → 설계 원칙 (P-15 ~ P-20)

### P-15 · 단일 월드 (D-15)

- **T1.** 도메인 타입에 `worldId` 스코핑 필드를 넣지 않는다. `World` 타입은 단일 레코드 전제(컬렉션 조회 API를 정의하지 않음).
- **T2.** `World`는 월드 시드(`worldSeed`)·현재 시즌·현재 페이즈·배속·정지 상태를 담는 **런타임 상태의 뿌리 타입**이다. 시드 계층 타입(7일차)은 여기서 파생한다.
- **T2-a. `World.worldSeed`는 53비트 안전 정수(`Number.MAX_SAFE_INTEGER`, `number`)로 표현한다** (2026-07-21 32비트로 최초 확정 → **2026-07-27(5일차) D-28로 53비트 완화 개정**, 구 `docs/ISSUES.md` I-32). `string`·`bigint`는 쓰지 않는다. 개정 근거: 2팀 시드 파생 규약이 상위 2비트를 네임스페이스 태그로 예약해 32비트 규약의 실효 payload가 30비트뿐이라 시즌당 이벤트 시드(~6×10⁵ 기준) 생일 충돌이 수백 건/시즌 발생했다(재현성 영향 0, 통계적 상관만 미미). 53비트로 완화하면 payload가 51비트가 되어 충돌이 소멸한다. `bigint` 도입은 ① PRNG(xoshiro128\*\*)가 32비트 워드 연산으로만 구성돼 있고 ② JSON 직렬화가 불가능해 스냅샷·API 계약이 깨지며 ③ 틱 핫패스 성능(Task 023 p95 ≤ 50ms)을 위협해 기각했다. 결정 원문: `docs/require/06-prioritization-and-risks.md` D-28. 만약 53비트로도 확정할 수 없는 사정이 뒤늦게 드러나면, **변환 함수 소유는 2팀(`src/lib/sim/rng/`)에 단일 위임**한다(1팀·타 팀은 변환을 재구현하지 않는다).
- **T2-b. 7일차 시드 타입은 브랜드만 씌우고 파생 로직을 재구현하지 않는다.** 시드 파생 규칙의 단일 소유는 2팀 `src/lib/sim/rng/derive.ts`다. 파생 규칙이 두 곳에 있으면 스냅샷 회귀(R-03) 시 원인 추적이 불가능하다.
- **T2-c. `PrngState = readonly [number, number, number, number]`(2팀 소유)를 그대로 채택한다.** 이 타입은 알고리즘 구현 타입이므로 `src/types/`로 승격하지 않는다. 이견은 **8일차 H-01 동결 전**에 제기한다.
- **T2-d. 2일차 `derive.ts` 실행 조건**: 2팀은 `WorldSeed`/`SeasonSeed` 등 **도메인 명칭 타입을 export하지 않고** 원시 `number` 또는 모듈 로컬 타입을 사용한다. 7일차 `src/types/brand.ts` 확정 후 `import`로 교체한다(선언 순서 역전에 따른 중복 선언 방지).
- **T3.** DataSource 메서드는 월드 인자를 받지 않는다. 월드 해석은 어댑터 내부 책임이다.

### P-16 · FM 실데이터 미사용, 전량 절차적 생성 (D-16)

- **T4.** 선수·클럽의 이름·능력치·국적을 표현하는 타입은 **외부 데이터 스키마를 모사하지 않는다.** 임포트 전용 DTO 타입을 만들지 않는다.
- **T5.** 능력치는 참고 체계인 **1~30 정수 스케일**을 타입 수준에서 표현한다(범위는 브랜드/리터럴 타입 또는 런타임 검증으로 강제, 7일차 확정).
- **T6.** 생성 결과는 **결정론적**이어야 하므로 생성 함수 계약에는 시드가 명시 인자로 들어간다. 타입에 `Date`·난수 기반 필드를 암묵 기본값으로 두지 않는다 (NFR-DT 정합).
- **T7.** 리뷰 게이트: 외부 축구 데이터 API·스크래핑·FM 데이터 파일을 전제한 타입/의존성은 **차단 대상**이다.

### P-17 · 국적 기반 다국적 이름 (D-17)

- **T8.** **국적 코드는 enum성 값으로 단일 선언**한다(6일차 `src/types/enums.ts`). 문자열 리터럴을 각 파일에서 개별 정의하지 않는다.
- **T9.** 이름 풀·국가별 비중은 **공통코드 대상**이므로, 타입에 국가 목록을 하드코딩한 유니온으로 고정하지 않고 공통코드 조회 결과와 정합하는 형태로 둔다.
- **T10.** Mock 팩토리와 실제 생성기는 **동일한 이름 생성 로직**을 공유한다(`src/lib/naming/` 단일 소유). 따라서 이름 생성 함수의 입출력 타입은 `src/types/`에 두고 양쪽이 함께 참조한다.
- **T11.** 실명 블랙리스트 필터는 생성기 계약의 일부다(타입 레벨에서는 생성 옵션으로 표현).

### P-18 · 다국어(i18n) 대응 (D-18)

- **T12.** **enum → 번역 키 매핑을 "타입으로 강제하는 규약"만 정의한다**(7일차 산출물). 포지션·이벤트·수상·부상 등 표시명은 enum 값에서 번역 키를 도출하며, 표시 문자열을 도메인 타입에 담지 않는다.
  - **T12-a. 실제 번역 키 문자열·네이밍·키 공간은 4팀 단독 소유(H-09, 22일차)다** (2026-07-21 팀장 결정). 1팀은 7일차에 **키 이름을 선점하지 않는다.** 우리가 정의하는 것은 "enum 값과 번역 키가 타입 수준에서 1:1로 대응되어야 한다"는 **구조 제약**뿐이다.
  - **T12-b.** 3팀은 표시명 **값**만 기여하고, 5팀의 프리픽스 표기(`home.live.*` 등)는 **제안 지위**다. 확정은 4팀 H-09 수신 후.
- **T13.** 도메인 타입은 **표시 서식이 아니라 원시값**을 담는다. 날짜/시각은 타임스탬프, 포인트·배당은 숫자로 두고 로케일 서식은 UI 계층 책임이다.
- **T14.** **선수·클럽 이름은 번역 비대상**(고유명사)이므로 번역 키 타입에 포함하지 않는다. 국가명 표시는 번역 대상.
- **T15.** UI 문자열 하드코딩 금지는 **10일차 ESLint 가드레일(18일차 룰)** 로 강제한다.

### P-19 · 승부차기 득점 분리 (D-19)

- **T16.** 통계 타입에서 **`goals`(정규+연장)와 승부차기 기록을 별도 필드/별도 타입으로 분리**한다. 합산 가능한 형태로 나란히 두지 않는다.
- **T17.** 득점왕·시즌/통산 득점 집계 타입은 승부차기 값을 **입력으로 받을 수 없게** 설계한다(브랜드 타입 또는 별도 집계 타입으로 오용 차단).
- **T18.** 경기 결과 타입은 승패 판정용 승부차기 스코어를 **별도 필드**로 보유한다(스코어 표기와 통산 집계의 의미를 구분).

### P-20 · 감독 독립 엔티티 (D-20)

- **T19.** 감독은 팀 타입의 속성이 아니라 **독립 엔티티 타입**이며 팀과 **N:1**로 연결한다(교체 이력 추적 가능).
- **T20.** 감독은 **명성·수상 트랙**을 보유한다. 수상 타입은 선수/감독을 함께 표현할 수 있어야 한다.
- **T21.** 팀의 감독 참조는 **공석(null) 허용**이며, 공석 시 전술 폴백은 `BALANCED`다.
- **T22.** 이적은 프리시즌 경질/선임 수준까지만 — **협상·계약·이적료 타입을 감독에 대해 만들지 않는다.** 감독 전용 화면 타입도 1차 범위 밖.
