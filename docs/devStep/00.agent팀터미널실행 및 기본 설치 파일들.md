# Agent 팀 실행 방식 — 두 가지를 구분할 것

`docs/TEAM.md`의 팀원 10명을 실제로 굴리는 방법은 두 가지이며, **서로 다른 것**입니다.

| | **A. 서브에이전트 방식** | **B. tmux 패널 방식** |
|---|---|---|
| 형태 | 메인 세션 내부의 백그라운드 에이전트 | 패널마다 독립 `claude` 프로세스 |
| 화면 | 안 보임. 완료 시 메인 세션에 결과 통보 | 각 패널에서 실시간으로 보임 |
| 컨텍스트 | 메인 세션이 지시문에 담아 전달 | **빈 세션** — 각자 문서를 처음부터 읽어야 함 |
| 조율 | 메인 세션이 결과를 받아 취합 | **사용자가 직접** 패널을 오가며 조율 |
| 적합한 일 | 조사·분석처럼 **산출물이 문서 하나**로 끝나는 일 | 구현처럼 **오래 걸리고 중간 개입이 필요한** 일 |
| 파일 충돌 | 메인 세션이 순서를 통제 | **파일 소유권 경계로만 막힘** — 규칙 준수 필수 |

**현재 프로젝트의 운용 방침 (2026-07-20 결정)**
- 사업·기획 3인(팀원 1~3)의 조사는 **A 방식으로 이미 수행**했다.
- **개발 팀원(4~10)의 구현부터는 B 방식(tmux 패널)을 사용한다.**

## B 방식 — tmux 팀 실행 절차

### 1. 사전 조건

- `docs/TEAM.md`의 **3.2 파일 소유권 경계**를 팀원 전원이 지킨다. 이게 유일한 충돌 방지 장치다.
- `ROADMAP.md`의 **4.1 직렬 게이트(S-1~S-8)**를 위반하는 Task를 동시에 열지 않는다.
- 한 번에 띄우는 패널은 **그 Phase에서 실제로 병렬 가능한 팀원만** (`docs/TEAM.md` 4.2 병렬 가능 구간 참조). 전원을 항상 띄우지 않는다.

### 2. 패널 생성

프로젝트 루트에서 실행한다. 아래는 **Phase 1 초반(P-1) 4인 병렬** 예시다.

```bash
cd /mnt/e/claudeStudy/workspaces/football4

# 팀 작업용 창을 새로 만들고 4분할
tmux new-window -n phase1 -c "$PWD"
tmux split-window -t phase1 -c "$PWD"
tmux split-window -t phase1 -c "$PWD"
tmux split-window -t phase1 -c "$PWD"
tmux select-layout -t phase1 tiled
```

패널 이동은 `Ctrl-b` + 방향키, 확대/축소는 `Ctrl-b z`, 창 전환은 `Ctrl-b n`/`Ctrl-b p`.

### 3. 각 패널에 팀원 투입

패널마다 `claude`를 실행하되, **첫 프롬프트로 역할 브리핑을 준다.** 빈 세션이므로 이걸 빼먹으면 아무것도 모른다.

```bash
tmux send-keys -t phase1.0 'claude "docs/TEAM.md와 ROADMAP.md를 읽고 팀원 4(아키텍트/타입 설계) 역할을 맡는다. Task 002(도메인 TypeScript 타입 47종 정의)를 수행하라. 소유 디렉터리는 src/types/ 이며 타 팀원 소유 파일은 수정하지 않는다."' C-m

tmux send-keys -t phase1.1 'claude "docs/TEAM.md와 ROADMAP.md를 읽고 팀원 10(QA/테스트) 역할을 맡는다. Task 008(Vitest 도입 및 검증 스위트 골격)을 수행하라. 소유 범위는 src/**/*.test.ts, vitest.config.ts, CI 설정이다."' C-m

tmux send-keys -t phase1.2 'claude "docs/TEAM.md와 ROADMAP.md를 읽고 팀원 9(DB/Supabase) 역할을 맡는다. Task 009(Supabase 물리 스키마 설계, 구현 제외)를 수행하라. 소유 디렉터리는 docs/db/, supabase/, src/lib/data/supabase/ 이다."' C-m

tmux send-keys -t phase1.3 'claude "docs/TEAM.md와 ROADMAP.md를 읽고 팀원 5(시뮬레이션 엔진) 역할을 맡는다. Task 006(시드 계층 PRNG와 결정론 유틸리티)을 수행하라. 소유 디렉터리는 src/lib/sim/ 이며 React·Supabase import는 금지된다."' C-m
```

> 프롬프트에 **① 읽을 문서 ② 팀원 번호·역할 ③ 담당 Task ④ 소유 디렉터리 경계** 네 가지를 반드시 넣는다.
> ④를 빼면 다른 팀원 파일을 건드려 충돌이 난다.

### 4. 브리핑 템플릿

```
docs/TEAM.md와 ROADMAP.md를 읽고 팀원 {N}({역할}) 역할을 맡는다.
Task {XXX}({제목})를 수행하라.
소유 디렉터리는 {경로}이며, 타 팀원 소유 파일은 수정하지 않는다.
근거 요구사항은 docs/require/ 를 참조한다.
착수 시 /tasks/{XXX}-{slug}.md 를 생성하고, 완료 시 체크박스를 갱신한다.
```

### 5. 작업 중 규칙

- 브랜치는 `task/XXX-<slug>`. **동일 파일을 건드리는 두 Task를 동시에 열지 않는다.**
- 타 팀원 소유 파일 수정이 필요하면 **직접 고치지 말고** 해당 패널의 팀원에게 요청한다(사용자가 중계).
- Next.js 16 API를 쓰는 Task는 `node_modules/next/dist/docs/` 선확인이 필수다 (AGENTS.md).
- 발견한 미결·개선사항은 `docs/ISSUES.md`에 기록한다.

### 6. 완료·취합

1. 각 팀원이 `/tasks/XXX-*.md` 체크박스를 갱신하고 산출물 경로를 기재한다.
2. 팀원 10(QA)이 `docs/TEAM.md` 6.3의 **리뷰 게이트 9개 조건**으로 검수한다.
3. 통과하면 `ROADMAP.md`의 해당 Task에 `✅ - 완료` + `See: /tasks/XXX-xxx.md`를 반영한다(팀원 4가 반영).
4. 패널을 닫는다: 해당 패널에서 `exit`, 또는 `tmux kill-window -t phase1`.

### 7. 주의점

- **패널마다 별도 세션이라 컨텍스트가 공유되지 않는다.** 팀원 A가 내린 결정을 팀원 B는 모른다. 결정은 반드시 `docs/ISSUES.md`나 PRD 결정 기록 같은 **파일로 남겨야** 전파된다.
- 패널을 많이 띄울수록 토큰 사용량이 배로 늘어난다. 병렬 가능 구간에서만 쓴다.
- 조사·분석처럼 산출물이 문서 하나로 끝나는 일은 **A 방식이 더 싸고 빠르다.** 무조건 tmux를 쓰지 않는다.
