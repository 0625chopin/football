## B. `EnumTranslationCatalog<T>` (T12) 판정 — **동의**

`src/types/config.ts:170-172` 정의 확인:

```ts
export type EnumTranslationCatalog<TMember extends string> = {
  readonly [Member in TMember]: string;
};
```

및 주변 "번역 대상 경계" 절(`config.ts:147-186`).

### 판정 근거

1. **키 문자열 선점 여부** — 이 타입은 `[Member in TMember]: string` 매핑 제약만 걸며, `string` 값의 **내용**(실제 키 네이밍 규칙이든 로케일 원문이든)에는 어떤 제약도 없다. 코드 주석(T12-a, `config.ts:14-17`)도 "실제 번역 키 문자열·네이밍·키 공간은 4팀 H-09(22일차) 단독 소유"라고 명시하고, 예시(`config.ts:163-167`)도 키 문자열을 **예시일 뿐**이라 못 박았다. **I-14 합의(구조 제약만, 키 선점 없음)를 준수한다.**
2. **22일차 H-09 키 규약과의 충돌 여부** — 없다. `EnumTranslationCatalog<T>`의 값 타입은 `string`이라 4팀이 22일차에 확정할 `<namespace>.<component|screen>.<element>` 키 형식이든, 로케일별 원문 직접 값이든 **둘 다 그대로 대입 가능**하다. 팀 md 19일차 산출물이 `src/i18n/messages/{ko,en}/enums.ts`(로케일별 파일 분리) 형태로 계획돼 있는데, 이 구조에서는 카탈로그가 "enum → 로케일 원문"으로 직접 매핑될 것으로 예상되며, 그 경우에도 이 타입은 문제없이 적용된다(예: `positionLabels: EnumTranslationCatalog<Position> = { GK: '골키퍼', ... }` 형태). 반대로 "enum → 키" 간접 방식을 택해도 마찬가지로 적용된다 — **어느 쪽 설계를 택해도 타입이 강제하는 바가 없다.**
3. **enum 멤버 전수 매핑 강제의 실작동 가능성** — `enums.ts`의 대상 유니온들(`Position`, `InjurySeverity`, `TacticalStyle`, `AwardType` 등)이 전부 문자열 리터럴 유니온으로 선언돼 있음을 확인했다(`string`으로 widen되지 않음). TypeScript의 매핑 타입은 이런 리터럴 유니온에서 **멤버 추가/삭제 시 대입 객체의 초과/누락을 `tsc` 컴파일 오류로 즉시 잡는다** — 팀 md 22일차 수락 기준("번역 키 누락이 tsc 오류로 검출")과 정확히 맞물린다. 런타임 카탈로그 로딩·네임스페이스 분리(도메인별 파일 분할, 19일차 계획)·로케일 분리(ko/en 파일별 각각 적용) 모두 이 타입과 무관하게 4팀이 자유롭게 설계 가능하다.
4. **복수형·성별 변형** — 이 카탈로그는 "enum 멤버당 고정 표시명 1개"를 표현하는 용도이고, ko는 문법적 성별이 없고 이 도메인의 enum 표시명(포지션·부상등급·전술성향 등)은 개수에 따른 복수형 처리 대상이 아니다(문장 템플릿의 복수형은 별도 `t()`/포맷터 함수의 역할, 11일차 확정 예정). 짧은 표기/긴 표기(예: FormStrip의 W/D/L 단일 문자 vs 접근성용 풀네임)가 둘 다 필요하면 `EnumTranslationCatalog<T>`를 **같은 유니온에 두 번**(`formResultShort`, `formResultFull`) 적용하면 되므로 표현력 문제도 없다.

### 결론

**동의.** 구조 제약(멤버 전수 매핑 강제)만 걸고 키 문자열은 선점하지 않아 I-14 합의를 지켰고, 22일차 H-09에서 4팀이 정할 키 규약·카탈로그 파일 구조(로케일별 분리든 키 간접 참조든) 어느 쪽과도 충돌하지 않는다. 수정 요청 없음.
