## 1. 시드 — `bigint` vs 53비트 안전 정수 (I-29 최종 판정)

- **① 충돌 내용**: `docs/ISSUES.md` I-29는 "물리 컬럼(`bigint`)이 도메인 타입 정수 범위를 보장하는가"를 미해결로 남겨두었다. D-28(3일차)이 도메인 타입을 53비트로 완화하면서 표기 불일치 자체는 해소됐다고 문서화되어 있으나("12일차 H-08에서 확인"), **`bigint`가 53비트 범위를 자동으로 보장하지는 않는다** — Postgres `bigint`(int8)의 실제 허용 범위는 ±(2⁶³−1)(약 9.22×10¹⁸)로 53비트 안전 정수 상한(2⁵³−1 ≈ 9.007×10¹⁵)보다 훨씬 넓다.
- **② 타입 쪽**: `src/types/brand.ts` `Seed`/`WorldSeed`/`SeasonSeed`/`MatchSeed` = `Brand<number, ...>`, 런타임 상한 `Number.MAX_SAFE_INTEGER`(D-28). 타입 시스템은 이 상한을 컴파일 타임에 강제하지 못한다(brand.ts 자체 주석: "정수 보장은 생성 지점의 런타임 책임").
- **③ 물리 쪽**: DC-08 "포인트는 bigint"(시드는 명시 안 됐으나 05:E-01/E-03 원문이 `bigint`로 명시, I-29 원문 인용). `bigint` 컬럼 자체엔 53비트 상한 제약이 없다.
- **④ 판정**: **타입 변경 불필요.** 대신 **9~11일차 스키마 설계(Task 009)에서 시드 계열 4개 컬럼(`world_seed`/`season_seed`/`match_seed`/(범용 `Seed`를 쓰는 `Team.crest_seed` 등)에 `CHECK (col >= 0 AND col <= 9007199254740991)` 제약을 추가**해야 왕복 손실이 구조적으로 차단된다. 이 CHECK가 없으면 (a) 버그로 53비트를 넘는 값이 저장될 경우 (b) Supabase JS 클라이언트/PostgREST가 `int8`을 JSON 숫자로 직렬화하는 경로에서 배정밀도 부동소수 손실이 발생할 수 있다 — **엔진 자체는 절대 53비트를 넘는 값을 생성하지 않지만(derive.ts payload 51비트), DB 레이어가 이를 스키마로 강제하지 않으면 방어선이 하나 비는 상태**다.
- **부가 확인 필요(17일차 매퍼 작성 시)**: `mcp__supabase__generate_typescript_types`가 `bigint`/`int8` 컬럼을 TS `number`로 생성하는지 `string`으로 생성하는지 오늘은 실제 테이블이 없어 확인 불가(§4 베이스라인 참조 — DB에 테이블 0개). 32일차 첫 마이그레이션 직후 재확인 권장. `string`으로 생성된다면 매퍼에서 `Number(row.world_seed)` 후 `Number.isSafeInteger` 검증을 거쳐 브랜드 캐스트해야 한다.

**I-29 오늘 판정 (요청받은 항목)**: **가능** — "물리 컬럼이 53비트를 보장하는가"의 답은 "**보장하지 않음, CHECK 제약 추가로 보장하도록 설계해야 함**"이다. 이는 타입 변경이 아니라 9~11일차 스키마 설계 시 필수 반영 사항이므로 이슈 등록 대신 이 문서 + 팀장 보고로 직접 인계한다.

## 2. 포인트(경제) — `bigint`와 `economy.ts`/`stat.ts`

- **① 충돌 내용**: 없음. `Points`(brand.ts)는 정수 고정(DC-08), 물리 `bigint`와 표현 폭·정밀도 완전 일치.
- **② 타입 쪽**: `Points = Brand<number, 'Points'>`. `Contract.wagePerSeason`/`Transfer.fee`/`Team.balance`/`PointTransaction.amount`/`balanceAfter`/`TeamSeasonStat`의 재정 블록 8개 필드 전부 `Points`.
- **③ 물리 쪽**: DC-08 "포인트는 정수(`bigint`)".
- **④ 판정**: 불필요. **회계 항등식(NFR-QA-005) 위험 재확인**: `Points`가 정수로만 연산되는 한(가감산은 JS `number`가 2⁵³−1 이하에서 정확), 부동소수 누적 오차는 발생하지 않는다 — "부동소수 회계 오차" 우려는 **곱셈(스폰서 분배율 `sharePct` 적용 등)에서 소수 결과가 나올 때만** 실체화되며, 그 결과값도 최종적으로 `Points`(정수)로 반올림돼 저장되는 구조이므로 원장 자체는 항상 정수다. 단, **§6(퍼센트 필드) 정밀도가 스폰서 분배 계산의 입력**이므로 그쪽 정밀도 규약이 간접적으로 회계 정확도에 영향을 준다 — §6 참조.
- 시드와 동일하게 53비트 상한 CHECK을 포인트 컬럼에도 걸지 여부는 9~11일차 판단 사항으로 넘긴다(포인트는 게임 재화라 실질적으로 2⁵³ 근처에 도달할 우려는 낮지만, 방어적으로 동일 CHECK을 걸어도 무해하다는 점만 공유).

## 3. 배당·확률 — `numeric(8,2)` / `numeric(9,8)`과 `betting.ts`

- **① 충돌 내용**: 없음(확인 결과 정합).
- **② 타입 쪽**: `Odds.decimalOdds: number`(주석 "1.01 ~ 500.00"), `BetSelection.probability: number`(주석 "0~1"), `BetMarket.overround: number`(주석 "기본 1.0600" = 1.06, 6%마진 — 유효 소수는 2자리).
- **③ 물리 쪽**: DC-08 "배당은 `numeric(8,2)`, 확률은 `numeric(9,8)`".
- **④ 판정**: 불필요.
  - `decimalOdds`: 2자리 소수·최대 500.00 → `numeric(8,2)`(정수부 최대 6자리)로 무손실 저장.
  - `probability`: 엔진의 **유일한 비교 진입점**은 `src/lib/sim/rng/precision.ts`이며, `PROBABILITY_DECIMALS = 6`(백만분율)로 고정 비교한다. 물리 정밀도 `numeric(9,8)`은 소수 8자리로 엔진 비교 정밀도(6자리)보다 **더 촘촘**하므로, DB에 저장/재조회해도 엔진이 실제로 구별하는 정밀도 이하로는 정보가 손실되지 않는다 — 정합.
  - `overround` 1.0600은 수치상 1.06과 동일(2자리)하므로 `numeric(8,2)`로 저장해도 무손실이나, **배팅 엔진(5팀, 2차 릴리스)이 향후 오버라운드를 4자리 이상 정밀도로 산출하게 되면 절삭 위험이 생긴다** — 1차 범위(오늘 동결 대상)엔 영향 없음, 2차 배팅 설계 착수 시점(약 45일차 이후로 추정)에 재확인 권장 메모만 남긴다.

## 4. 평점·컨디션 — `numeric(3,1)`

- **① 충돌 내용**: 없음(원시 필드), **파생 평균 필드에 한해 반올림 발생 가능**(경미, 아래 설명).
- **② 타입 쪽**: `PlayerState.condition: number`(1.0~10.0), `PlayerMatchStat.matchRating: number`(1.0~10.0) — 둘 다 1자리 소수.
- **③ 물리 쪽**: DC-08 "컨디션은 `numeric(3,1)`"(최대 99.9, 도메인 범위 10.0 이내로 여유 충분).
- **④ 판정**: 원시 필드는 불필요. **참고 사항**: `PlayerSeasonStat.avgCondition`/`TeamSeasonStat.avgCondition`은 여러 라운드 `condition`(1자리 소수) 값의 **평균**이라 산술 결과가 1자리 소수를 넘을 수 있다(예: (7.3+7.4+7.6)/3 = 7.4333…). `numeric(3,1)`로 저장하면 반올림되지만, 이 필드는 재계산 입력이 아니라 표시·리포트 값이므로(stat.ts "합산형만 저장, 파생 비율은 계산" 원칙과 별개로 이미 저장 컬럼으로 설계됨) **허용 가능한 손실**로 판단한다. 더 높은 정밀도가 필요하면 9~11일차 스키마 설계에서 `avgCondition`만 `numeric(4,2)` 등으로 예외 지정 가능(타입 변경 불요).

## 5. 퍼센트형 필드 — DC-08 규약 밖 (신규 발견)

- **① 충돌 내용**: `SponsorContract.sharePct`(주석 "≤ 30.00(%)")와 `Loan.wageSharePct`(주석 "%기본 50")는 소수 2자리 퍼센트 값인데, **DC-08 4종 규약(포인트/배당/확률/컨디션)에 퍼센트 카테고리가 없다.** 9일차 스키마 설계 시 담당자가 임의로 타입을 고르게 되면 팀 간 정밀도 기준이 갈릴 수 있다.
- **② 타입 쪽**: 둘 다 `number`(브랜드 없음, 범용 실수).
- **③ 물리 쪽**: 미정.
- **④ 판정**: **타입 변경 불필요**(이미 범용 `number`라 물리 정밀도를 뭘로 정하든 타입 쪽 수정 없이 흡수됨). **9일차 스키마 설계 착수 시 명시적으로 결정해달라고 요청**: `sharePct`는 "≤ 30.00%" 원문 그대로 2자리 소수(`numeric(5,2)` 권장, 정수부 3자리까지 여유), `wageSharePct`는 원문에 소수 자리 언급이 없으나 스폰서 분배율과 동일 계열로 보고 동일 정밀도 권장.

## 6. Enum성 값(30여 종) 물리 표현과 왕복 보존

- **① 충돌 내용**: 없음(구조적으로), **운영 리스크 1건 + 이 문서 자체의 검증 오류 1건(아래 정정)**.
- **② 타입 쪽**: `enums.ts`에 30여 개 리터럴 유니온이 **단일 선언**되어 있음(체크리스트 C-6). `Formation`/`TasteTag`/`NationalityCode`(코드 체계만 확정, 값 목록은 공통코드)는 아직 `string`(개방형).
- **③ 물리 쪽**: 팀 md에 "체크 제약/룩업 테이블/공통코드 중 어떤 형태로 내려가는가"를 판정 요청받음 — **CHECK 제약 또는 네이티브 Postgres ENUM 둘 다 구조적으로 무손실 왕복이 가능**하다(PostgREST가 두 경우 모두 JSON 문자열로 직렬화). 공통코드(E-41~44) 테이블은 **런타임 튜닝 가능한 파라미터**(예: `ROUND_INTERVAL_MIN`) 전용이며, 이 enum성 값들(이벤트 타입·포지션·상태 등)은 요구사항에 값이 이미 고정된 **구조 상수**이므로 공통코드 대상이 **아니다** — 혼동하지 않아야 한다.
- **④ 판정**: 타입 변경 불필요. **운영 리스크**: 물리 CHECK/ENUM 목록이 `enums.ts`의 실제 값 집합과 어긋나면(수기 복사 오타 등) 왕복 자체는 되지만 매퍼가 유효하지 않은 문자열을 브랜드 캐스트하는 사고가 날 수 있다.

### ⚠ 정정 (8일차 초판 발행 이후, 팀장·3팀 교차 검증 반영)

**초판 오류**: 이 문서 초판은 `PointTransactionReasonCode`를 "11개"라고 적었다. **틀렸다 — 실제로는 12개다.**

```
LEAGUE_FINISH, PLAYOFF_PRIZE, CUP_PRIZE, GIANT_KILLING_BONUS,
SPONSOR_INCOME, SPONSOR_SHARE, TRANSFER_IN, TRANSFER_OUT,
WAGE, REBUILD_GRANT, YOUTH_COST, ACADEMY_INVEST   ← 12개, FR-EC-001 원문 "12종"과 정확히 일치
```

**원인**: `src/types/enums.ts:207-210`의 **주석**이 "요구사항 원문은 12종이라 적었으나 실제 열거값은 11개뿐(문서 표기 오류로 추정)"이라고 적혀 있었고, 이 문서 초판은 **주석을 그대로 신뢰**하고 리터럴 유니온 본문(11~223행)을 직접 세지 않았다. 3팀이 같은 날 독립적으로 리터럴을 직접 세어 12개(요구사항과 일치)임을 확인했고, 팀장이 원본 대조로 재검증해 3팀이 맞다고 확정했다. **주석 자체의 정정은 1팀 소유**(`src/types/**`)라 6팀은 이 문서만 정정한다.

**왜 위험했나**: §10(매퍼/스키마 흡수 목록)에서 "enum CHECK 목록은 `enums.ts` 실값 그대로 복사"라고만 적고 개수를 11로 잘못 기억한 채 9일차에 넘어갔다면, 32일차 매퍼가 `ACADEMY_INVEST`(또는 실제로 빠졌을 12번째 값)를 삽입할 때 CHECK 제약 위반으로 거부됐을 것이다 — 팀 md가 경고한 "여기서 잡지 못하면 32일차 매퍼에서 터집니다"가 실제로 발생할 뻔한 사례.

**규칙(9일차 이후 나머지 enum 30여 종에도 동일 적용)**: **enum CHECK 제약 목록의 단일 소스는 `enums.ts`의 리터럴 유니온 본문이며, 그 위/근처의 한글 주석이 아니다.** 개수·값 목록을 옮길 때는 항상 리터럴을 직접 세거나 코드로 추출하고, 주석의 서술(개수·범위 언급)은 참고만 하되 검증 근거로 삼지 않는다. 9일차 스키마 설계 착수 시 6팀 내부적으로 이 규칙을 재확인한다.

## 9. 타임스탬프 · jsonb 필드 — 부가 확인

- `Timestamp = string`(ISO-8601, `brand.ts`), DC-07 "모든 시각 UTC `timestamptz`로 저장, 클라이언트에서 로컬 변환". PostgREST가 `timestamptz`를 ISO-8601 문자열로 직렬화하므로 왕복 자체는 무손실이나, **정확한 포맷(예: `Z` 서픽스 vs `+00:00`)은 타입 레벨에서 강제되지 않는 런타임 규약**이다. 17일차 매퍼 작성 시 포맷 통일 여부만 확인하면 된다(타입 변경 불요).
- `detail`/`effectModifiers`/`negotiationLog`/`oddsSnapshot`/`criteria`/`constants`/`valueJson`/`jsonSchema` 등 `Readonly<Record<string, unknown>>` 필드는 물리 `jsonb`와 구조적으로 대응하며 확인된 정밀도 충돌 없음. 다만 jsonb 내부에 숫자가 실릴 경우 최상위 필드와 동일한 53비트 규약이 암묵 적용된다는 점만 매퍼 작성자가 인지하면 된다(신규 이슈 아님, §1과 동일 원리).
