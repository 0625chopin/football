## 3. 이 측정이 대표하지 않는 것 (범위 한계 — 정직하게 명시)

- **로컬 dev(webpack) 종단 시간이다.** 프로덕션 빌드는 이 환경에서 빌드 자체가 실패해
  (`CLAUDE.md` "프로덕션 빌드는 번들러와 무관하게 실패") 배포 런타임 기준 재측정이
  불가능했다 — Vercel 등 실제 배포 환경에서는 dev 서버의 요청별 오버헤드(웹팩 dev
  미들웨어, 소스맵 등)가 없어 이보다 빠를 가능성이 높다.
- **네트워크 홉 1개(로컬 → 원격 Supabase REST) 기준**이다. 이 라우트는 내부적으로
  `Promise.all`로 6개 조회(cron_run 최신 1건, SUCCESS 1건, PARTIAL 1건, `CRON_PARAM`
  공통코드, 다음 킥오프, `fixture` 백로그 스캔)를 **병렬** 발사하므로 순차 합산이 아니라
  가장 느린 1건에 수렴한다 — 그래도 지연의 대부분은 로컬↔원격 왕복(RTT) 자체다.
- **현재 데이터 규모 기준**이다. `fixture` 백로그 스캔(`BACKLOG_SCAN_LIMIT=1000`)은
  `SCHEDULED` 건수가 적은 지금 상태에서 측정됐다 — 데이터가 크게 늘면(NFR-SC-001) 이
  스캔 쿼리 비용이 커질 수 있어 10시즌 누적 데이터 적재 후 재측정이 필요하다(이슈 후보,
  §4 참조).
- **동시성 없음.** 순차 40회 호출이며 동시 부하(concurrent request) 시나리오는 측정하지
  않았다 — "1차 측정"의 목적(엔드포인트 존재 후 최초 기준선 확보)에 한정했다.
