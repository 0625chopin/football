### D-18 예외 — `description`은 운영 언어(ko) 우선

ROADMAP Task 021이 명시한 제약이다: **`description`(E-41·E-42)은 번역 카탈로그를 거치지 않고 DB 원문(ko)을 그대로 표시**한다. 이는 R-2(하드코딩 금지)의 예외가 아니라 **애초에 R-2 대상이 아니다** — `description`은 UI 라벨이 아니라 **운영자를 위한 데이터 값**(자유 텍스트)이기 때문이다. 카탈로그 확장 시 en 병기가 가능한 구조로만 열어 둔다(스키마에 이미 `description` 단일 컬럼 — en 병기가 필요해지면 컬럼 추가는 6팀·1팀 협의).

주변 UI 라벨("현재값", "기본값", "저장", "사유" 등)은 **정상적으로 R-2 번역키 대상**이다. `description`만의 예외임을 혼동하지 않는다.

## 07/08 분할 근거 (참고)

`00-공통규약.md` 1절이 1일차에 이미 `07=/admin`, `08=/admin/config`+`/admin/scheduler`로 매핑해 두었다. 이번 4일차 작성도 그 매핑을 그대로 따랐다 — 근거는 3가지다.

1. **위험도 축이 다르다.** 07(`/admin`)은 배속·정지·**월드 리셋**처럼 즉시 상태를 바꾸는 "제어" 화면이고, 08(config·scheduler)은 상수를 신중히 조정하거나 **읽기 전용으로 이력을 훑는** "열람·설정" 화면이다. 위험 조작(2단계 확인)이 있는 화면과 없는 화면을 섞지 않는 편이 각 문서의 R-1~R-14 체크리스트를 적용하기 쉽다.
2. **구현 일정이 붙어 있다.** 05-화면배팅UX팀.md 기준 07 내용은 54~55일차, 08 내용은 56~59일차로 **연속 작업 블록**이 이미 나뉘어 있다.
3. **UI 패턴이 유사하다.** config의 "36개 그룹 목록 + 코드 테이블 + 이력"과 scheduler의 "실행 이력 테이블 + 지표 카드 + 중단 구간"은 둘 다 **목록형·테이블형 화면**으로 컴포넌트 설계 방식(master-detail, 필터, 자체 가로 스크롤)이 겹친다.
