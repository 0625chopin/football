## 2. 문서 포맷 (전 화면 공통 — 이 순서를 반드시 지킨다)

각 화면 문서는 아래 9개 절을 같은 번호·같은 제목으로 갖는다.

```
# NN. 화면명

## 1. 개요            라우트 / UC / FR / 우선순위 / 주 페르소나 / 담당 Task·일차
## 2. 화면 목표       "이 화면에서 사용자가 3초/30초 안에 무엇을 달성하는가" 1~2문장
## 3. 레이아웃        3-1 모바일(320~767) / 3-2 데스크톱(1024+) ASCII 블록도
## 4. 영역 명세       영역ID·명칭·내용·데이터 출처(엔티티)·번역키 프리픽스·비고 표
## 5. 4상태 명세      Loading / Empty / Error / Success 각각의 화면 문구·구조
## 6. 인터랙션        사용자 조작 → 결과. 폴링·카운트다운 등 시간 동작 포함
## 7. 접근성·반응형   NFR-A11Y / NFR-RS 대응 사항을 영역ID로 지목
## 8. 사용 컴포넌트   4팀(013A)·자팀(013B) 컴포넌트 목록 + 신규 필요 여부
## 9. 미확정·확인요청 팀장/타팀 확인이 필요한 항목 (Q-·I-·L- ID 병기)
```

## 3. ASCII 블록도 표기 규약

```
┌─ A1 헤더 ────────────────────────┐   ┌─ ─┘ : 영역 경계
│ [로고]  [리그▾] [시즌 3 · 정규]   │   [ ]   : 컨트롤(버튼/셀렉트/입력)
│                      ⏱ 다음 12:04 │   ▾     : 드롭다운
├─ A2 라이브 그리드 ────────────────┤   ⏱     : 시간/카운트다운
│ ┌────────┐ ┌────────┐            │   ◆     : 상태 배지
│ │◆LIVE 67'│ │◆LIVE 12'│  ← 카드   │   ↻     : 폴링/자동갱신 지점
│ └────────┘ └────────┘   ↻5s      │   …     : 반복(N개)
└──────────────────────────────────┘   ← …   : 주석
```

- 영역 ID는 **화면 문서 내에서 `A1`, `A2`, … 로 유일**하게 부여한다. 4절 표·7절 접근성 명세가 이 ID로 서로를 참조한다.
- 블록도는 **모바일 세로 먼저** 그린다(PS-1이 세로 모바일 주 사용, NFR-RS-001 320px 기준).

## 4. 전 화면 강제 규칙 (예외 없음)

### R-1. 4상태 필수 — FR-UI-002~007·019·025·026
모든 화면은 **Loading / Empty / Error / Success** 4상태를 명세한다. Empty·Error 문구는 FR-UI 요구사항 표에 명문화된 문구를 **그대로** 쓴다(예: `"현재 진행 중인 경기가 없습니다 — 다음 킥오프 HH:MM"`). 임의 창작 금지.

### R-2. 하드코딩 문자열 0 — D-18
와이어프레임에 적는 한글 문구는 **번역 키의 ko 값 초안**이다. 각 영역에 `번역키 프리픽스`를 반드시 지정한다(예: `home.live.*`, `league.table.*`).

> ⚠️ **키 공간 소유권 — 팀장 확정(1일차)**
> **enum 번역키의 네이밍·구조·키 공간은 4팀 단독 소유**다(H-09, 22일차 확정 / 23일차 수신).
> - 1팀 T12는 "enum → 키 매핑을 **타입으로 강제하는 규약**"만 정의하며 **실제 키 문자열은 정하지 않는다.**
> - 3팀은 표시명 **값**만 기여한다(`enums.ko` / `enums.en`).
> - **본 문서 및 화면 문서에 등장하는 모든 번역키 프리픽스는 「제안 지위」다.** 4팀 H-09 수신(23일차) 후 대조하여 확정하며, 그때까지 확정 사실로 인용하지 않는다.

### R-3. 고유명사는 변수 주입 — D-17 / 1팀 T14
**선수명·클럽명은 번역 대상이 아니다.** 문구 템플릿에는 `{playerName}`, `{teamName}` 형태의 플레이스홀더로만 표기한다.

- **국가명은 번역 대상이다** (1팀 T14 확정). 국적은 enum성 값이므로(1팀 P-17 T8) **국적 enum → 번역키 매핑**(1팀 T12, 7일차 / 키 문자열은 4팀 H-09)을 경유해 표시한다. 선수 상세·클럽 상세의 국적 표기는 하드코딩하지 않는다.
- **구장명은 미확정**이다. D-17 원문에 언급이 없으며 절차적 생성 고유명사로 추정되나, 1팀 7일차 `enums.ts` 확정 시 대조한다 → **W-12**.

### R-4. 색상 단독 금지 — NFR-A11Y-002
승격/플레이오프/강등 존, LIVE/예정/종료 배지, 적중/미적중/보이드 등 **모든 상태 구분은 색 + 아이콘 + 텍스트 라벨 3중 표기**로 설계한다. 와이어프레임 4절 표에 아이콘·라벨을 명시한다. (색맹 시뮬 3종 통과가 016 수락 기준)

### R-5. PA 원값 비노출 — FR-PL-004
선수 잠재능력(PA) 원값은 **어떤 화면에도 표기하지 않는다.** 선수 상세에서는 **스카우트 등급 ★1~5**만 노출한다. 와이어프레임 단계에서 PA 수치 칸을 그리지 않는다.

### R-6. 넓은 콘텐츠는 자체 가로 스크롤 — NFR-RS-002
순위표·스탯 테이블·브래킷·36그룹 상수 목록은 **자체 `overflow-x: auto` 컨테이너**로 감싼다. body 가로 스크롤은 0이어야 한다(NFR-RS-001, 320px 검증).

### R-7. 테이블 시맨틱 — NFR-A11Y-005
순위표·스탯·이력 테이블은 `<caption>` + `scope` 명시. 와이어프레임 4절에 caption 문구 초안을 적는다.

### R-8. 폴링 명세 — FR-UI-022 (**계약과 값을 분리 표기**)
자동 갱신 영역은 블록도에 `↻` + 주기를 표기한다. 신규 값 변경 영역은 `aria-live="polite"`(NFR-A11Y-004).

| 구분 | 출처 | 내용 |
|---|---|---|
| **폴링 훅 계약** | **1팀 H-02** (`src/lib/data/polling.ts`, 11일차) | 화면은 이 훅을 **소비만** 한다. `setInterval` 등 폴링 로직을 화면에서 자체 구현하지 않는다 |
| **폴링 주기 값** | **3팀 H-05** 공통코드 | 기본 5초 / 라이브 경기 상세 3초. 하드코딩 금지 |
| **탭 비활성 시 중단** | **1팀 H-02 (확정)** | **탭 비활성 시 폴링 중단은 훅 계약(1팀 H-02)의 책임 범위다. 화면은 별도 처리를 구현하지 않는다.** 근거: 1팀 11일차 Task 004 정의 — "폴링 추상화 훅 계약(기본 5초/라이브 3초, 주기는 공통코드, **탭 비활성 중단**)". FR-UI-022 수락 기준 ③ |

> 5팀 일정의 36일차 "탭 비활성 시 중단", 38일차 "탭 비활성 중단 검증"은 **훅이 제공하는 동작을 적용·검증**하는 작업이다. 중단 로직을 화면에서 새로 만들지 않는다. (팀장 확정, 1일차 — W-14 해소)

### R-9. 시각 표기 — DC-07
서버 저장은 UTC. 화면 표기는 **로케일 로컬 시각으로 변환**한다. 와이어프레임에는 `HH:MM (로컬)` 형태로 적는다. (T13 — 도메인 타입은 타임스탬프 원시값, **서식은 UI 계층 책임**)

### R-10. 데이터 페칭 위치
컴포넌트는 데이터를 페칭하지 않는다(FR-UI-021 수락 기준 ③). 페칭은 화면(page) 레벨 훅 → `dataSource` 어댑터 경유(FR-UI-023). 와이어프레임 4절 `데이터 출처`에는 **엔티티명(E-xx)** 만 적고 쿼리는 적지 않는다.

### R-11. 미래 정보 노출 금지 — NFR-SEC-004 / FR-MT-016
경기 결과는 선계산되어 있어도 **엔진이 제공한 경기 경과분 이후의 이벤트를 화면에 그리지 않는다.** 종료 전 결과 필드는 null로 취급하고 와이어프레임에도 자리만 비워 둔다.
(여기서 "경과분"은 실시간 경과 초가 아니라 **엔진 제공 경기분**이다 — R-14 참조.)

### R-12. 소유 경계
와이어프레임에 등장하는 전역 헤더·사이드내비·푸터·로딩/에러 골격은 **4팀 소유(005·013A)** 이다. 본 문서는 그것을 **참조만** 하고 사양을 바꾸지 않는다. 변경이 필요하면 9절 「확인요청」에 적는다.

### R-13. 승부차기 스코어 분리 표기 — D-19 / 1팀 T18
경기 결과 타입은 승패 판정용 승부차기(PSO) 스코어를 **별도 필드**로 보유한다(1팀 T18). 화면도 이를 **분리해서** 표기한다.

- **① 표기 형식** — `정규+연장 스코어 (승부차기 홈-원정)`. 예: `2 - 2 (승부차기 4-3)`. **두 값을 합산해 표기하지 않는다.**
- **② 집계 제외** — 순위표의 득점·실점·득실, 통계의 시즌/통산 득점, 득점왕 산정에 **PSO 골을 포함하지 않는다**(D-19). 연장전 득점은 **정상 득점으로 포함**한다.
- **③ 승자 판정** — 승패·진출 표시에는 PSO 결과를 정상 반영한다(D-19: 경기 승패 판정에는 반영).
- **④ 번역·접근성** — 번역키 `match.score.pso.*`(제안 지위, R-2). 스크린리더 텍스트에 **"승부차기"를 명시**해 정규 득점과 혼동되지 않게 한다.
- **⑤ 적용 화면** — 경기 상세 스코어보드, 홈 라이브 카드(컵·플레이오프 경기), 일정/결과, 브래킷(`BracketTree`), 클럽 상세 최근 경기.

### R-14. 월드 시간과 실시간의 분리 — FR-AD-001·002 / 2팀 **H-24**
**이 월드의 경기는 배속(0.25×~20×)으로 흐르고 리그별 라운드 간격도 75/90/115분으로 다르다. 화면은 "실시간 1분 = 경기 1분"을 전제하지 않는다.**

- **① 경과분·추가시간은 엔진 제공값을 표시만 한다.** UI가 `(now − kickoff)` 같은 실시간 타이머로 자체 계산하지 않는다. 산출식은 엔진 소유다 — **2팀 H-24 ① (30일차 인계, `speed.ts` 작성은 29일차)**.
- **② UI 자체 tick의 허용 범위 — 카운트다운은 실시간 tick이 정답이다.** 카운트다운 등 UI가 매초 감소시키는 tick은 **킥오프 예정 시각(실시간 타임스탬프)까지의 실시간 잔여**에만 사용한다.
  - 근거: **FR-AD-001** — "배속 변경 시 **미래 킥오프 시각이 비례 재계산**되고 이미 종료된 경기는 영향받지 않는다". 즉 `kickoff_at`은 **서버가 배속을 반영해 산출한 실제 타임스탬프**이므로 클라이언트가 다시 환산하면 **이중 적용**이 된다.
  - 따라서 **카운트다운 환산식은 H-24 계약 범위 밖**이다(초안에 있었으나 철회).
- **③ 재동기화 의무** — 배속 변경 시 미래 킥오프 시각이 **서버에서 재계산**되고(FR-AD-001), 정지/재개 시 **전체 스케줄이 오프셋 지연**된다(FR-AD-002). 따라서 카운트다운은 자체 tick만 믿지 말고 **배속·정지 상태를 구독**해(**H-24 ②**) 변화 시 **서버 값으로 재동기화**한다. ②와 모순되지 않는 **별개 요건**이다.
- **④ 정지 상태 표기** — 시뮬 정지 중에는 카운트다운을 멈추고 "일시정지 중" 상태를 표기한다(FR-AD-002). 정지 구간 오프셋 처리는 **H-24 ③** 규약을 따른다.
- **⑤ 점프 허용** — 배속 변경 시 표시 경과분·잔여 시간이 **불연속적으로 점프할 수 있다.** 이를 오류가 아닌 정상 동작으로 보고, 애니메이션·전환 설계가 점프를 전제하도록 한다(부드러운 보간이 오히려 잘못된 값을 그린다).
- **⑥ 라운드 간격 표기** — 75/90/115분은 **월드 시간 기준값**이다. 화면 문구에서 실시간으로 오해되지 않게 표기한다.
