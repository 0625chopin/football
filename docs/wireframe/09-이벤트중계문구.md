## 3. 단일 소스 및 전제

### 3.1 이벤트 타입 23종 (`src/types/enums.ts` `MatchEventType`, 6일차 확정)

```
KICKOFF, SHOT_ON, SHOT_OFF, SHOT_BLOCKED, GOAL, ASSIST, OWN_GOAL,
PENALTY_AWARDED, PENALTY_SCORED, PENALTY_MISSED, YELLOW_CARD, SECOND_YELLOW,
RED_CARD, FOUL, OFFSIDE, CORNER, SAVE, INJURY, SUBSTITUTION,
HALF_TIME, FULL_TIME, EXTRA_TIME_START, PENALTY_SHOOTOUT
```

값 변경·추가는 이 파일 소관(1팀)이며, 여기서는 **정확히 이 23개만** 다룬다.

### 3.2 D-17 — 고유명사는 변수 주입

**선수명·팀명은 번역 대상이 아니다.** 모든 템플릿에서 `{playerName}`, `{teamName}` 등은 플레이스홀더로만 표기하며 ko/en 템플릿 문자열 안에 실제 이름을 쓰지 않는다. 국가명은 예외적으로 번역 대상이나(`00-공통규약.md` R-3), 이번 6종에는 국적 표기가 없어 해당 없음.

### 3.3 D-18 — 이벤트는 타입 코드만 저장, 문구는 UI 카탈로그

`MatchEvent.type`은 `MatchEventType` 코드값만 저장한다(DB/엔진 계약). 실제 표시 문구는 여기 정의된 **UI 카탈로그**(추후 `match.ts` i18n 카탈로그)가 코드값 → 문구로 매핑한다. 서버가 문구 문자열을 내려주지 않는다.

### 3.4 번역키 프리픽스 — 제안 지위 (R-2)

이 문서의 번역키는 `match.event.<eventKeyLowerCamel>.*` 형태로 제안한다(예: `match.event.goal.label`). **enum 번역키의 네이밍·구조·키 공간은 4팀 단독 소유**이며(H-09, 22일차 확정/23일차 수신), 이 문서의 키는 전부 **제안 지위**다. 4팀 H-09 수신 후 대조해 확정한다. 아래 §5 표에서는 반복 표기하지 않고 이 절을 1회 참조한다.

### 3.5 변수 주입 지점 — `MatchEvent` 필드 매핑 규약

`src/types/match.ts`의 `MatchEvent`가 실제로 갖는 필드는 다음 5개뿐이다. 템플릿 변수는 반드시 이 중 하나에서 파생되어야 하며, 존재하지 않는 필드를 전제한 변수를 만들지 않는다.

| MatchEvent 필드 | 의미 | 템플릿 변수 예시 |
|---|---|---|
| `primaryPlayerId` | 이벤트의 주체 선수 | `{playerName}` (주체 선수명으로 해석) |
| `secondaryPlayerId` | 이벤트 타입별로 의미가 달라지는 범용 보조 참조. **`GOAL`에는 사용하지 않음**(6일차 팀장 지시, 어시스트는 `relatedEventSequence`로만 연결) | `{blockerName}`(SHOT_BLOCKED), `{playerOut}`/`{playerIn}`(SUBSTITUTION, 10일차) 등 이벤트별로 의미를 재선언 |
| `teamId` | 이벤트 관련 팀 | `{teamName}` |
| `relatedEventSequence` | 같은 경기 내 다른 `MatchEvent.sequence` 참조. 현재 확정된 용례는 **`ASSIST` → `GOAL`**(자식→부모, I-37 반영 완료) | ASSIST 템플릿이 GOAL 템플릿에 병합될 때의 연결 키 |
| `detail` (jsonb) | 위 4개로 표현 안 되는 나머지(슛 위치, 카드 사유, 부상 등급 등). **구체 스키마 미확정**(match.ts 주석: "구체 스키마는 소비 시점 확정") | 이번 7일차 6종에는 사용하지 않음. **카드 사유는 1팀 판정으로 enum 신설 보류(I-41 OPEN, §8 확정 3)** — 8~9일차 `YELLOW_CARD`/`SECOND_YELLOW`/`RED_CARD` 문구의 "사유" 자리는 **TBD 플레이스홀더로 비워두고, 자유문자열을 임시로 채우지 않는다**(추후 enum 배치 반영 시 문구가 틀어지는 것을 방지). 부상 등급은 `InjurySeverity`(6일차 확정)로 이미 해소 → **§8 참조** |

### 3.6 표시 유형 (04문서 D3 계승)

- **일반**: 타임라인 1행, 접히지 않음
- **강조 행**: 득점·퇴장 등 시각적으로 두드러지게(굵게/배경 강조 등 — 시각 토큰은 4팀 H-11 이후 확정, 여기서는 "강조 대상"만 지정)
- **구분선 행**: 킥오프/하프타임/종료 등 경기 흐름 전환점
- **병합 표시**: 독립 행을 만들지 않고 다른 이벤트 행에 붙여 표시(`ASSIST` → `GOAL`)
- **기본 접힘**: `FOUL`/`OFFSIDE`/`CORNER` (04문서 W-19 확정 사항, 8~9일차 해당 — 이번 6종에는 없음)

### 3.7 ko/en 템플릿 작성 형식

- 각 이벤트는 `ko 템플릿` / `en 템플릿` 두 문자열을 병기한다.
- 변수는 `{camelCase}` 표기, 실제 렌더 시 값으로 치환된다.
- 문장부호·존댓말 톤은 ko는 관전 중계체(현재형 종결, "~합니다/~!"), en은 스포츠 캐스터체(현재형)로 통일한다.

**9일차 (#13~18)**: `RED_CARD`, `FOUL`, `OFFSIDE`, `CORNER`, `SAVE`, `INJURY`

**10일차 (#19~23)**: `SUBSTITUTION`, `HALF_TIME`, `FULL_TIME`, `EXTRA_TIME_START`, `PENALTY_SHOOTOUT`

> `YELLOW_CARD`/`RED_CARD`(사유)·`INJURY`(등급)는 `detail` jsonb 또는 enum 매핑에 의존한다 — 8~9일차 착수 전 **§7 이슈·§8 확정 사항**을 먼저 확인할 것. 카드 사유는 **TBD 슬롯**으로 남긴다(§8 확정 3).
>
> **`PENALTY_SHOOTOUT`(10일차, #23) 설계 유의** — 7일차 교차 점검 **확정 2**(§8 참조): 승부차기는 **킥마다 별도 `MatchEvent`**(같은 `type=PENALTY_SHOOTOUT`, 다른 `sequence`/`detail`)로 여러 건 발생한다. 10일차 문구는 **단일 요약 문구가 아니라 킥 단위 반복 문구**(예: `{playerName}` 성공/실패 + 누적 스코어)로 설계해야 한다 — 04문서 D3-p 절의 킥 축 표기와 정합.

## 6. 문구 작성 시 공통 체크리스트 (8~10일차 담당자용)

1. 아이콘·변수명은 04문서 D3 표와 동일한가 (임의로 새 아이콘/변수명을 만들지 않았는가)
2. 모든 고유명사가 `{변수}` 형태인가 (D-17 — 하드코딩된 이름이 없는가)
3. 변수가 `MatchEvent`의 실재 필드(`primaryPlayerId`/`secondaryPlayerId`/`teamId`/`relatedEventSequence`/`detail`)에서 파생 가능한가 (§3.5)
4. `detail` jsonb에 의존하는 필드(카드 사유 등)는 **§8 확정 사항**을 먼저 확인했는가 — 카드 사유는 **TBD 슬롯**으로 남기고 자유문자열을 임의로 채우지 않는다(I-41 OPEN)
5. 번역키가 `match.event.*` 프리픽스이며 "제안 지위"임을 전제했는가

## 8. 교차 점검 확정 사항 (7일차 1차 교차 점검, 팀장 승인)

7일차 1차 교차 점검(1팀 코어·품질 / 2팀 시뮬레이션엔진 / 5팀 대상)에서 5팀이 제기한 안건에 대해 팀장이 확정한 사항을 기록한다. 원 근거는 1팀 `docs/ISSUES.md`, 2팀 `docs/devStep/05.SP1사전검토_2팀_7일차.md`이며, 이 절은 **인용·요약**이다.

| # | 확정 내용 | 반영 위치 |
|---|---|---|
| **확정 1** | **정규시간·연장 PK 득점은 `PENALTY_SCORED` 단독 발생. `GOAL`을 추가로 발생시키지 않는다.** 5팀이 제기한 "이중 발생 시 스코어 접기(E-1) 오집계·문구 중복" 우려에 대해 5팀 요청 (a)안(단독 발생)이 채택됨. 2팀도 독립적으로 같은 결론에 도달 | §4 #5 GOAL 비고 반영 완료. 8일차 `PENALTY_SCORED` 문구는 이 전제로 확정 작성(재작성 불필요) |
| **확정 2** | **승부차기는 킥마다 별도 `MatchEvent`**(같은 `type=PENALTY_SHOOTOUT`, 다른 `sequence`/`detail`)로 여러 건 발생한다 | §5 남은 분량 절에 10일차 설계 유의사항으로 반영 완료 |
| **확정 3** | **카드 사유 enum 신설 보류**(1팀 판정, **I-41 OPEN** 등재). FR-MT-011에 사유 taxonomy 근거가 없어 지금 값을 만들면 근거 없는 가정이 되며, 8일차 동결 블로커는 아니다. 8~9일차 카드 문구는 "사유"를 **TBD 슬롯**으로 비워 진행하고 자유문자열을 임시로 채우지 않는다(추후 배치 반영 시 문구 어긋남 방지). 값은 2팀 023/024 파울·카드 로직 설계 후 1팀이 배치 반영(C-7) | §3.5 `detail` 행, §6 체크리스트 4번, §7 이슈 표 반영 완료 |
| **W-25 회신** | 엔진은 `GOAL`·`ASSIST`를 같은 틱의 한 판정 배치에서 함께 생성하며 로그상 인접 `sequence`(`GOAL`=N, `ASSIST`=N+1, `relatedEventSequence`=N)로 존재하도록 설계(2팀 확정). 단 **"클라이언트 동시 도달"은 엔진 보장 범위 밖**이며 전송 층(폴링 배치)은 1팀 H-02 폴링 계약(11일차)에서 재확인 필요. `OWN_GOAL`/`PENALTY_SCORED`에 `ASSIST` 비연결 전제는 확정(I-37과 일치) | §4 #6 ASSIST 비고, §7 이슈 표 반영 완료 |
