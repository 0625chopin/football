## 3. 단일 소스 및 전제

### 3.1 이벤트 타입 23종 (`src/types/enums.ts` `MatchEventType`, 6일차 확정)


값 변경·추가는 이 파일 소관(1팀)이며, 여기서는 **정확히 이 23개만** 다룬다.

### 3.2 D-17 — 고유명사는 변수 주입

**선수명·팀명은 번역 대상이 아니다.** 모든 템플릿에서 `{playerName}`, `{teamName}` 등은 플레이스홀더로만 표기하며 ko/en 템플릿 문자열 안에 실제 이름을 쓰지 않는다. 국가명은 예외적으로 번역 대상이나(`00-공통규약.md` R-3), 8일차분(#7~12)에도 국적 표기가 없어 해당 없음.

### 3.3 D-18 — 이벤트는 타입 코드만 저장, 문구는 UI 카탈로그

`MatchEvent.type`은 `MatchEventType` 코드값만 저장한다(DB/엔진 계약). 실제 표시 문구는 여기 정의된 **UI 카탈로그**(추후 `match.ts` i18n 카탈로그)가 코드값 → 문구로 매핑한다. 서버가 문구 문자열을 내려주지 않는다.

### 3.4 번역키 프리픽스 — 제안 지위 (R-2)

이 문서의 번역키는 `match.event.<eventKeyLowerCamel>.*` 형태로 제안한다(예: `match.event.goal.label`). **enum 번역키의 네이밍·구조·키 공간은 4팀 단독 소유**이며(H-09, 22일차 확정/23일차 수신), 이 문서의 키는 전부 **제안 지위**다. 4팀 H-09 수신 후 대조해 확정한다. 아래 §4·§5 표에서는 반복 표기하지 않고 이 절을 1회 참조한다.

### 3.5 변수 주입 지점 — `MatchEvent` 필드 매핑 규약

`src/types/match.ts`의 `MatchEvent`가 실제로 갖는 필드는 다음 5개뿐이다. 템플릿 변수는 반드시 이 중 하나에서 파생되어야 하며, 존재하지 않는 필드를 전제한 변수를 만들지 않는다.

| MatchEvent 필드 | 의미 | 템플릿 변수 예시 |
|---|---|---|
| `primaryPlayerId` | 이벤트의 주체 선수 | `{playerName}` (주체 선수명으로 해석) |
| `secondaryPlayerId` | 이벤트 타입별로 의미가 달라지는 범용 보조 참조. **`GOAL`에는 사용하지 않음**(6일차 팀장 지시, 어시스트는 `relatedEventSequence`로만 연결) | `{blockerName}`(SHOT_BLOCKED), `{foulerName}`(PENALTY_AWARDED, 8일차), `{keeperName}`(PENALTY_MISSED, 8일차), `{playerOut}`/`{playerIn}`(SUBSTITUTION, 10일차) 등 이벤트별로 의미를 재선언 |
| `teamId` | 이벤트 관련 팀 | `{teamName}` |
| `relatedEventSequence` | 같은 경기 내 다른 `MatchEvent.sequence` 참조. 현재 확정된 용례는 **`ASSIST` → `GOAL`**(자식→부모, I-37 반영 완료) | ASSIST 템플릿이 GOAL 템플릿에 병합될 때의 연결 키. **`PENALTY_AWARDED` → `PENALTY_SCORED`/`PENALTY_MISSED`의 연결 여부는 미확정**(§8 확인요청 신규) |
| `detail` (jsonb) | 위 4개로 표현 안 되는 나머지(슛 위치, 카드 사유, 부상 등급 등). **구체 스키마 미확정**(match.ts 주석: "구체 스키마는 소비 시점 확정") | 8일차 6종에는 사용하지 않는다. **카드 사유는 1팀 판정으로 enum 신설 보류(I-41 OPEN, §9 확정 3)** — `YELLOW_CARD`/`SECOND_YELLOW` 문구의 "사유" 자리는 **TBD 플레이스홀더로 비워두고, 자유문자열을 임시로 채우지 않는다**(추후 enum 배치 반영 시 문구가 틀어지는 것을 방지). 부상 등급은 `InjurySeverity`(6일차 확정)로 이미 해소 → 9일차 `INJURY` 작성 시 참조 |

### 3.6 표시 유형 (04문서 D3 계승)

- **일반**: 타임라인 1행, 접히지 않음
- **강조 행**: 득점·퇴장 등 시각적으로 두드러지게(굵게/배경 강조 등 — 시각 토큰은 4팀 H-11 이후 확정, 여기서는 "강조 대상"만 지정)
- **구분선 행**: 킥오프/하프타임/종료 등 경기 흐름 전환점
- **병합 표시**: 독립 행을 만들지 않고 다른 이벤트 행에 붙여 표시(`ASSIST` → `GOAL`)
- **기본 접힘**: `FOUL`/`OFFSIDE`/`CORNER` (04문서 W-19 확정 사항, 9일차 해당 — 8일차 6종에는 없음)

### 3.7 ko/en 템플릿 작성 형식

- 각 이벤트는 `ko 템플릿` / `en 템플릿` 두 문자열을 병기한다.
- 변수는 `{camelCase}` 표기, 실제 렌더 시 값으로 치환된다.
- 문장부호·존댓말 톤은 ko는 관전 중계체(현재형 종결, "~합니다/~!"), en은 스포츠 캐스터체(현재형)로 통일한다.

## 6. 남은 분량 (10일차 예정, 미작성)

아래는 이벤트명만 나열한 자리다. §4·§5·§5-2와 동일한 표 스키마(아이콘/표시유형/변수 주입 지점/ko/en/번역키/비고)로 이어서 채운다.

**10일차 (#19~23)**: `SUBSTITUTION`, `HALF_TIME`, `FULL_TIME`, `EXTRA_TIME_START`, `PENALTY_SHOOTOUT`

> **`PENALTY_SHOOTOUT`(10일차, #23) 설계 유의** — 7일차 교차 점검 **확정 2**(§9 참조): 승부차기는 **킥마다 별도 `MatchEvent`**(같은 `type=PENALTY_SHOOTOUT`, 다른 `sequence`/`detail`)로 여러 건 발생한다. 10일차 문구는 **단일 요약 문구가 아니라 킥 단위 반복 문구**(예: `{playerName}` 성공/실패 + 누적 스코어)로 설계해야 한다 — 04문서 D3-p 절의 킥 축 표기와 정합.
>
> **`SUBSTITUTION`(10일차, #19) 참고**: 04문서 D3 표(#19)의 주체 변수가 `{playerOut}` → `{playerIn}` 2개 조합이며 부가 변수가 `{teamName}`이다. `match.ts` 주석은 `secondaryPlayerId`의 "이벤트 타입별로 의미가 달라지는 범용 보조 참조" 용례 중 하나로 `SUBSTITUTION`의 교체 아웃 선수를 명시적으로 들고 있다 — 즉 `{playerOut}` ← `secondaryPlayerId`, `{playerIn}` ← `primaryPlayerId`(또는 그 반대)로 해석될 여지가 있으므로, 10일차 착수 시 어느 쪽이 "들어오는 선수"인지 임의로 정하지 말고 `match.ts` 주석 원문을 재확인할 것.

## 7. 문구 작성 시 공통 체크리스트 (10일차 담당자용)

1. 아이콘·변수명은 04문서 D3 표와 동일한가 (임의로 새 아이콘/변수명을 만들지 않았는가)

4. `detail` jsonb에 의존하는 필드(카드 사유 등)는 **§9 확정 사항**을 먼저 확인했는가 — 카드 사유는 **TBD 슬롯**으로 남기고 자유문자열을 임의로 채우지 않는다(I-41 OPEN)
5. 번역키가 `match.event.*` 프리픽스이며 "제안 지위"임을 전제했는가
6. **(8일차 추가)** 04문서 D3 표에 변수가 누락되거나(예: #9 `{teamName}`) 서브 케이스 분기가 없는(예: #10 골키퍼 선방/골대 이탈) 경우, 임의로 지어내지 말고 **가정을 명시**하고 §8에 확인요청으로 등재했는가

- **동결을 막는 타입 갭: 0건.** 검토 도중 한때 미완성 상태였던 E-45~47(§11.2)도 같은 세션 내 1팀이 완료해 확인 완료로 종결했다.
- 이 팀이 34~73일차에 만들 7개 화면/기능 중 나머지는 전부 기존 타입으로 표현 가능하거나(§11.4), 5팀 자신의 후속 설계 범위로 이미 문서화돼 있다(§11.3).
- `BetMarket.marketType`처럼 "구체화가 미뤄진" 필드들은 전부 **5팀 자신이 60일차 이후에 채우는 설계**이므로, 지금 폭넓게(`string`) 둔 것이 오히려 올바른 설계다 — 동결을 서두르라고 좁혀 달라는 요청을 하지 않는다.
