## 4. 혼합타입 8그룹 상세 분석 — I-58 정밀화

카탈로그 헤더 주석과 `docs/ISSUES.md` I-58은 8개 그룹(`FITNESS_PARAM`·`INJURY_PARAM`·`SPONSOR_PARAM`·`ODDS_PARAM`·`RETIREMENT_PARAM`·`SANCTION_PARAM`·`CUP_PARAM`·`TRANSFER_PARAM`)의 그룹 대표 `valueType`이 상위집합(superset) 채택이라 실제 코드별 값 형태와 어긋날 수 있다고 지적했다. I-58은 "위젯 선택을 그룹 `valueType`에 전적으로 의존하지 말고 코드 개별 값 형태로 보조 판단하라"고 권고했는데, 8개 그룹의 `description` 코드 예시를 개별 파싱해보면 **실제 위젯 카테고리 충돌은 1개 그룹에서만 발생**한다.

### 4.1 DECIMAL/INT 혼재 7그룹 — 위젯 충돌 없음

| 그룹 | 코드 예시(카탈로그 description) | 값 형태 |
|---|---|---|
| `FITNESS_PARAM` | `MULT_BASE=0.75`(소수), `DRAIN_FULL=18`(정수), `RECOVER=12`(정수), `STREAK_FACTOR=0.7`(소수) | 전부 스칼라 숫자 |
| `INJURY_PARAM` | `SEVERITY_1_MULT=0.93`(소수), `S2_MIN/MAX=1/3`(정수) 등 | 전부 스칼라 숫자 |
| `SPONSOR_PARAM` | `MAX_PER_TEAM=3`, `CONTRACT_MIN=1`, `SHARE_PCT_CAP=30` 등 | 전부 스칼라 숫자 |
| `ODDS_PARAM` | `MC_N_MATCH=3000`(정수), `OVERROUND=1.06`(소수) 등 | 전부 스칼라 숫자 |
| `RETIREMENT_PARAM` | `RISK_START_AGE=34`(정수), `BASE_PROB`(소수 확률) | 전부 스칼라 숫자 |
| `SANCTION_PARAM` | `REP_PENALTY_PERMANENT=3`(정수), `GRANT_PCT=0.40`(소수) | 전부 스칼라 숫자 |
| `TRANSFER_PARAM` | `RATE_MIN_PCT=8`(정수), `SUCCESS_MIN=0.05`(소수) | 전부 스칼라 숫자 |

이 7개 그룹은 정수 코드와 소수 코드가 섞여 있을 뿐, **위젯 카테고리는 전부 "숫자 입력"으로 동일**하다. 정수/소수 차이는 입력 `step`(1 vs 0.01)만 다르며 위젯 자체를 바꿀 필요가 없다. 즉 **이 7개 그룹은 I-58이 우려한 "과잉 위젯" 문제가 실제로는 발생하지 않는다** — 그룹 대표 `valueType`(DECIMAL)을 그대로 숫자 입력에 써도 무방하다.

### 4.2 `CUP_PARAM` — 유일한 실제 위젯 충돌

`CUP_PARAM`의 코드 예시: `BYE_COUNT=4`(스칼라 정수), `INSERT_ROUNDS=[6,12,18,24,32,40]`(배열). 그룹 대표 `valueType`은 `JSON`(INT/JSON 혼재의 상위집합)이지만, `BYE_COUNT`를 JSON 코드 에디터로 노출하면 단순한 정수 입력에 불필요한 편집 복잡도를 준다 — I-58이 예시로 든 문제가 정확히 이 그룹에서만 실재한다.

**권고 위젯 분기 (코드 단위, `value` 실제 파싱 형태 기준)**:

| 코드 | 실제 값 형태 | 권고 위젯 |
|---|---|---|
| `BYE_COUNT` | 스칼라 정수 | 숫자 입력 |
| `INSERT_ROUNDS` | 배열 | JSON 에디터(배열 표기) |

즉 `CUP_PARAM`만 그룹 전체를 JSON 에디터로 통일하지 말고, **코드별로 값이 스칼라인지 배열/객체인지 파싱해 위젯을 분기**해야 한다.

**분기 신호 정정(3팀 회신 반영, 13일차)**: 최초 초안은 이 분기를 `CommonCode.valueNum`/`valueJson` 필드의 null 여부로 판별할 수 있다고 적었으나 **이는 틀렸다**. `CUP_PARAM` 그룹의 `valueType`은 그룹 대표값이 아니라 **그룹 전체의 단일 `'JSON'`**이며(`catalog.ts` 484~495행), E-42의 DB CHECK 제약이 코드 단위가 아니라 **그룹 단위**로 걸리므로 이 그룹 안에서는 `BYE_COUNT`를 포함한 **모든 코드가 `valueJson`에 값을 채우고 `valueNum`은 항상 null**이다 — 즉 `valueNum`이 채워지는 코드가 애초에 0건이라 분기 신호로 쓸 수 없다. 실제로 3팀 `fallback.ts:258~261`(`installHardcodedFallback` 안 `SAFE_DEFAULT_VALUES.CUP_PARAM`)도 스칼라값까지 `{ value: 4 }`처럼 JSON으로 감싸 처리하고 있다.

**올바른 분기 신호는 `valueJson`을 파싱한 뒤의 값 형태(shape)뿐이다**: `typeof parsed.value === 'number'`(또는 스칼라)면 숫자 입력 위젯으로 렌더하고 값만 반영, `Array.isArray(parsed.value)`(또는 객체)면 JSON 에디터로 렌더한다. 이 판별 로직은 **현재는 5팀 화면 로컬(`ConfigEditForm` 내부)에 둔다** — 3팀 논거를 팀장이 채택 확정한 바에 따르면 `{ value: ... }` 래핑은 `fallback.ts`의 임시 표현일 뿐 공개 타입 계약이 아니고(`jsonSchema: null`인 현재 상태가 이를 뒷받침), 이 지식이 정식 계약이 되는 시점은 **37일차 `schema.ts`가 `CUP_PARAM.jsonSchema`를 채울 때**다. **021 착수(56일차)는 37일차보다 뒤이므로, 그 시점엔 이미 `jsonSchema`를 소스로 한 분기로 교체 가능할 수 있다 — `ConfigEditForm` 구현 시 이 로컬 판별 로직을 `jsonSchema` 유무로 교체 가능하도록 설계해 둘 것.**

### 4.3 순수 JSON 4그룹 — 코드별 분기 불필요

`WEATHER_EFFECT`(날씨별 계수 객체)·`RATING_WEIGHT`(포지션군별 가중치 객체)·`OVR_WEIGHT`(11군별 가중치 객체)·`MANAGER_MATCHUP`(성향 상성 매트릭스)은 그룹 내 모든 코드가 일관되게 객체 형태다("~별 계수 객체"). 코드별 분기 없이 그룹 전체를 JSON 에디터로 처리하면 된다.

## 5. 범위 검증(`min_value`/`max_value`) 및 `json_schema` 현황

- 카탈로그 36개 그룹 **전량이 `minValue: null`, `maxValue: null`, `jsonSchema: null`**이다(3팀 헤더 주석 근거: 05문서 표에 그룹별 숫자 범위 컬럼이 없어 실측 없이 추정값을 넣지 않았다는 명시적 설계 결정).
- 따라서 08문서 H3 검증 규칙(V-1~V-3: 저장 시 범위·스키마 인라인 검증)은 **현재는 값이 없어 검증이 항상 통과(스킵)하는 상태**다. "범위 슬라이더"는 wireframe에 없는 표현이므로 이 문서도 "숫자 입력 + 범위 인라인 검증(값 세팅 시 활성화)"으로만 표기한다.
- **활성화 시점 정정(3팀 회신 반영, 13일차)**: 최초 초안은 "11일차 `fallback.ts`·37일차 `schema.ts` 확정 후 활성화"라고 적었으나 **틀렸다**. 범위·스키마 검증 활성화는 **37일차 `schema.ts`(NFR-CFG-004) 단독** 소관이다. 11일차 `fallback.ts`는 "하드코딩 안전 기본값(값 자체)" 테이블이지 범위 활성화가 아니며, `fallback.ts:30`이 "min/max 전 그룹 `null` 원칙을 그대로 따른다"고 명시적으로 밝히고 있다 — 즉 11일차 산출물도 범위값을 채우지 않는다. 021 착수(56~57일차)는 37일차보다 뒤이므로 그 시점에는 실제 범위값이 채워져 있을 가능성이 높으며, 그때 이 표의 위젯 매핑에 범위 표시(`허용 범위 N~M`)만 얹으면 된다.

## 6. 발효 시점(`apply_policy`) 표시 — 08문서 H3-p 재사용

36개 그룹의 `applyPolicy` 분포: `NEXT_SEASON` 32개, `IMMEDIATE` 2개(`UI_PARAM`, `CRON_PARAM`), `NEXT_MARKET` 2개(`ODDS_PARAM`, `BET_LIMIT`). 08문서 H3-p가 이미 3분기 표시 규칙을 정의했으므로 이 문서는 그 규칙을 그대로 쓰고, 그룹별 분류값만 위 §3 표의 "applyPolicy" 열로 제공한다. 그룹마다 정책이 고정이라 운영자가 정책 자체를 바꿀 수 없다는 원칙(08문서)도 동일하게 적용된다.
