### D4·D5 — 실시간 스탯 노출 제약 (팀장 승인, 2일차 / W-21 해소) — **S-1 ~ S-4**

팀 스탯·선수 평점의 **종료 전 실시간 노출은 승인**되었다. 2팀 확인: 스탯은 이벤트 로그 파생(**AS-10**)이므로 `minute ≤ 경과분` 이벤트만 접으면 정확히 산출되며 성능 부담도 없다(경기당 이벤트 수백 건).

당초 우려했던 **"누적 급증으로 다음 이벤트 유추"는 발생하지 않는다** — 노출 범위가 이벤트와 **동일 경계**라 스탯이 이벤트보다 앞서 나가지 않기 때문이다. 다만 1팀(침투 테스트 담당)이 지목한 **실제 유출 경로 4가지**를 반드시 지킨다.

| ID | 제약 | 위반 시 |
|---|---|---|
| **S-1** | **노출된 이벤트에서 재계산한 값만** 전송한다. 사전 집계된 **최종 스탯 직송 금지** | 최종 스탯이 그대로 유출 |
| **S-2** | **서버가 컷오프를 강제**한다. 클라이언트 필터링 금지, **클라이언트 시계 미신뢰**(FR-MT-016 ④) | 시계 조작으로 전체 열람 |
| **S-3** | 이벤트와 스탯의 컷오프는 **단일 소스·단일 응답**이다 | 산정 타이밍이 갈리면 스탯이 이벤트보다 **1틱 앞서 갱신되어 다음 이벤트가 선노출** |
| **S-4** | **파생 지표에도 동일 컷오프** — xG·점유율·**선수 평점** | **최종 평점이 새면 경기 결과가 역산됨** |

> 1팀이 **041 침투 테스트에 "스탯–이벤트 컷오프 불일치" 케이스를 편입**한다. S-3이 이 화면의 가장 미묘한 실패 지점이다.

**xG는 D5에서 제외하지 않는다 (요구사항 근거 확정, 팀장 판정)**
`03-functional-requirements.md:990`(선수 `xg`) · `:1027`(GK `xg_prevented`) · `:1059`(팀 `xg_for`/`xg_against`/`xg_diff`) · `05-data-requirements.md:261`(`match_event.detail`에 xG 포함). 날씨·감독 계수가 xG를 피연산자로 정의하므로(`:241`, `:269`) **엔진 핵심 변수**다. W-23(주심)과 달리 근거가 명확하다.

### 엔진 계약 정합 (2팀 확인, 2일차) — **E-1 ~ E-4**

이 화면은 엔진 반환 구조를 그대로 소비한다. 아래 4건은 **UI가 임의로 다르게 전제하면 곧바로 어긋나는 지점**이다.

- **E-1. 스코어 스냅샷은 이벤트에 저장되지 않는다.** `GOAL`·`HALF_TIME`·`FULL_TIME`·`EXTRA_TIME_START` 행에 보이는 스코어는 **UI가 노출된 이벤트를 접어(fold) 산출**한다. 이벤트에 스코어를 넣는 것은 **AS-10(이벤트 로그가 SSOT) 위반이자 이중 소스**다.
  → R-11상 UI는 **노출분만 보유**하므로 누적 계산 결과는 항상 경과분 시점과 일치한다. **별도 컷오프 처리가 불필요하다**(구조적으로 안전).
- **E-2. `ASSIST`는 `GOAL`에 연결 참조로 병합한다.** 엔진이 연결 참조(`relatedEventIndex` 등)를 제공하며(2팀 10일차 `events.ts`), UI는 **시간 근접 추정으로 짝짓지 않는다.** 같은 분에 두 골이 나면 추정은 반드시 틀린다.
  - **참조 방향 = `ASSIST` → `GOAL` (자식 → 부모)** — 팀장 확정(2일차). UI는 `GOAL` 행을 그릴 때 "나를 가리키는 `ASSIST`"를 찾는다. 반대 방향이면 "도움 없는 골"과 "참조 null"을 구분하는 분기가 늘어나므로 소비 측 분기가 적은 이 방향으로 고정한다.
  - **동시 노출 보장** — 두 이벤트는 같은 `minute`이므로 컷오프에서 **원자적으로 함께 노출**되어야 한다. `GOAL`만 먼저 노출되면 "도움 없는 골"이 표시됐다가 도움이 붙는 **깜빡임**이 발생한다. 2팀 10일차 설계에 반영 지시됨(회신 대기).
  - `OWN_GOAL`·`PENALTY_SCORED`에는 `ASSIST`가 붙지 않는 것으로 전제한다(2팀 회신 대기).
- **E-3. 시각은 `minute` + `stoppage` 2필드다.** 추가시간을 정규 분에 **더해서 접지 않는다.**
  - 표기: `45+2'` (minute=45, stoppage=2) — **`47'`로 접는 것은 금지**
  - `stoppage = 0`이면 `45'`로만 표기
  - 정렬은 `(minute, stoppage, eventIndex)` 복합 키. `45+2`는 `46'`보다 **앞**이다
  - 번역키: `match.time.withStoppage` / `match.time.plain`
- **E-4. 카드 사유·부상 등급은 enum 코드로만 제공된다.** 자유 문자열이 아니다. 아래 이벤트 표의 "사유"·"부상 등급" 칸은 **enum → 번역키 매핑**(R-2, 4팀 H-09 키 공간)을 경유해 표시하며, 서버 문자열을 그대로 출력하지 않는다.

**규약 준수 사항**
- **R-3 / D-17** — `{playerName}`·`{teamName}`은 **변수 주입**이며 번역 대상이 아니다. **문구 템플릿만 번역 카탈로그**(`src/i18n/messages/*/match.ts`, 44일차 4팀 협의)에 둔다.
- **R-2** — 이벤트 타입 → 번역키 매핑을 사용한다. 표시 문구 하드코딩 0.
- **R-13** — `PENALTY_SHOOTOUT`은 타임라인 본문에 섞지 않고 **별도 섹션**으로 분리하며, D1 스코어보드에도 `2 - 2 (승부차기 4-3)` 형식으로 분리 표기한다. **개인 통산 득점에 포함되지 않음**(D-19)을 UI에서도 오해 없게 한다.
- **볼륨 관리 — 부가 이벤트 기본 접힘 (팀장 확정, 2일차 / W-19 해소)**
  `FOUL`·`OFFSIDE`·`CORNER`는 90분간 수십 건 발생해 득점·퇴장을 묻어버린다. **기본 접힘 + "부가 이벤트 표시" 토글**로 처리한다.
  - **근거**: FR-UI-007이 요구하는 것은 **이벤트가 분 단위로 제공되는 것**이지 전량이 동시에 시야에 있어야 한다는 뜻이 아니다. 접힌 이벤트도 **DOM에 존재하고 토글로 즉시 접근 가능**하면 요구사항은 충족된다. 오히려 수십 건의 파울로 득점이 묻히면 **UC-002(경기 흐름 파악)의 실질 목적**을 해친다.
  - **조건 ①** — **접힘 대상 이벤트 타입은 공통코드로 조정 가능해야 한다.** 어떤 타입을 접을지는 밸런싱 대상이므로 **화면에 하드코딩하지 않는다**(3팀 H-05 경유. 1팀 체크리스트 **C-12** 적용 대상).
  - **조건 ②** — 접힘 상태에서도 **건수를 표시**한다(예: `파울 12건 ▾`). 정보 은닉이 아니라 접힘임이 드러나야 한다.
  - 접힘 토글은 **R-11 경계와 무관**하다. 접힘은 "이미 받은 것을 접는 것"이고, R-11은 "받지 않는 것"이다. 두 개념을 구현에서 섞지 않는다.

### D3 — R-11 경계 표시 (**이 화면의 핵심 제약**)

```
  67' ⚽ …                    ← 엔진이 반환한 마지막 이벤트
  ──────────────────────────
  ⏳ 경기 진행 중 — 68분 이후는 아직 공개되지 않았습니다
```

- 서버가 `min(now − kickoff_at, 경기 길이)` 로 노출 범위를 산정하며(FR-MT-016), **클라이언트는 받은 것만 그린다.**
- **금지**: 전체 이벤트를 미리 받아 CSS/JS로 가리는 구현. 그것은 DOM·네트워크 탭에서 결과가 새어나가므로 **NFR-SEC-004 위반**이다.
- 경계 문구는 "데이터가 없음"이 아니라 **"아직 공개되지 않음"** 으로 표현해 Empty 상태와 구분한다.

### D3-p — 승부차기 섹션: **킥 축 R-11 경계** (팀장 승인, 2일차 / W-22 해소)

승부차기는 **킥 단위로 실시간 노출**한다. 결과만 던지면 UC-002의 긴장감이 사라진다. 엔진이 이미 킥 단위 결정론 산출을 하며 각 킥이 개별 이벤트(`kickIndex`·성공여부·누계)로 제공된다.

**⚠️ 노출 축이 이원화된다 — PSO에는 경기 분(minute) 축이 없다.**

| 구간 | 노출 축 | R-11 경계 판정 |
|---|---|---|
| 정규·연장 | **분 축**(`minute` + `stoppage`) | `min(now − kickoff_at, 경기 길이)` |
| **승부차기** | **킥 축**(`kickIndex`) | **킥당 월드시간 간격**으로 산정 — 2팀 **H-24 ① 부속 항목**(30일차 계약 포함) |

```
  ┌─ 승부차기 (4 - 3) ──────────────┐
  │ 홈  ⚽ ⚽ ✕ ⚽ ⚽              │
  │ 원정 ⚽ ⚽ ⚽ ✕ ⏳             │  ← 5번째 키커 결과 미공개
  │ ─────────────────────────────── │
  │ ⏳ 다음 키커 결과는 아직         │
  │    공개되지 않았습니다           │
  └─────────────────────────────────┘
```

- **다음 키커 결과 비노출**이 이 섹션의 R-11 요건이다. 킥당 간격은 **공통코드**로 정의되며 화면에 하드코딩하지 않는다(C-12).
- 킥 축 경계도 **서버가 산정**한다. 클라이언트가 전체 킥을 받아 순차 공개하는 구현은 **금지**(C-23 제안 항목과 동일 취지).
- PSO 스코어는 **정규 스코어와 합산 표기하지 않는다**(R-13 ①). 개인 통산 득점에도 미포함(D-19)임이 UI에서 오해되지 않도록 섹션 제목에 "승부차기"를 명시한다.
