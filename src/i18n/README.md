# `src/i18n/` 안내서

Task 011(4팀, 15~22일차)의 국제화 모듈. 이 문서는 (1) 현재 구조를 처음 보는 사람을 위한
안내, (2) **번역 대상 경계**를 실행 가능한 규칙으로 명문화하는 것을 목적으로 한다.

## 2. 번역 대상 경계 (D-18)

원칙 선언은 `src/types/config.ts`의 `EnumTranslationCatalog` 바로 아래 주석(D-18,
I-14/I-15 해소)이 단일 소스다:

> - **번역함**: UI 레이블, `EnumTranslationCatalog`로 감싼 enum 표시명(포지션·이벤트·부상
>   등급·전술 성향·페이즈·수상 종류 등), 안내·에러 문구.
> - **번역하지 않음(고유명사, D-17 절차적 생성)**: 선수·감독·클럽·스폰서 이름, 구장명
>   (`Team.stadiumName`, I-15), 시드 값.

이 문서는 그 원칙을 **`src/i18n/` 카탈로그를 작성/리뷰할 때 실제로 부딪히는 형태**로
풀어 쓴다.

### 2.1 기본 규칙

- **카탈로그(`messages/**`)에 넣는다** = 유한 집합의 UI 문구·표시명. 새 항목이 생겨도
  "번역해야 할 목록"이 코드 리뷰로 드러난다(enum 멤버 추가 → `EnumTranslationCatalog<T>`
  타입 오류로 강제 노출).
- **카탈로그에 넣지 않는다** = 시드 절차 생성으로 개방형(open-ended)인 문자열. 몇 개인지
  미리 알 수 없고, "번역"이 애초에 성립하지 않는다(고유명사는 로케일이 바뀐다고 다른
  문자열이 되지 않는다).
- 판별 질문: **"이 값의 가능한 전체 목록을 지금 나열할 수 있는가?"** — 나열 가능하면
  번역 대상(카탈로그행), 시드마다 새로 생성되면 번역 비대상(원문 그대로 표시).

### 2.2 애매한 경계 사례

**(a) 열거형 표시명 vs 시드에서 온 고유명사 — "이름이 붙어 있다"는 표면 유사성에 속지
않는다.**

| 필드 | 판정 | 이유 |
|---|---|---|
| `Player.nationality`(`NationalityCode`) 표시명 | **번역함** | `enums.ts`의 국가 코드는 유한 집합(T8) — `EnumTranslationCatalog<NationalityCode>`로 감싸 3팀이 채운다 |
| `Team.stadiumName` | **번역 안 함** | D-17 절차적 생성 — 시드마다 새 문자열이 나오는 개방형 값(I-15 확정) |
| `Player.name` / `Manager.name` / 클럽·스폰서 이름 | **번역 안 함** | D-18에 이미 고유명사로 확정(T14) |

국적처럼 "코드값(유한, 번역 대상) → 표시 문자열"인 필드와, 구장명처럼 "이미 완성된
표시 문자열 자체가 필드값(개방형, 번역 비대상)"인 필드를 구분하는 게 핵심이다. 필드에
"~명/~name"이 붙어 있다는 것만으로 고유명사라 단정하거나, 반대로 코드가 아니라 문자열
필드라는 것만으로 카탈로그 대상이라 단정하지 않는다 — 매번 "유한 집합인가"로 되묻는다.

**(b) 에러/안내 문구 안에 고유명사가 끼는 경우 — 치환 방향이 두 가지다.**

`t.ts`의 `{name}` 치환(`PLACEHOLDER_PATTERN`)은 **템플릿 문자열 자체**와 **주입되는
값**을 별개로 다룬다. 이 둘의 번역 대상 여부가 반대로 갈리는 두 패턴이 실제로 공존한다:

1. **템플릿은 번역, 주입값은 원문 그대로** — 예: (미구현, 가상 사례) `"{name}을(를) 찾을
   수 없습니다"`처럼 에러 문구에 선수 이름을 끼워 넣는 키를 새로 만든다면, 템플릿 문자열은
   카탈로그(번역 대상)에 두되 `t(locale, key, { name: player.name })`로 넘기는
   `player.name` 값 자체는 그대로 주입한다 — **번역하지 않는다**(고유명사는 로케일이
   바뀌어도 같은 문자열).
2. **템플릿도, 주입값도 번역** — 이미 구현된 예: `common.header.seasonPhaseLabel`의
   `"시즌 페이즈: {phase} (준비 중)"`. `{phase}`에 들어갈 값은 `SeasonPhase` enum의
   **번역된 표시명**(`messages/*/enums.ts`의 `seasonPhase` 카탈로그, 3팀 기여)이어야
   한다 — enum 코드값(`"REGULAR"`)을 그대로 주입하면 안 된다.

리뷰 시 새 파라미터화 키를 볼 때마다 "이 자리에 원문 그대로 넣을 값인가, 이미 번역된
표시명을 넣을 값인가"를 키 옆 주석으로 명시한다(현재 8개 네임스페이스에는 (1) 패턴
사례가 아직 없다 — `error.ts` 4개 키 모두 파라미터 없음. 추가될 때 이 절을 갱신).

**(c) 고유명사·수치가 섞여 "완성된" 문자열은 카탈로그에 아예 넣지 않는다.**

`NewsFeedItem.headline`(`src/types/ops.ts`)처럼 고유명사와 구체 수치가 이미 혼합된 채
생성되는 필드는 (a)의 "구장명"과 같은 축이지만 한 단계 더 나아간 경우다 — 부분적으로도
카탈로그 템플릿화가 불가능하다(등장하는 고유명사 개수·위치가 매번 다르다). 이런 필드는
`t()` 경유 없이 생성된 문자열을 그대로 렌더링한다.

## 3. 자동 검출 vs 사람 눈 검출

1팀이 21일차에 CI에 편입하는 "번역 키 누락 검사"는 **구조**만 잡는다 — 위 §2의
**경계(콘텐츠) 위반은 tsc가 잡지 못한다.** 리뷰에서 반드시 사람이 확인해야 한다.

| 위반 유형 | 자동 검출 | 검출 지점 |
|---|---|---|
| en 카탈로그에 ko에 없는/다른 키가 있음, 또는 키 누락 | ✅ tsc | `en/*.ts`가 `ko/*.ts`의 `*Messages` 타입을 import해 `satisfies` — 구조 불일치 시 컴파일 오류 |
| 3단(`namespace.section.element`) 미달/초과 경로 참조 | ✅ tsc | `keys.ts`의 `DotPath`(중간 그룹은 유니온에서 제외) + `translationKey()` 호출부 |
| 오탈자 키, 존재하지 않는 네임스페이스 참조 | ✅ tsc | `TranslationKey` 유니온 밖 → 호출부 타입 오류(`keys.type-test.ts`가 회귀 고정) |
| enum 신규 멤버가 표시명 카탈로그에 안 채워짐 | ✅ tsc | `EnumTranslationCatalog<T>`(`@/types`)가 `T`의 전 멤버 매핑을 강제 |
| `src/**/*.tsx`의 JSX 자식 텍스트/표현식에 리터럴을 직접 박음 | 🟡 ESLint(warn) | `eslint.config.mjs`의 `no-restricted-syntax`(D-18, Task 010/18일차). 한글·영문자 포함 시 잡는다. **속성값(`title="..."` 등)은 범위 밖**, `src/i18n/**` 자신은 예외. 22일차 수락 대상 111건이 이 규칙 |
| ESLint가 하드코딩으로 잡은 문자열이 **사실은 고유명사라 원래 번역 비대상**인 경우(오탐) | ❌ | 사람 리뷰 — 규칙은 "한글/영문자 포함 리터럴"만 보지 D-18 경계는 모른다. 예: JSX에 선수 이름을 직접 찍으면 경고는 뜨지만 정작 고쳐야 할 건 "번역 키로 바꿔라"가 아니라 "그대로 둬도 된다" |
| **고유명사(선수·클럽·구장명 등)를 카탈로그에 번역 키로 넣음** | ❌ | 사람 리뷰 — 타입은 "문자열이 있다"만 보지 "번역 가능한 문자열인가"는 모른다 |
| 속성값(`alt=`/`aria-label=` 등)에 번역 대상 UI 텍스트를 하드코딩 | ❌ | 사람 리뷰 — ESLint 규칙이 JSX 텍스트 노드로 범위를 의도적으로 좁혀 속성은 탐지하지 않는다(§2.2(a)~(c) 판단 자체가 필요 없는 영역이라도 마찬가지) |
| §2.2(b)의 파라미터 치환 방향(원문 유지 vs 번역된 값 주입)을 반대로 배선 | ❌ | 사람 리뷰 — `t()` 시그니처상 `params`는 `string \| number` 아무 값이나 받는다 |
| enum 표시명 값이 여전히 placeholder(echo)인 채로 콘텐츠 확정 보고됨 | ❌ | 사람 리뷰 — 타입은 "문자열이 채워졌다"만 보지 "실제 번역인지 코드값 그대로인지"는 모른다 |

즉 tsc 게이트는 **"키가 존재하고 형태가 맞는가"**까지, ESLint 게이트는 **"카탈로그를
아예 거치지 않은 리터럴이 JSX 텍스트에 있는가"**까지만 보장한다. 둘 다 **그 리터럴/키에
들어간 내용이 §2의 번역 대상 경계를 지켰는가**는 판단하지 못한다 — 그 표현식이 카탈로그
번역 키(`t(...)`) 형태로만 바뀌면 두 게이트 모두 조용해지기 때문이다. 이 판단은 여전히
코드 리뷰(일차 마감 교차 점검) 몫이다.
